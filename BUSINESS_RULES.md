## Что такое BR

**BR (Business Rule)** — нерушимый инвариант предметной области, нарушение
которого ломает либо бизнес-смысл (себестоимость, ЗП, ответственность),
либо compliance-обязательство (152-ФЗ, ТК РФ, 402-ФЗ, 54-ФЗ), либо целостность
данных (антидублирование клиентов, связь материала с заказом).

BR ≠ feature, BR ≠ requirement. BR — это **то, что система обязана
гарантировать всегда**, независимо от UI / роли / редкого сценария.

## Как читать запись

```
### BR-NN — Название

**Описание.** Что именно гарантируем (одно-два предложения).

**Why.** Какую боль из § 2.2 ТЗ закрывает / какой compliance-риск снимает.

**Where enforced.** Backend (DRF/Node serializer / domain service) →
Database (constraint / trigger) → Frontend (UI-validation) → Mobile
(form-rule). По возможности — в нескольких слоях (defense in depth).

**Tests.** Unit / integration / e2e / negative.

**Related modules.** ТЗ § 6.X, § 7.Y; конкретные секции `Docs/04-modules.md`.

**TBD.** Если требует ответа клиента 🔴 #N или ADR-NNNN — ссылка.
```

## Как enforce'ить

| Слой | Инструменты | Когда обязателен |
| --- | --- | --- |
| **Database constraint** | UNIQUE, CHECK, FK, partial index, trigger (PG) | Всегда, если правило сводится к статической инвариантности данных (BR-02, BR-08, BR-15) |
| **Domain service / serializer** | DRF validators / Zod / pydantic / class-методы | Всегда — основной слой. Не полагаться только на UI |
| **API gateway / RBAC** | Middleware / permissions / policies | RBAC-правила (BR-03, BR-06, BR-14, BR-32, BR-34) |
| **Background job** | Celery beat / pg_cron / Temporal | Расчётные правила (BR-05, BR-12, BR-22, BR-29) |
| **Frontend validation** | react-hook-form + zod | UX (предупредить раньше, чем backend); не безопасность |
| **Mobile (PWA)** | Form rule + offline queue + service-worker | Когда правило проверяется на устройстве сотрудника |
| **Audit log** | `apps/audit_log` (или эквивалент) | Всегда для BR с compliance-весом (BR-06, BR-13, BR-20, BR-21, BR-36) |

**Принцип:** одно правило enforce'ится **минимум в двух слоях** (БД + service,
либо service + audit-log). UI-only enforce запрещён.

## Список инвариантов

### BR-01 — Списание материалов только на заказ

**Описание.** Любое движение материала со склада в производство фиксируется
с обязательной привязкой `material_movement.order_id`. Списание «в цех»,
«в дизайн», «общим расходом» запрещено. Возврат материала — также через
movement с типом `return` и связкой с тем же заказом.

**Why.** Критическая проблема AS-IS из ТЗ § 2.1: «материалы списываются
в цех, а не на конкретный заказ → невозможно рассчитать себестоимость,
определить ответственного, выявить перерасход». ТЗ § 2.3 называет это
**«КЛЮЧЕВОЕ»**. Без BR-01 теряется смысл всей финансовой подсистемы.

**Where enforced.**
- **DB:** `material_movement.order_id NOT NULL` для типа `outflow`/`return`,
  FK на `orders(id)`, CHECK на пары (тип, batch_id NOT NULL).
- **Service:** `WarehouseService.write_off()` — обязательный аргумент
  `order_id`, метод не существует без него.
- **Mobile (PWA):** форма списания на складе требует выбор заказа из
  активных, без выбора кнопка «Списать» disabled.
- **Audit:** каждое движение → `audit_log` (кто, что, когда, заказ,
  батч, цена-снимок).

**Tests.**
- *unit*: попытка `write_off(material_id, qty)` без `order_id` → 400.
- *e2e (PWA mobile)*: складщик не может submit'нуть форму без выбора заказа.
- *negative*: SQL-insert в обход API → нарушение FK / NOT NULL.

**Related modules.** § 6.11 Склад, § 6.12 Брак (повторное списание на тот
же заказ), § 7.2 Расход материалов, § 7.9 Себестоимость.

### BR-02 — Антидублирование клиентов и контрагентов по телефону

**Описание.** В системе не может существовать двух активных профилей
клиента с одинаковым нормализованным телефоном (формат E.164,
`+79...`). При попытке создать клиента с уже существующим телефоном —
система возвращает 409 + ссылку на существующий профиль. То же —
для юр-контрагентов (по ИНН + телефону руководителя).

**Why.** § 2.1: «один клиент может быть зарегистрирован несколько раз →
база клиентов неструктурированная, аналитика искажена». § 2.3 явно
требует «антидублирование, проверка по номеру телефона». BR-02 —
основа для корректного учёта LTV, истории заказов, recurring billing.

**Where enforced.**
- **DB:** `clients.phone_normalized` — UNIQUE partial index
  `WHERE deleted_at IS NULL`.
- **Service:** `ClientService.create()` — нормализация → lookup → если
  найден активный, возврат `409 Conflict` с `existing_client_id`.
- **Frontend:** форма «Новый клиент» — на blur поле телефона делает
  lookup и показывает «Клиент уже существует: <ФИО>, перейти к профилю?».

**Tests.**
- *unit*: `normalize_phone("+7 (985) 123-45-67")` == `"+79851234567"`.
- *integration*: повторный POST с тем же телефоном → 409.
- *e2e*: менеджер вводит уже зарегистрированный телефон → видит
  существующий профиль.

**Related modules.** § 6.3 Клиенты, § 6.1 Заказы, § 6.15 Клиентский
кабинет (логин по телефону через SMS-код).

### BR-03 — Брак фиксирует только складщик

**Описание.** Запись в `defect_log` может создать только пользователь
с ролью `warehouse_keeper` (складщик) или `admin` (override). Производство
(`production_*` роли) не имеет endpoint'а для создания записи о браке;
оно только готовит изделие и передаёт его на склад на проверку.

**Why.** ТЗ § 4.2: «производство НЕ фиксирует брак — ответственность
лежит на складщике» (роль контроля качества). Это разделение
полномочий: производство мотивировано НЕ фиксировать брак (штраф),
складщик мотивирован фиксировать (защита от приёмки). Нарушение BR-03
возвращает AS-IS-проблему «брак не фиксируется».

**Where enforced.**
- **RBAC:** permission `defects.create` — только для `warehouse_keeper`,
  `admin`. У `production_*` — отсутствует.
- **API gateway:** middleware блокирует POST `/api/defects` без
  permission.
- **UI:** в Mobile App (PWA) у роли «производство» нет кнопки «Зафиксировать
  брак». Виден только статус «На проверке» при передаче.
- **Audit:** все попытки нарушения логируются (security-event).

**Tests.**
- *unit*: `DefectService.create(actor=production_user)` → `PermissionError`.
- *e2e*: производственник не видит формы фиксации брака в Mobile App.
- *security*: прямой POST на endpoint в обход UI → 403.

**Related modules.** § 6.12 Брак, § 6.13 Повторное производство,
§ 4.1 Складщик, § 4.2 Производство.

### BR-04 — Услуги только из справочника, ручное создание запрещено

**Описание.** Поле `order_item.service_id` — обязательная FK на
`services` справочник. Ручной ввод названия услуги (text) при
создании заказа в production-флоу запрещён. Добавление новой услуги —
отдельный workflow с правом `admin` (см. BR-14).

**Why.** § 2.1: «услуги вводятся вручную → "Баннер", "баннер", "банер",
"другое" → дублирование, невозможность аналитики». § 2.3 «единый
справочник, запрет ручного ввода». Без BR-04 финансовая аналитика по
услугам бессмысленна.

**Where enforced.**
- **DB:** `order_items.service_id NOT NULL`, FK на `services(id)`.
- **Service:** `OrderService.add_item()` принимает только `service_id`,
  не `service_name`.
- **UI:** форма «Добавить услугу к заказу» — combobox с search по
  справочнику, без free-text. Кнопка «Создать новую услугу» доступна
  только админу через отдельный модуль 6.10.

**Tests.**
- *unit*: `add_item(name="Баннер")` без `service_id` → 400.
- *e2e*: менеджер не может ввести «банер» руками — только выбрать.

**Related modules.** § 6.4 Услуги/товары, § 6.10 Нормирование, § 6.18
Аналитика.

### BR-05 — Баланс сотрудника = выплачено − заработано (без ухода в минус по ТК)

**Описание.** Для каждого сотрудника ведётся `payroll_balance`. Формула:
`balance = sum(payments) − sum(earnings)`. Если за период заработано
меньше минимальной выплаты, разница доплачивается компанией и
фиксируется как «займ компании сотруднику». В следующем периоде, если
заработано больше — займ погашается. **Ограничение ТК РФ ст. 137:**
удержание из зарплаты не может превышать 20% (50% / 70% в особых
случаях), поэтому погашение долга разносится по нескольким периодам;
**выплата не уходит в минус**.

**Why.** ТЗ § 7.15 + § 6.22. Стабилизирует доход сотрудников при
переменной загрузке (типография — сезонный бизнес). Compliance: ТК РФ
ст. 137 защищает работника от обнуления зарплаты в счёт долга.

**Where enforced.**
- **Background job:** `PayrollCalculator.run_period(month)` вычисляет
  earned, payment, balance с проверкой потолка удержания.
- **DB:** `payroll_period(taken, paid, earned, debt)` колонки + CHECK
  `paid >= MIN_WAGE` (минимальная гарантированная выплата); `debt >= 0`
  (отрицательной выплаты не существует).
- **UI:** сотрудник видит баланс read-only в Mobile App (кабинет
  сотрудника) — не редактирует.
- **Audit:** все начисления / выплаты / доплаты → audit_log с
  иммутабельной историей.

**Tests.**
- *unit*: пример из ТЗ § 7.15 — мес.1 заработал 1.8M, выплачено 3M,
  долг −1.2M; мес.2 заработал 5M, выплата 3.8M, долг 0.
- *property-test*: при любых earned ≥ 0 → paid ≥ MIN_WAGE и
  удержание ≤ 20% от earned (ТК).
- *e2e*: сотрудник видит баланс, не может ввести правки.

**TBD.** Точное значение `MIN_WAGE`, периодичность (месяц / 2 недели),
формула удержания → 🟡 уточнение владельца + ADR. (Двойная формула
ЗП = тариф × кол-во | % стоимости — закрыто 2026-05-05, см. BR-31.)

**Related modules.** § 6.22 Сдельная ЗП и баланс, § 7.6-7.10 Расчёты
ЗП, § 7.15 Баланс, § 8.3 Логирование.

### BR-06 — Face Control события read-only (с отдельным workflow корректировки)

**Описание.** Записи в `face_control_event` (вход/выход с timestamp +
employee_id) приходят из Face Control SDK, не редактируются
пользователем. При необходимости корректировки (e.g. сотрудник забыл
карту, пропустил камеру) — отдельный workflow `attendance_adjustment`:
сотрудник заполняет форму запроса → менеджер/админ подтверждает →
создаётся запись `attendance_adjustment` с указанием original_event,
причины, подписью утвердившего. **Оригинальное событие НЕ изменяется.**

**Why.** ТЗ § 6.20: «сотрудник не может изменить данные; данные поступают
автоматически». Это основа для BR-22 (сравнение присутствия с фактической
работой) и для BR-28 (контроль дисциплины). Compliance: 152-ФЗ ст. 11
требует прозрачную обработку биометрических ПДн — иммутабельность
журнала фиксации.

**Where enforced.**
- **DB:** `face_control_event` — таблица **append-only**, без UPDATE/
  DELETE триггеров (raise exception).
- **Service:** ingest endpoint доступен только для service-account
  Face Control SDK.
- **Workflow:** `AttendanceAdjustmentService.request()` /
  `.approve()` — отдельные методы, никогда не пишут в `face_control_event`.
- **UI:** сотрудник видит события read-only с маркером «корректировка
  ожидает утверждения», менеджер видит очередь approval.
- **Audit:** каждая корректировка → запись в audit_log с до/после.

**Tests.**
- *unit*: попытка `face_control_event.update()` → exception.
- *integration*: workflow корректировки создаёт adjustment, не
  модифицирует event.
- *security*: insert через SQL в `face_control_event` без service-token
  → отклонён.

**TBD.** Vendor Face Control (формат событий, timezone,
duplicate-handling) → 🔴 #2 + ADR-0002.

**Related modules.** § 6.20 Face Control, § 6.8 Учёт времени,
§ 6.21 KPI, § 7.7 Простои, § 9.6 Безопасность.

### BR-07 — 1 заказ = 1 тип (цех / офис / товар), смена типа = новый заказ

**Описание.** `order.type` — обязательный enum: `workshop` (цех / наружная
реклама), `office_polygraphy` (полиграфия), `goods` (продажа товара).
Тип фиксируется при создании заказа и **не меняется**. Если клиент
пересматривает запрос (например, из «купить готовое» в «изготовить») —
старый заказ закрывается со статусом `transferred`, создаётся новый
заказ с правильным типом и ссылкой `parent_order_id`.

**Why.** Каждый тип имеет свою статус-машину (цех — 15 шагов, офис —
4 шага, товар — 3 шага). Смена типа на лету ломает машину состояний,
аналитику, расчёт себестоимости (разные формулы), KPI.

**Where enforced.**
- **DB:** `orders.type` enum, без триггера на UPDATE этой колонки.
- **Service:** `OrderService.update()` явно отвергает изменение
  `type` → 400 «type is immutable, create a new order».
- **UI:** в форме редактирования заказа поле `type` — read-only после
  сохранения.

**Tests.**
- *unit*: попытка PATCH `/orders/{id}` с `type=goods` (был `workshop`)
  → 400.
- *e2e*: менеджер видит заблокированное поле type на редактировании.

**Related modules.** § 5.1 Определение типа заказа, § 6.1 Заказы, все
производственные модули.

### BR-08 — 1 лид → максимум 1 заказ

**Описание.** `lead.converted_order_id` — UNIQUE NULLable. Лид может
быть конвертирован максимум в один заказ. Повторное использование
лида (например, для другого замера) запрещено — создаётся новый лид.

**Why.** ТЗ § 6.2: «лид может быть преобразован только в один заказ».
Защищает аналитику конверсии лидов от двойного учёта.

**Where enforced.**
- **DB:** UNIQUE INDEX на `lead.converted_order_id WHERE NOT NULL`.
- **Service:** `LeadService.convert()` проверяет, что лид ещё не
  конвертирован.

**Tests.**
- *unit*: повторный convert → 409.

**Related modules.** § 6.2 Лиды, § 6.1 Заказы.

### BR-09 — Списание материалов FIFO по партиям (partybatch)

**Описание.** Каждая поступающая партия материала фиксируется в
`material_batch` с датой поступления и количеством. При списании
система автоматически выбирает партию (по умолчанию **FIFO** — первая
поступившая первой; настраиваемо через `material.outflow_strategy`).
В `material_movement` фиксируется конкретная `batch_id` и
`unit_cost_at_writeoff`. Это позволяет точно считать себестоимость и
трекать качество от партии.

**Why.** ТЗ § 1.5 Термин «Партия (Batch)»; § 6.11 Склад «все движения
фиксируются». Без партионного учёта невозможно: 1) выявить партию-источник
брака (e.g. бракованный рулон оракала); 2) корректно рассчитать
себестоимость при росте цен на материал между партиями.

**Where enforced.**
- **DB:** `material_movement.batch_id NOT NULL`, FK на `material_batch`;
  `material_batch.qty_remaining >= 0` CHECK.
- **Service:** `WarehouseService.write_off()` выбирает партию по
  стратегии в транзакции с `FOR UPDATE`, фиксирует в movement.
- **UI:** складщик может вручную override партию (e.g. использовать
  более старую) — но обязан указать причину.

**Tests.**
- *unit*: при наличии 3 партий FIFO выбирает первую.
- *unit*: при override — записывается в movement с notes.
- *concurrency*: 2 параллельных списания не уходят в минус по `qty_remaining`.

**Related modules.** § 6.11 Склад, § 7.2 Расход материалов, § 7.9
Себестоимость.

### BR-10 — Заказ нельзя завершить без списанных материалов и зафиксированного времени

**Описание.** Переход `order.status → completed` (для типов цех / офис)
блокируется, если: (а) `order.items` имеет связанные операции,
требующие материалов, и `material_movement.order_id = order.id` пуст
(или explicit override от складщика «материалов не требовалось»),
**ИЛИ** (б) есть `operation_log` без `started_at`/`finished_at`. Заказ
завершается только когда вся себестоимость собрана.

**Why.** Фундамент финансовой аналитики: `Себестоимость = Материалы +
Работа + Потери` (ТЗ § 7.9). Если хоть один из компонентов не
зафиксирован при завершении, прибыль рассчитывается неверно.

**Where enforced.**
- **Service:** `OrderService.complete()` запускает валидацию
  `OrderCompletenessChecker.is_complete(order_id)`, отбрасывает 409
  с детальным описанием.
- **UI:** кнопка «Завершить» disabled с tooltip «Не списаны материалы:
  …, не закрыты операции: …».

**Tests.**
- *unit*: complete без movements → 409.
- *unit*: complete без operation_log finish → 409.
- *e2e*: менеджер видит подсказку, что не хватает.

### BR-13 — История изменений критичных сущностей immutable (audit-log)

**Описание.** Любое изменение `orders.*`, `clients.*`, `material_movement.*`,
`payroll_*`, `defect_log.*`, `face_control_event.*`, справочников

Физическое удаление перечисленных сущностей запрещено (только soft-delete).

заказа». ТК РФ — хранение зарплатных ведомостей 75 лет.

**Where enforced.**
- **DB:** `audit_log` — таблица append-only (revoke UPDATE/DELETE
  для всех ролей кроме `audit_archive`).
- **Trigger:** на изменения в orders/order_items/etc пишется запись
  в audit_log автоматически.
- **Service:** ручные изменения через API проходят через декоратор
  `@audit`.

**Tests.**
- *integration*: PATCH /orders/{id} → запись в audit_log с before/after.
- *security*: попытка DELETE из audit_log → revoked.

**Related modules.** § 6.1 Заказы, § 8.3 Логирование, § 9.7 Backup.

### BR-14 — Все справочники меняет только администратор

**Описание.** Таблицы `services`, `materials`, `operations`,

старых заказов использует **исторический snapshot** норматива на момент

- **RBAC:** permissions `services.edit`, `materials.edit`, `norms.edit`
  → только `admin`.
- **API:** все mutate-endpoints → 403 для прочих ролей.
- **DB:** snapshot нормативов в `order_items.norm_snapshot_json` на
  момент создания.
- **Audit:** изменения нормативов отдельный канал нотификаций — owner
  получает уведомление.

**Tests.**
- *unit*: PATCH /services/{id} от менеджера → 403.
- *integration*: change to norm_rate → audit + notification + старый
  заказ считается по snapshot.

### BR-26 — Логистика рассчитывается через Yandex Maps + формула § 7.18

(закреплено 2026-05-05) между точкой выдачи и адресом клиента.
Эта сумма входит в себестоимость заказа отдельной статьёй; пересчёт
при смене адреса клиентом.

**Why.** ТЗ § 6.24, § 7.18; решение владельца 2026-05-05. Без
правильного учёта логистики маржинальность выездных заказов искажена.

**Where enforced.**
- **Service:** `LogisticsCalculator.compute(order_id, distance,
  duration)` вызывает Yandex Maps; fallback на ручной ввод
  km менеджером при ошибке API.
- **DB:** `logistics_cost` per order.

**Tests.**
- *unit*: пример расчёта по формуле.
- *integration*: интеграция Yandex Maps падает → fallback.

**TBD.** Тариф ГСМ по марке авто, ставки амортизации авто, ставки
водителей → 🟡 владелец.

**Related modules.** § 6.24 Логистика, § 7.18 Затраты, § 7.9

### BR-27 — НДС / налоги автоматически добавляются к цене (по СНО)

**Описание.** При расчёте стоимости заказа применяется налоговая
надстройка по `tax_profile` контрагента/компании. По умолчанию НДС 20%
для ОСН, 0 для УСН (зависит от Q1 юрисдикции). Налоговая часть в
финансовой аналитике отображается отдельно от чистой прибыли.

**Why.** ТЗ § 7.16 «автоматическое добавление НДС». Защита от
ошибок при формировании ценника клиенту.

**Where enforced.**
- **Service:** `PriceCalculator.with_tax(amount, profile)`.
- **DB:** `tax_profile` справочник, `order.tax_amount`, `order.tax_rate`,
  `order.tax_period`.

**Tests.**
- *unit*: 100 ₽ + НДС 20% = 120 ₽; в финансовой аналитике 100 ₽
  выручка + 20 ₽ налог.

**TBD.** Конкретный СНО клиента → 🔴 #1 (юрисдикция) → ADR.

### BR-28 — Опоздания и ранние уходы фиксируются автоматически

**Описание.** На основе `face_control_event` + графика смены сотрудника

(если > THRESHOLD), ранний уход = `shift_end - exit_at`. Эти события
записываются в `discipline_event` и используются в KPI / штрафах.

**Why.** ТЗ § 6.20: «опоздания, ранний уход» как аналитика. Бизнес —
дисциплина без ручного контроля.

**Where enforced.**
- **Background job:** ежедневно, после закрытия дня.
- **UI:** дашборд директора и личный кабинет сотрудника.

### BR-29 — Уведомление о ТО оборудования при достижении порога наработки

**Описание.** Каждое `equipment` имеет `maintenance_threshold_hours` и/или
`_meters`. При накоплении наработки до 90% порога — система создаёт
warning-уведомление admin/owner. При 100% — оборудование помечается
`needs_maintenance` и блокируется в выборе для новых операций. После
выполнения ТО — admin закрывает задачу, наработка обнуляется (либо
переносится — настраивается).

**Why.** ТЗ § 6.25: «автоматические уведомления о необходимости ТО».
Бизнес: предотвращение поломок (стоимость ТО ниже стоимости простоя
+ ремонта).

**Where enforced.**
- **Background job:** проверяет накопление + создаёт задачи.
- **DB:** `equipment.last_maintenance_at`, `cumulative_usage`,
  статус `OK | WARNING | MAINTENANCE_REQUIRED`.
- **UI:** в выборе оборудования для операции — статус
  `MAINTENANCE_REQUIRED` делает позицию disabled.

**TBD.** Конкретные пороги для каждого станка, формат «обнуления» →
🟡 владелец + R3 (доменный консультант полиграфии).

**Related modules.** § 6.25 Оборудование, § 6.16 Уведомления.

### BR-30 — Оплата клиента синхронизируется со статусом производственного цикла

**Описание.** Для заказов с предоплатой переход в производство
блокируется, если `payment.status != paid`. Для заказов «оплата по
факту» — выдача клиенту (`order.status → completed`) блокируется,
если оплата не зарегистрирована или клиент не подтвердил отсрочку.
Платежи и возвраты — атомарные транзакции с `Idempotency-Key`.

**Why.** ТЗ § 9.11: «автоматическая синхронизация статуса оплаты с
производственным циклом». Защита от кассовых разрывов и долгов клиентов.

**Where enforced.**
- **Service:** state-machine guards в `OrderService`; `PaymentService.process()`
  внутри одной транзакции; outbox для внешних вызовов.
- **API:** принимает `Idempotency-Key` header, дубль возвращает тот же ответ.
- **UI:** менеджер видит флаг «ожидает оплаты, нельзя производить».

**TBD.** Эквайринг (YooKassa / Тинькофф / Сбер / СБП) → 🔴 #5 + ADR-0005.

**Related modules.** § 9.11 Методы оплаты, § 6.1 Заказы, § 6.14 Продажа,
§ 7.13 Прибыль.

### BR-31 — Сдельная ЗП — двойная формула, конфигурируется per operation/role

**Описание.** ЗП за операцию рассчитывается одной из двух формул на
выбор (per `operation × role`):
- **Тарифная:** `сумма = rate_per_unit × quantity` (например, 50 ₽/м² печати).
- **Процентная:** `сумма = client_cost × percent / 100` (например,
  30% от стоимости работы клиенту).
Выбор фиксируется в `operation_payroll_rule(operation_id, role_id,
formula_type, rate_or_percent)`.

**Why.** ТЗ § 6.22, § 7.12; решение владельца 2026-05-05 — двойная
формула; гибкость под разные операции (печать — тариф, монтаж — процент).

**Where enforced.**
- **DB:** `operation_payroll_rule.formula_type ENUM('TARIFF', 'PERCENT')`,
  `rate_or_percent NUMERIC(10,4) > 0`.
- **Service:** `PayrollService.earned_for_operation()` диспатчит на
  формулу.
- **UI:** админ в справочнике задаёт правило для пары (операция, роль).
- **Audit:** изменение правила фиксируется в `catalog_changelog`;
  старые расчёты используют snapshot.

**Tests.**
- *unit (TARIFF)*: 100 м² × 50 ₽ = 5000 ₽.
- *unit (PERCENT)*: client_cost = 20000 ₽, 30% = 6000 ₽.
- Изменение rate за прошлый период не пересчитывает закрытые ведомости.

**Related modules.** § 6.22 Баланс ЗП, § 7.12 Расчёт ЗП, § 6.10
Нормирование, BR-05.

### BR-32 — Клиент видит только свои заказы (без себестоимости и сотрудников)

**Описание.** Клиентский кабинет (роль `client`) показывает: свои
заказы (по `client_id == request.user.client_id`), статус, цену,
документы. Скрывает: себестоимость, имена производственных сотрудников,
чужие заказы, внутренние комментарии, audit-log.

**Why.** ТЗ § 4.7 «нет доступа к внутренним данным, к себестоимости,
к сотрудникам». Бизнес: защита маржи от расчётов «обратно» клиентом.

**Where enforced.**
- **RBAC:** field-level permissions в DRF/serializer (для клиента —
  только public-fields).
- **API:** queryset фильтр `Order.objects.filter(client=request.user.client)`.

**Tests.**
- *unit*: клиент видит свой заказ, не видит чужой → 404.
- *security*: клиент не может получить cost-breakdown через API.

**Related modules.** § 6.15 Кабинет, § 8.2 RBAC, § 4.7 Клиент.

### BR-33 — Макеты хранятся в Yandex Object Storage минимум 12 месяцев

**Описание.** Файлы макетов (10 MB – 1 GB) хранятся в **Yandex Object
Storage** (S3-совместимое, закреплено 2026-05-05) с lifecycle-политикой
≥ 12 месяцев с момента `order.completed_at`. Метаданные (path, size,
hash, version) — в БД. Удаление автоматическое; для длительного
хранения админ может пометить `retain = true`.

**Why.** ТЗ § 9.10: «срок гарантированного хранения макетов — не
менее 12 месяцев». Бизнес: повторные заказы клиента с правкой макета
+ юридическая защита (доказательство утверждённой версии).

**Where enforced.**
- **S3 lifecycle policy:** автомат-перевод в архив после X месяцев,
  delete после 12.
- **DB:** `design_version.s3_path`, `s3_lifecycle_status`.

**TBD.** Регион Yandex Object Storage зависит от хостинга → 🔴 #3 +
ADR-0003.

**Related modules.** § 9.10, § 6.5 Дизайн.

### BR-34 — Производство не редактирует параметры заказа (только статус и operation_log)

**Описание.** Роль `production_*` имеет write-доступ только к
`operation_log` (start/finish операций) и к ограниченному набору
полей `order` (status в строго определённых переходах). Изменить
размер, услугу, цену, клиента — производство не может.

**Why.** Разделение полномочий. Если производство меняет параметры
заказа — теряется контроль качества (BR-11), искажается финансовая
картина.

**Where enforced.**
- **RBAC:** field-level permissions.
- **Service:** методы изменения order-полей проверяют actor.role.

**Tests.**
- *unit*: PATCH `/orders/{id}` от производственника с size=… → 403.

**Related modules.** § 4.2 Производство, § 6.7, § 6.1.

### BR-35 — Резервное копирование RPO ≤ 24h, RTO ≤ 4h

**Описание.** PostgreSQL базы (production) и Yandex Object Storage
(макеты) бэкапятся ежедневно (PITR-WAL для PG, snapshot для S3).
RTO ≤ 4h, RPO ≤ 24h. Бэкапы хранятся минимум 30 дней + ежемесячный —
12 месяцев.

**Why.** ТЗ § 9.7. Compliance: 402-ФЗ хранение первички 5 лет
(косвенно — через бэкапы), 152-ФЗ обязательство сохранности ПДн.

**Where enforced.**
- **Infrastructure:** Yandex Managed PostgreSQL автобэкапы (фаза 1+).
- **Runbook:** `Docs/runbook/backup-restore.md`.

**TBD.** Конкретные tooling и retention politics → 🔴 #3 + ADR-0003.

### BR-36 — Расчётный лист сотрудника генерируется при каждой выплате (ТК ст. 136) + B2C-чек 54-ФЗ

**Описание.** При каждом начислении/выплате ЗП сотрудник получает
расчётный лист с разбивкой: оклад/база, начисления за операции,
KPI-надбавки, штрафы, удержания (включая погашение долга по BR-05),
итог к выплате. Лист доступен в Mobile App + email. Параллельно — для
B2C-оплат клиентов выпускается фискальный чек через ОФД-провайдера
(54-ФЗ) автоматически на событие `PaymentSucceeded`. Для B2B-операций
(расчётный счёт) чек не выпускается — только акт + счёт.

**Why.** ТК РФ ст. 136 — обязательное информирование работника о
составных частях ЗП. 54-ФЗ — обязательная фискализация B2C-операций
(закреплено 2026-05-05: микс B2B+B2C → ОФД нужен).

**Where enforced.**
- **Service:** `PayrollService.generate_payslip(employee, period)` +
  `FiscalService.issue_receipt(payment_id)` (срабатывает на event
  `PaymentSucceeded` если `client.type == B2C`).
- **Document template:** в `apps/docflow`.
- **DB:** `fiscal_receipt(fp, fd, qr_url, payment_id UNIQUE)`.
- **Audit:** факт получения / отправки.

**TBD.** ОФД-провайдер (Платформа ОФД / Такском / Контур.ОФД) → 🔴 #5
(под-блокер от Q14).

**Related modules.** § 6.22 Баланс, § 7.12 ЗП, § 6.23 Документооборот,
NFR § 9.11 Методы оплаты, `Docs/09-compliance.md` § ТК РФ + § 54-ФЗ.
