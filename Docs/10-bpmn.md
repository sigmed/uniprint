## Как читать диаграммы

Диаграммы написаны в **Mermaid** — text-based формат, рендерится
автоматически в GitHub / GitLab / VS Code (с расширением
`bierner.markdown-mermaid`) / любом просмотрщике с поддержкой Mermaid.

В этом документе используются:

- **`flowchart TD`** — top-down схемы процессов (точки решений, ветки,
  параллели).
- **`stateDiagram-v2`** — статус-машины заказа / расчёта ЗП.
- **`sequenceDiagram`** — последовательности взаимодействия систем
  (Face Control события).

Сложные диаграммы разбиты на под-схемы (например, P1 «Услуга-цех» —
3 фрагмента).

## Условные обозначения

| Узел | Что означает |
| --- | --- |
| `[Прямоугольник]` | Действие / задача (task). Пишется глаголом: «Создать заказ», «Списать материал». |
| `{Ромб}` | Точка решения (gateway). Пишется вопросом: «Брак?», «Approved?». |
| `((Круг))` | Событие старта / окончания (start / end event). |
| `[/Параллелограмм/]` | Получение / отправка данных извне (Face Control event, payment webhook). |
| `[(База)]` | Запись / чтение из БД. |
| Подписи на стрелках | Условие перехода: «да», «нет», «брак», «OK». |

Все процессы привязаны к **бизнес-правилам** (BR-XX, см.
[`BUSINESS_RULES.md`](../BUSINESS_RULES.md)) — они помечены прямо в
диаграммах. Точки касания UI помечены как «(Mobile-PWA)» или «(Web)»
или «(client-portal)».

## P0. Роутер типа заказа

> **ТЗ § 5.1** — менеджер при создании заказа обязан выбрать тип. От
> типа зависит весь дальнейший workflow (BR-07).

```mermaid
flowchart TD
    Start((Клиент обратился))
    Start --> CreateOrder["Менеджер: открыть форму нового заказа<br/>Web / Mobile-PWA"]
    CreateOrder --> FindClient{"Найден клиент<br/>по телефону?<br/>(BR-02)"}
    FindClient -- "Найден" --> AttachClient["Прикрепить существующий профиль"]
    FindClient -- "Не найден" --> NewClient["Создать клиента<br/>(имя, тел., B2B/B2C, ИНН)"]
    AttachClient --> SelectType
    NewClient --> SelectType
    SelectType{"Выбрать тип заказа<br/>(BR-07)"}
    SelectType -- "Услуга-цех" --> P1["P1. Услуга (цех)<br/>15 шагов, ТЗ § 5.1.1"]
    SelectType -- "Услуга-офис" --> P2["P2. Услуга (офис/полиграфия)<br/>4 шага, ТЗ § 5.1.2"]
    SelectType -- "Продажа-товар" --> P3["P3. Продажа (товар)<br/>3 шага, ТЗ § 5.1.3"]
    P1 --> EndCex((Заказ-цех в работе))
    P2 --> EndOffice((Заказ-офис в работе))
    P3 --> EndGoods((Продажа в работе))
```

**Пояснение.**

- **Антидублирование клиента (BR-02).** Поиск идёт по нормализованному
  телефону через `libphonenumber` — `+7`, `8`, `7-XXX-...`, `+7 (XXX)`
  приводятся к единому формату. При неоднозначности — fuzzy-match по
  имени с подсказкой «возможно, это один клиент».
- **Выбор типа (BR-07).** Это **обязательное** поле, защищено валидацией
  на бэке. Без типа заказ не сохраняется. Тип нельзя поменять после
  создания (только отмена + новый заказ).
- **Услуги — только из справочника (BR-04).** Поле «услуга» — селект
  из `apps/catalog`, ручной ввод запрещён enforce'ом на бэке.

**Точки риска:** O3 (формат телефона), C2 (клиент создаёт сам в
кабинете — see J10).

## P1. Услуга (цех) — наружная реклама

> **ТЗ § 5.1.1** — самый длинный процесс, 15 шагов от первого контакта
> до выдачи / монтажа. Эталон статус-машины.

### P1.1. Шаги 1–8: лид → дизайн → согласование с клиентом

```mermaid
flowchart TD
    S1((Лид: контакт<br/>с потенц. клиентом))
    S1 --> S2["1. Поиск клиента + первичный контакт<br/>(Выездной менеджер · Mobile-PWA)"]
    S2 --> S3["2. Замер на объекте<br/>фото + GPS + размеры<br/>(Mobile-PWA)"]
    S3 --> S4["3. Создать лид в системе<br/>статус: Лид-замер выполнен"]
    S4 --> S5["4. Конвертировать лид в заказ<br/>тип = Услуга-цех (BR-07)"]
    S5 --> S6["5. Передать заказ дизайнеру<br/>статус: На дизайне<br/>WebPush + Email дизайнеру"]
    S6 --> S7["6. Подготовить макет<br/>загрузка в Yandex Object Storage<br/>multipart, версия 1<br/>(Дизайнер · Web)"]
    S7 --> S8["7. Проверка макета<br/>нач. цеха<br/>(Web)"]
    S8 --> Q1{"Ошибки в макете?<br/>(грамматика / размеры)"}
    Q1 -- "Да, правки" --> S7
    Q1 -- "OK" --> S9["8. Согласование с клиентом<br/>(client-portal + WebPush + Email)"]
    S9 --> Q2{"Клиент<br/>Approved?"}
    Q2 -- "Rework" --> S7
    Q2 -- "Approved" --> Next1((переход к P1.2))
```

### P1.2. Шаги 9–14: производство → склад → брак-fix

```mermaid
flowchart TD
    Start1((После согласования макета))
    Start1 --> S10["9. Подготовка макета к производству<br/>припуски, ICC, m.п.<br/>(Дизайнер · Web)"]
    S10 --> S11["10. Передача в производство<br/>статус: В производстве<br/>создание задач по операциям"]
    S11 --> S12["11. Списание материалов на заказ<br/>(BR-01) только на заказ!<br/>(Складщик · Mobile-PWA warehouse)"]
    S12 --> S13["12. Выполнение операций<br/>Начать → Завершить + таймер<br/>(Производство · Mobile-PWA)"]
    S13 --> S14["13. Передать готовое на склад<br/>статус: На проверке"]
    S14 --> S15["14. Контроль качества<br/>(Складщик · Mobile-PWA)"]
    S15 --> Q3{"Брак?<br/>(BR-03 — только<br/>складщик фиксирует)"}
    Q3 -- "Да" --> S16["Зафиксировать брак<br/>фото + причина + этап<br/>+ ответственный"]
    S16 --> S17["Инициировать переделку<br/>модуль 6.13<br/>статус: Брак — на переделку"]
    S17 --> S12
    Q3 -- "OK" --> Next2((переход к P1.3))
```

### P1.3. Шаг 15: выдача / монтаж

```mermaid
flowchart TD
    Start2((После КК — заказ принят))
    Start2 --> S18["15. Выдача / монтаж<br/>(Менеджер / монтажник / водитель)"]
    S18 --> Q4{"Тип<br/>выдачи?"}
    Q4 -- "Самовывоз" --> S19["Клиент приехал в офис<br/>Mobile-PWA: подтверждение выдачи"]
    Q4 -- "Доставка / монтаж" --> S20["Назначить водителя/монтажника<br/>модуль 6.24 (фаза 3)<br/>Yandex Maps маршрут"]
    S20 --> S21["GPS-трек, статус<br/>В пути → На монтаже"]
    S21 --> S22["Фотофиксация монтажа<br/>(Mobile-PWA)"]
    S22 --> S23
    S19 --> S23["Завершить заказ<br/>статус: Завершён"]
    S23 --> S24["Автогенерация документов<br/>счёт / акт / ТТН (модуль 6.23)<br/>в Yandex Object Storage"]
    S24 --> S25["Триггер: задача<br/>постобслуживания через 1-2 дня<br/>(см. P6)"]
    S25 --> End((Завершён))
```

### Статус-машина заказа цеха

```mermaid
stateDiagram-v2
    [*] --> Лид
    Лид --> Создан: конвертация
    Создан --> НаДизайне: передача дизайнеру
    НаДизайне --> ПроверкаМакета: загружен макет
    ПроверкаМакета --> НаДизайне: правки нач. цеха
    ПроверкаМакета --> СогласованиеКлиент: OK
    СогласованиеКлиент --> НаДизайне: Rework
    СогласованиеКлиент --> ГотовКПроизводству: Approved
    ГотовКПроизводству --> ВПроизводстве: задачи созданы
    ВПроизводстве --> НаПроверке: production done
    НаПроверке --> БракПеределка: брак (BR-03)
    БракПеределка --> ВПроизводстве: материал списан повторно
    НаПроверке --> ГотовКВыдаче: OK
    ГотовКВыдаче --> ВПути: водитель принял (фаза 3)
    ВПути --> НаМонтаже: прибыл
    НаМонтаже --> Завершён: монтаж done
    ГотовКВыдаче --> Завершён: самовывоз
    Завершён --> [*]
```

**Пояснение.**

- **Шаг 6 / 9** — макет в **Yandex Object Storage** (🔴 #11 закрыт).
  Multipart upload до 2 ГБ, версионирование через S3 versions API.
- **Шаг 8** — клиент согласует макет в client-portal. Уведомления —
  WebPush + Email (Telegram **не используется**, 🔴 #7).
- **Шаг 11 (BR-01)** — критическая инвариаранта: списание возможно
  **только на заказ**, бэк не позволит создать `MaterialMovement`
  с `order_id = null` для типа «списание». Это решает главную AS-IS
  проблему типографии.
- **Шаг 14 (BR-03)** — брак фиксирует **только складщик**. Производство
  не имеет UI для фиксации. Это устраняет конфликт интересов.
- **Брак-цикл** — при браке материал списывается **повторно на тот же
  заказ** (`order_id = #1234`, type = `defect_rework`). История
  списаний показывает реальный перерасход.

**Открытые вопросы:**
- Кто платит за переделку при браке материала (поставщик / типография /
  клиент)? Уточнить с владельцем.
- Согласование макета: молчание клиента N дней = согласие? Юр. статус
  при претензии.

## P2. Услуга (офис / полиграфия)

> **ТЗ § 5.1.2** — оперативная печать, 4 шага.

```mermaid
flowchart TD
    Start((Клиент<br/>пришёл/онлайн))
    Start --> S1["1. Приём заказа<br/>тип = Услуга-офис (BR-07)<br/>параметры: формат, тираж, материал<br/>(Менеджер / клиент · Web / client-portal)"]
    S1 --> Q1{"Нужен макет?"}
    Q1 -- "Готовый" --> S2A["2a. Загрузить готовый макет<br/>(client-portal / Web)"]
    Q1 -- "Свой нужен" --> S2B["2b. Быстрая подготовка макета<br/>(Дизайнер · Web)"]
    S2A --> S3
    S2B --> S3["3. Печать в офисе<br/>(Полиграфист · Mobile-PWA)<br/>Начать / Завершить + таймер"]
    S3 --> S4["Списание материалов<br/>на заказ (BR-01)<br/>(Складщик · Mobile-PWA)"]
    S4 --> Q2{"Контроль<br/>качества:<br/>брак?"}
    Q2 -- "Брак" --> S5["Фиксация брака (BR-03)<br/>+ повторная печать"]
    S5 --> S3
    Q2 -- "OK" --> S6["4. Выдача клиенту<br/>оплата + документы<br/>WebPush 'готов к выдаче'"]
    S6 --> End((Завершён))
```

**Пояснение.**

- В отличие от P1, дизайн часто **отсутствует** или сводится к
  минимальной правке — клиент приходит со своим готовым PDF.
- Списание материалов BR-01 действует и здесь: 100 листов A4 — это
  100 единиц SKU «бумага A4 80 г/м²», списанных на заказ.
- BR-03 действует — даже на маленьких тиражах брак фиксирует только
  складщик / контролёр на выдаче.
- Канал нотификации клиенту — WebPush (если PWA / браузер открыт) +
  Email-копия + SMS для критичных случаев.

**Открытые вопросы:**
- Срочный заказ «через час» — отдельный тип услуги или наценка к
  обычной? Влияет на каталог.

## P3. Продажа (товар)

> **ТЗ § 5.1.3** — реализация со склада готовой продукции, 3 шага.

```mermaid
flowchart TD
    Start((Клиент хочет<br/>готовый товар))
    Start --> S1["1. Приём заказа<br/>тип = Продажа-товар (BR-07)<br/>выбор товара из каталога<br/>проверка наличия (онлайн-остатки)"]
    S1 --> Q1{"Товар<br/>в наличии?"}
    Q1 -- "Нет" --> R1["Отказ / задача закупщику<br/>(вне платформы)"]
    R1 --> EndR((Отменён))
    Q1 -- "Да" --> S2["2. Оформление продажи<br/>+ оплата"]
    S2 --> Q2{"Тип<br/>оплаты?"}
    Q2 -- "B2C: карта/СБП" --> S3A["Эквайринг (после 🔴 #5)<br/>+ онлайн-чек ОФД (54-ФЗ)"]
    Q2 -- "B2B: р/с" --> S3B["Счёт PDF<br/>оплата по р/с в банке"]
    Q2 -- "Наличные B2C" --> S3C["Наличный приём<br/>+ онлайн-чек ОФД (54-ФЗ)"]
    S3A --> S4
    S3B --> S4
    S3C --> S4["3. Выдача товара<br/>(Складщик / менеджер · Mobile-PWA)"]
    S4 --> S5["Списание со склада<br/>type = sale (не production)<br/>статус: Выдан"]
    S5 --> S6["Автогенерация документов<br/>счёт-фактура / акт / ТТН"]
    S6 --> End((Завершён))
```

**Пояснение.**

- **54-ФЗ применим** — по ответу 🔴 #14 (закрыт) типография работает
  с миксом B2B + B2C → ОФД-провайдер нужен. Под-блокер от 🔴 #5
  (эквайринг + ОФД-провайдер выбираются вместе).
- Тип движения — `sale`, отличается от `production` (списание на
  производство по BR-01). В аналитике это разные категории.
- Для B2B клиент получает счёт PDF и оплачивает по р/с — пробитие
  чека не требуется.
- BR-03 формально не применим — товар уже готов. Но при возврате
  клиентом → товар заново проверяется складщиком.

**Открытые вопросы:**
- **Возвраты товара** — отдельный workflow, не описан в ТЗ. Включает
  чек коррекции через ОФД (54-ФЗ ст. 4.7). Фаза 2+.

## P4. Управление складом

> **ТЗ § 5.2.1, 5.2.2** + модуль 6.11. Три сценария: приёмка,
> списание (BR-01), инвентаризация.

```mermaid
flowchart TD
    Start((Складская операция))
    Start --> Q1{"Тип<br/>операции?"}
    Q1 -- "Приёмка" --> Branch1
    Q1 -- "Списание" --> Branch2
    Q1 -- "Инвентаризация" --> Branch3

    %% Приёмка
    Branch1["A1. Приёмка от поставщика<br/>(Складщик · Mobile-PWA)"]
    Branch1 --> A2["Сканер штрих-кода<br/>BarcodeDetector API<br/>или поиск по SKU"]
    A2 --> Q2{"SKU<br/>в справочнике?"}
    Q2 -- "Нет" --> A3["Заявка админу:<br/>добавить SKU<br/>(BR-04 — только админ)"]
    A3 --> EndA((Прервано))
    Q2 -- "Да" --> A4["Ввести: количество,<br/>цена партии, дата,<br/>срок годности, поставщик"]
    A4 --> A5["Фото накладной<br/>в Yandex Object Storage"]
    A5 --> A6["Сохранить:<br/>остаток += qty<br/>movement = 'income'"]
    A6 --> EndIncome((Принято))

    %% Списание
    Branch2["B1. Списание на заказ<br/>(Складщик · Mobile-PWA)"]
    Branch2 --> B2["Поиск заказа #N<br/>(by ID / QR / запрос производства)"]
    B2 --> Q3{"Заказ<br/>существует и<br/>в статусе<br/>В производстве?"}
    Q3 -- "Нет" --> B3["Отказ:<br/>BR-01 запрещает списание<br/>без активного заказа"]
    B3 --> EndB((Прервано))
    Q3 -- "Да" --> B4["Выбрать материал<br/>+ количество"]
    B4 --> Q4{"Остаток<br/>достаточен?"}
    Q4 -- "Нет" --> B5["Алерт закупщику<br/>+ блок списания"]
    B5 --> EndB
    Q4 -- "Да" --> B6["Подтвердить списание"]
    B6 --> B7["Сохранить:<br/>остаток -= qty<br/>movement = 'production'<br/>+ order_id + employee_id<br/>+ stage + timestamp<br/>(BR-01 enforce)"]
    B7 --> EndOutcome((Списано))

    %% Инвентаризация
    Branch3["C1. Создан акт инвентаризации<br/>(Админ · Web)<br/>статус: В работе"]
    Branch3 --> C2["Складщик в Mobile-PWA:<br/>список SKU с системными остатками"]
    C2 --> C3["По каждой позиции:<br/>физический пересчёт<br/>+ ввод факта"]
    C3 --> Q5{"Расхождение?"}
    Q5 -- "Нет" --> C5
    Q5 -- "Да" --> C4["Указать причину:<br/>списание / излишек /<br/>ошибка / порча"]
    C4 --> C5["Следующая позиция"]
    C5 --> Q6{"Все<br/>позиции?"}
    Q6 -- "Нет" --> C3
    Q6 -- "Да" --> C6["Закрыть акт<br/>корректировочные движения<br/>movement = 'inventory_adj'"]
    C6 --> EndInv((Инвентаризация закрыта))
```

**Пояснение.**

- **BR-01 enforce.** Бэк отказывает в `MaterialMovement` если
  `type = 'production'` и `order_id IS NULL`. Список разрешённых
  типов: `income` (приёмка), `production` (на заказ — обязателен
  `order_id`), `sale` (продажа товара — обязателен `order_id` тип
  «продажа-товар»), `inventory_adj` (корректировка по акту), `defect`
  (брак — обязателен `order_id` исходного заказа).
- **BR-04 enforce.** Создавать SKU может только админ (модуль 6.4).
  Складщик при приёмке нового материала **не может** создать SKU
  на лету — только заявка админу.
- Сканер штрих-кода — Web Platform API `BarcodeDetector` (поддержка
  Chrome / Edge на Android). Для iOS PWA — fallback на ручной ввод
  или фотография ID карточки материала.

**Точки риска:** O3 (антидубль материалов), O5 (справочник с нуля),
T6 (offline-mode при приёмке без Wi-Fi на складе).

## P5. Контроль качества и брак (BR-03)

> **ТЗ § 5.1.1.13–14** + модули 6.12, 6.13. Складщик — **единственный**
> фиксатор брака. Эта инвариант устраняет конфликт интересов.

```mermaid
flowchart TD
    Start((Производство передало<br/>готовый заказ на склад))
    Start --> S1["Складщик принимает физически<br/>(Mobile-PWA)"]
    S1 --> S2["Открыть карточку заказа #N<br/>проверить соответствие макету"]
    S2 --> Q1{"Качество<br/>OK?"}
    Q1 -- "OK" --> Done["Статус: Готов к выдаче<br/>WebPush менеджеру"]
    Done --> EndOK((готов))

    Q1 -- "Брак" --> S3["Фиксировать брак<br/>(BR-03 — ТОЛЬКО складщик)"]
    S3 --> S4["Привязка к заказу #N<br/>(обязательно)"]
    S4 --> S5["Фото брака<br/>в Yandex Object Storage<br/>(обязательно)"]
    S5 --> S6["Указать количество<br/>испорченного"]
    S6 --> S7["Указать этап:<br/>дизайн / производство /<br/>материал / монтаж"]
    S7 --> S8["Указать ответственного:<br/>сотрудник или партия"]
    S8 --> S9["Сохранить запись Defect<br/>статус заказа:<br/>Брак — на переделку"]
    S9 --> S10["WebPush менеджеру + нач. цеха<br/>+ Email-копия"]
    S10 --> Q2{"Виновник?"}
    Q2 -- "Сотрудник" --> S11A["Связь с расчётом ЗП:<br/>штраф (BR-05 при настройке)"]
    Q2 -- "Партия материала" --> S11B["Алерт закупщику:<br/>претензия поставщику<br/>(вне платформы)"]
    Q2 -- "Дизайн" --> S11C["Возврат дизайнеру<br/>статус: На дизайне"]
    Q2 -- "Монтаж" --> S11D["Локальная переделка монтажа<br/>(не всего заказа)"]

    S11A --> S12
    S11B --> S12
    S11D --> S12
    S11C --> S7c["Дизайнер исправляет макет<br/>возврат к P1.1"]
    S7c --> EndC((Возврат к дизайну))

    S12["Повторное производство<br/>(модуль 6.13)<br/>заказ возвращается в очередь"]
    S12 --> S13["Списание материалов<br/>повторно на тот же заказ<br/>(BR-01, type=defect_rework)"]
    S13 --> S14["Производство выполняет<br/>заново (P1.2 шаг 12-14)"]
    S14 --> Start
```

**Пояснение.**

- **BR-03 enforce.** UI «Фиксировать брак» виден **только** роли
  «Складщик» и «Администратор» (для админ-исправлений). Производство,
  дизайн, менеджер не могут создать запись `Defect`.
- **Обязательность фото.** Без фото запись `Defect` не сохраняется
  (валидация на бэке). Фото — в Yandex Object Storage с привязкой к
  `defect_id` и `order_id`.
- **Этап брака** — фиксируется в `Defect.stage`: `design` / `production`
  / `material` / `assembly`. По этапу — разная маршрутизация
  переделки.
- **Связь с ЗП.** При `stage = production` и установленном
  `responsible_employee_id` — в фоновую задачу расчёта ЗП передаётся
  штраф (формула штрафа уточняется при настройке `apps/payroll`,
  продолжение 🔴 #8).
- **Bottleneck-риск (O2).** При одном складщике — очередь «На
  проверке» может расти. Митигация — дублирующий складщик в смене
  или роль «помощник» в фазе 2+.

**Открытые вопросы:**
- Что делать с браком на этапе **материала** — кто оплачивает повторное
  списание (типография / поставщик)? Зависит от условий контракта
  с поставщиком.
- Возможны ли «частичные» браки (50% тиража годен, 50% брак)? ТЗ
  однозначно не говорит. Нужна модель: `Defect.qty` и `Order.qty_total`
  — `Order.qty_done` после переделки.

## P6. Постобслуживание клиента

> **ТЗ § 5.2.3, модуль 6.17.** Через 1-2 дня после выдачи — задача-
> звонок для сбора обратной связи.

```mermaid
flowchart TD
    Start((Заказ → статус<br/>'Завершён'))
    Start --> S1["Триггер cron / scheduler:<br/>через 1-2 дня создать<br/>задачу 'Постобслуживание #N'"]
    S1 --> S2["Задача попадает<br/>офисному менеджеру<br/>(в фазе 3+ — колл-центр)"]
    S2 --> S3["WebPush + Email менеджеру<br/>'Позвонить клиенту по заказу #N'"]
    S3 --> Q1{"Канал<br/>обратной связи?"}
    Q1 -- "Звонок (MVP)" --> A1["Менеджер звонит клиенту"]
    Q1 -- "Авто-опрос (фаза 2+)" --> A2["WebPush + Email клиенту:<br/>оценка 1-5 + комментарий"]
    A1 --> S4["Менеджер заполняет:<br/>оценка 1-5,<br/>комментарий клиента,<br/>дата звонка, кто звонил"]
    A2 --> S4
    S4 --> Q2{"Оценка<br/>≤ 3?"}
    Q2 -- "Да" --> S5["Эскалация:<br/>WebPush + Email<br/>нач. цеха + учредителю"]
    Q2 -- "Нет" --> S6
    S5 --> S6["Сохранить запись<br/>Feedback"]
    S6 --> S7["Метрики:<br/>NPS, средняя оценка,<br/>% постобслуженных заказов"]
    S7 --> End((Закрыто))
```

**Пояснение.**

- **Канал в MVP — звонок.** Менеджер открывает задачу в Web-кабинете,
  звонит, заполняет форму. В фазе 3+ — колл-центр (отдельный модуль).
- **Авто-опрос — фаза 2+.** WebPush в кабинете с короткой формой
  «Оцените заказ 1-5 + комментарий». Email-копия. Telegram **не
  используется** (🔴 #7).
- **Эскалация при низкой оценке** — алерт нач. цеха + учредителю.
  Низкая оценка может говорить о скрытом браке, который не выявил
  складщик.
- **Метрики** — попадают в Owner Dashboard (J11): NPS, средняя оценка,
  % постобслуженных заказов.

**Открытые вопросы:**
- При оценке ≤ 3 — обязателен ли follow-up звонок учредителя? Или
  достаточно эскалации в систему?
- Сколько раз пытаться дозвониться, если клиент не отвечает?

## P7. Сдельная ЗП — конец периода

> **Модуль 6.22, расчёт 7.15, BR-05.** Двойная формула (тариф ₽ × кол-во
> **или** % от стоимости работы — 🔴 #8 закрыт). Активируется в
> фазе 2.

### P7.1. Расчёт за сотрудника

```mermaid
flowchart TD
    Start((Последний день периода<br/>23:59 — cron))
    Start --> S1["Для каждого сотрудника:<br/>собрать операции за период"]
    S1 --> S2["По каждой операции:<br/>тип расчёта (per operation/role)?"]
    S2 --> Q1{"Тип<br/>тарифа?"}
    Q1 -- "Фикс ₽" --> A1["сумма_op = ставка × количество"]
    Q1 -- "% от стоимости" --> A2["сумма_op = процент × work_cost_to_client"]
    A1 --> S3
    A2 --> S3["Σ заработано_за_период"]
    S3 --> S4["Применить штрафы за брак<br/>(если есть Defect.responsible = emp)"]
    S4 --> S5["Применить бонусы<br/>(переработка / сложность)"]
    S5 --> S6["Получили: 'Заработано'"]
    S6 --> Q2{"Заработано ><br/>= минимум?"}
    Q2 -- "Да" --> A3["Выплата = Заработано<br/>− долг прошлых периодов<br/>(если был)"]
    Q2 -- "Нет" --> A4["Выплата = минимум<br/>Долг = минимум − Заработано<br/>(BR-05: займ, не отрицательная ЗП<br/>ТК ст. 137)"]
    A3 --> S7
    A4 --> S7["Сохранить расчёт<br/>в apps/payroll"]
    S7 --> S8["Генерация расчётного листка<br/>PDF в Yandex Object Storage<br/>(ТК ст. 136 — обязательно)"]
    S8 --> S9["WebPush + Email сотруднику<br/>'Расчёт за период готов'"]
    S9 --> End((Сотрудник видит расчёт<br/>в Mobile-PWA))
```

### P7.2. Статус-машина расчёта периода

```mermaid
stateDiagram-v2
    [*] --> Открыт
    Открыт --> ВРасчёте: cron 23:59 LastDay
    ВРасчёте --> Готов: расчёт завершён,<br/>листок сгенерирован
    Готов --> Просмотрен: сотрудник открыл<br/>в Mobile-PWA
    Просмотрен --> ОжидаетВыплаты: бухгалтер ставит<br/>'к выплате'
    ОжидаетВыплаты --> Выплачено: банковский платёж<br/>проведён
    Выплачено --> Закрыт: баланс обновлён<br/>(BR-05)
    Закрыт --> [*]

    Готов --> Оспорен: сотрудник<br/>не согласен
    Оспорен --> ВРасчёте: пересчёт<br/>(админ + журнал)
```

**Пояснение.**

- **Двойная формула (🔴 #8).** Per operation / role админ настраивает
  один из двух типов:
  - `tariff_fixed` — `сумма = rate × qty` (например, 50 ₽ × 10 шт.
    напечатанных листов).
  - `tariff_percent_of_work` — `сумма = percent × work_cost_to_client`
    (например, 8% от стоимости работы для клиента; при изменении
    прейскуранта — заморозка ставки на дату операции через audit-log).
- **BR-05 — ТК ст. 137.** Если за период заработал меньше минимума —
  компания доплачивает разницу как **займ** (`balance += минимум −
  заработано`). В следующем периоде — погашение из заработанного.
  Никогда не уход в минус — выплата всегда ≥ 0.
- **Расчётный листок (ТК ст. 136).** Обязательная генерация PDF
  с детализацией: операции, тарифы, штрафы, бонусы, итого,
  баланс / долг. Хранение 5 лет (402-ФЗ).
- **Тесты на крайних случаях.** 0 операций (минимум только), `balance`
  > 0 (есть долг с прошлого), увольнение с долгом (списание долга или
  взыскание через суд — отдельный workflow), переполнение лимита
  удержаний 20% по ст. 138.

**Точки риска:** T5 (сдельная ЗП vs ТК — критический риск, скор 15).

**Открытые вопросы:**
- Минимум за период — сумма (₽/мес)? Уточняется примерами расчёта.
- Штрафы за брак — фикс ₽, % от заказа или удержание стоимости
  материала? Уточняется при настройке `apps/payroll`.
- Бонусы — за переработку / сложные заказы / выслугу лет — какие
  правила? Уточняется.

## P8. Face Control — события и корректировка

> **Модуль 6.20, BR-06.** События **не редактируются** сотрудником,
> только просмотр. Корректировка времени смены — через отдельный
> workflow с журналом.

### P8.1. Sequence — поток событий vendor → backend

```mermaid
sequenceDiagram
    participant Camera as Face Control SDK<br/>(vendor TBD 🔴 #2)
    participant Adapter as apps/face_control_adapter<br/>(provider abstraction)
    participant Backend as Backend
    participant DB as PostgreSQL<br/>+ Yandex Object Storage<br/>(шаблоны)
    participant Audit as Audit Log
    participant UI as Owner Dashboard<br/>+ Сотрудник Mobile-PWA

    Camera->>Adapter: Событие 'Вход' / 'Выход'<br/>(employee_id, timestamp,<br/>confidence, photo_ref)
    Adapter->>Backend: Нормализованное событие<br/>FaceControlEvent
    Backend->>DB: Сохранить event<br/>read-only для employee
    Backend->>Audit: Audit log: type=face_event,<br/>actor=system, employee_id
    Backend->>UI: WS push: статус<br/>'на смене / не на смене'
    Note over Backend,DB: BR-06: события<br/>не редактируются<br/>сотрудником

    UI->>Backend: Сотрудник просматривает<br/>свои события (read-only)
    Backend->>UI: Список событий<br/>+ время смены<br/>+ простой = присутствие − работа
```

### P8.2. Корректировка времени смены — workflow с журналом

```mermaid
flowchart TD
    Start((Сотрудник заметил<br/>пропуск Face Control<br/>например, забыл пройти))
    Start --> S1["Сотрудник в Mobile-PWA:<br/>'Запросить корректировку времени'"]
    S1 --> S2["Заполняет:<br/>фактическое время прихода/ухода,<br/>причина (текст),<br/>свидетели (если есть)"]
    S2 --> S3["Запись TimeCorrectionRequest<br/>статус: На рассмотрении"]
    S3 --> S4["WebPush + Email админу /<br/>нач. цеха"]
    S4 --> S5["Админ смотрит:<br/>оригинальные события Face Control,<br/>запрос сотрудника,<br/>журнал прошлых корректировок"]
    S5 --> Q1{"Утвердить<br/>корректировку?"}
    Q1 -- "Отклонить" --> R1["Статус: Отклонён<br/>+ причина отказа"]
    R1 --> NotifyEmp1["WebPush + Email сотруднику"]
    NotifyEmp1 --> EndR((Не применено))
    Q1 -- "Утвердить" --> A1["Создать запись<br/>FaceControlOverride<br/>(не изменяет оригинальный<br/>FaceControlEvent — тот неизменен!)"]
    A1 --> A2["В audit-log:<br/>actor=admin, action=time_override,<br/>original_event_id, new_time,<br/>reason, journal_entry"]
    A2 --> A3["Пересчёт смены:<br/>с учётом override<br/>(но оригинал виден в журнале)"]
    A3 --> NotifyEmp2["WebPush + Email сотруднику<br/>'корректировка применена'"]
    NotifyEmp2 --> EndA((Применено))
```

**Пояснение.**

- **BR-06 enforce.** Таблица `FaceControlEvent` — **append-only**.
  Бэк не предоставляет эндпойнтов на UPDATE / DELETE для этой таблицы
  (только админский SQL-доступ для исправления багов с журналированием).
- **Корректировка через override.** Если оригинальное событие не
  должно (по уважительной причине) применяться — создаётся отдельная
  запись `FaceControlOverride`. Расчёт времени смены берёт оригинал
  + applicable overrides. В UI сотрудник видит и оригинал, и override
  с указанием «скорректировано админом, причина: ...».
- **Журнал корректировок.** Все `TimeCorrectionRequest` и
  `FaceControlOverride` — в audit-log. История доступна владельцу
  (для контроля «не злоупотребляет ли админ»).
- **Биометрические шаблоны** хранятся в **нашем** Yandex Object Storage
  (не у вендора), шифрование at-rest, отдельный KMS-ключ через
  Yandex Lockbox (после ADR-0003 хостинга).
- **152-ФЗ ст. 11.** Перед первым включением Face Control сотруднику
  — обязательное **письменное** согласие (бумага + Mobile-PWA) с
  timestamp + хеш версии политики. Без согласия — no enrollment.

**Точки риска:** T1 (vendor-lock), T2 (точность Face Control при
потоке), T3 (биометрия 152-ФЗ ст. 11), T10 (утечка биометрии).

**Открытые вопросы:**
- Кто авторизует override — нач. цеха или только админ?
  Если нач. цеха — риск злоупотребления для своих сотрудников.
- Лимит на количество корректировок в месяц на сотрудника? (например,
  не более 2 — иначе triggered review учредителя).
- При увольнении сотрудника — что делать с биометрическим шаблоном?
  152-ФЗ ст. 14 = удаление по запросу субъекта; нужен явный workflow.
