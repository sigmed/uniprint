# 08 · UniPrint · Риск-карта

> Living-doc. Реестр рисков проекта UniPrint с матрицей вероятность ×
> impact, триггерами, митигацией, ownership. Обновляется на каждой
> вехе спринта или при появлении нового риска. Учитывает решения
> владельца от 2026-05-05 (см. `onboarding/owner-questions.md`).

---

## Как читать матрицу

- **Вероятность (P)** — 1-5: 1 = маловероятно, 5 = почти наверняка.
- **Impact (I)** — 1-5: 1 = некритичен, 5 = ставит под удар проект /
  бизнес / compliance.
- **Риск-скор** = P × I. Скор ≥ 15 = критический риск, ≥ 9 — высокий,
  ≥ 4 — средний, < 4 — низкий.
- **Триггеры** — что говорит о реализации риска (раннее предупреждение).
- **Митигация** — что мы делаем, чтобы снизить P или I.
- **Owner** — роль из [`team-structure.md`](team-structure.md).
- **Категория:**
  - 🔧 **T**echnical — технические / архитектурные / vendor-lock
  - 📜 **R**egulatory — compliance / законодательство РФ
  - ⚙️ **O**perational — операционные / процессные / людские
  - 💰 **C**ommercial — коммерческие / финансовые / market fit
  - 👥 **G**overnance — организационные / управленческие

## Топ-5 критических рисков (разбор)

### 1. O1 — Миграция legacy-данных (скор 16) 🔴

**Почему критический.** Без корректных справочников MVP не запустить;
дубли клиентов / неверные нормативы → ломаются BR-01, BR-02, BR-04
одновременно. На пилоте мы получим жалобы пользователей и потерю
доверия к системе с первого дня. Связан с O5 (справочник с нуля)
и зависит от 🔴 #4.

**Сценарии.**
- *Best case:* у владельца есть 1С с экспортом, мы делаем шаблон,
  владелец верифицирует, импорт за 1-2 дня.
- *Realistic:* частично 1С, частично Excel, частично «в голове» —
  параллельный track 2-4 недели с R3 и владельцем.
- *Worst case:* данных нет → справочник собирается с нуля силами
  R3 + владельца → 4-8 недель параллельно.

**План действий.**
1. Закрыть 🔴 #4 (источник legacy) до старта sprint-1.
2. Выделить **отдельный sprint-track «справочник + миграция»**
   параллельно с MVP-кодом (sprint-0 уже стартовал по 🔴 #15).
3. Тестовая миграция на стейдже **до** prod.
4. Cut-over date: старые заказы завершаются в старой, новые открываются
   в UniPrint. Не пытаемся мигрировать активные.
5. Parallel-run первого месяца: старая система работает на чтение,
   новая на запись и чтение. Контрольные точки сверки.

### 2. O4 — Сотрудники саботируют систему (скор 16) 🔴

**Почему критический.** Если производственники / складщики не
пользуются Mobile-PWA, мы получаем «систему для отчётности», а не
«систему для управления». Все KPI становятся фикцией.

**Сценарии.**
- *Best case:* сдельная ЗП мотивирует — нет отметки «Завершить» = нет
  заработка → 100% adoption за неделю.
- *Realistic:* 60-70% adoption через 2-4 недели после онбординга,
  остальные «забывают» → нач. цеха давит.
- *Worst case:* саботаж лидерами мнений в коллективе → adoption < 30%,
  система фактически не работает → owner отказывается продолжать.

**План действий.**
1. UX-исследование на этапе спеки (cs-ux-researcher с производственниками).
2. **Экономический стимул**: BR-05 (баланс) и сдельная ЗП — отметился
   = заработал.
3. Touch-target ≥ 44pt, минимум кликов, perchatka-friendly UI.
4. Пилот на малой группе (один цех, 3-5 человек) → итерация до раскатки.
5. Онбординг + обучающие видео в Mobile-PWA.
6. KPI нач. цеха = adoption его команды.
7. Гибкость для краевых случаев (что делать, если сотрудник забыл
   нажать «Начать» — корректировка времени через workflow с журналом,
   как в BR-06).
8. PWA install-prompt и помощь «как поставить иконку на главный экран».

### 3. O5 — Справочник с нуля (скор 16) 🔴

**Почему критический.** По 🔴 #15 — справочник услуг (~30) и материалов
(~200 SKU) собирается **с нуля** силами владельца + R3 (типограф-
консультант). Если справочник не готов — не запустить sprint-1, не
протестировать BR-04 (запрет ручного создания услуг), не оформить
тестовые заказы.

**Сценарии.**
- *Best case:* R3 опытный, владелец чётко знает свой ассортимент →
  справочник готов за 2 недели.
- *Realistic:* 4 недели, нескольких раундов уточнений, дополнения по
  ходу sprint-1.
- *Worst case:* R3 недоступен / занят, владелец не может выделить время
  → задержка sprint-1 на 4-6 недель.

**План действий.**
1. Sprint-0 параллельный track «Справочник с нуля» — стартует
   немедленно.
2. Шаблон Excel от нас (категории / параметры / нормы расхода / нормы
   времени).
3. Контракт с R3 на 80-120 ч в discovery + 20-40 ч на ревью первого
   месяца.
4. Дефолтные данные для прототипа на моках (~30 услуг + ~50 SKU) —
   не блокируем демо.
5. Validation gate перед стартом sprint-1: «справочник полнота
   ≥ 80% услуг и ≥ 80% SKU» — иначе sprint-1 откладывается.
6. Admin tool для быстрого добавления SKU в продакшене (BR-04 =
   только админ).

### 4. T5 — Сдельная ЗП vs ТК РФ (скор 15) 🔴

**Почему критический.** Любая ошибка в формуле или нарушение ТК ст. 137
(удержание в минус) — судебный иск от сотрудника. Прецедент для всего
коллектива → массовые иски → крах операционки. Двойная формула
(тариф × кол-во **или** % от стоимости работы) добавляет
комбинаторику тестов.

**Сценарии.**
- *Best case:* формула проверена юристом + CFO, BR-05 правильно
  реализован, тесты на крайних случаях → 0 инцидентов.
- *Realistic:* 1-2 жалобы на расчёт за первый год, разбираются в
  частном порядке.
- *Worst case:* массовая жалоба → проверка трудовой инспекции →
  штрафы + восстановление сумм → reputation hit.

**План действий.**
1. Получить **примеры расчёта** по 2-3 типам сотрудников от владельца
   (продолжение 🔴 #8 — детализация формулы) **до** запуска модуля 6.22.
2. Юрист-аутсорс валидирует соответствие ТК (ст. 136, 137, 138).
3. CFO-advisor + Финансовый аналитик прогоняют формулу на 6-12 месяцах
   реальных данных (если есть).
4. **Тесты на крайних случаях**: 0 операций за месяц, отрицательный
   баланс, переполнение лимита удержаний (20% по ст. 138), увольнение
   с долгом, болезнь / отпуск, изменение прейскуранта посередине
   периода (для % от стоимости).
5. Расчётный листок (PDF) для каждого периода — обязательно (ст. 136).
6. `Docs/compliance/payroll-tk-compliance.md` — explicit чеклист.

### 5. T1 — Face Control vendor-lock (скор 15) 🟡

**Почему критический.** Биометрические шаблоны привязаны к формату
конкретного SDK. Смена вендора = пересобирать шаблоны (повторно
сканировать каждого сотрудника = повторное согласие 152-ФЗ ст. 11 =
огромный operational hit).

**Сценарии.**
- *Best case:* выбран NtechLab FindFace (РФ-вендор), 5+ лет работает
  стабильно → vendor-lock не проявится.
- *Realistic:* вендор поднимает цены каждые 2 года, мы платим, но
  это не критично.
- *Worst case:* санкции / банкротство вендора → срочная миграция на
  другого вендора → 1-2 месяца переподписания согласий и пересканирования.

**План действий.**
1. ADR-0002 с провайдер-абстракцией для 2-3 вендоров.
2. Хранение шаблонов в **нашем** Yandex Object Storage, не у вендора
   (где возможно).
3. Если вендор закрытый (только облако) — exit-план в ADR.
4. Контракт с SLA по доступности SDK ≥ 99%.
5. Регулярный re-evaluation вендоров раз в год.
