## 1. Регуляторная карта РФ для типографии

> **Ключевое отличие от HoReCa:** UniPrint — типография + наружная
> реклама. **НЕ применимы**: ХАССП, ЕГАИС, МДЛП, ТР ТС 022/2011 (БЖУ +
> аллергены), маркетинговые vs транзакционные SMS-тарифы. Регуляторика
> другая, ключевое — **152-ФЗ + биометрия** (Face Control).

| Закон / норма | Применимость | Зона ответственности |
| --- | --- | --- |
| **152-ФЗ** «О персональных данных» | **Главный** для UniPrint — есть ПДн сотрудников и клиентов + биометрия | См. § 2, § 3 |
| **ТК РФ** ст. 136, 137, 74 | Сдельная ЗП + расчётный лист + баланс | См. § 4 |
| **402-ФЗ** «О бухгалтерском учёте» | Хранение первички (счета / акты / ТТН) — 5 лет | См. § 5 |
| **54-ФЗ** «О ККТ» | **Применим** (Q14 ✅ — есть B2C). Онлайн-чеки через ОФД обязательны. | См. § 6 |
| **44-ФЗ / 223-ФЗ** | Только если работаем с госконтрактами | См. § 7 (фаза 2+) |
| ГК РФ ч. IV (авторское право на макеты) | Макеты клиентов — собственность клиента; договор-оферта фиксирует условия использования | Юрист (out of scope для софта) |

**Out of scope для UniPrint** (ради ясности):

- ХАССП / ТР ТС 022/2011 (только HoReCa)
- ЕГАИС (только алкоголь)
- МДЛП (только лекарства)
- 218-ФЗ КИИ (становится применим, если попадаем в реестр критической
  инфраструктуры — для UniPrint **неприменимо**)
- ФЗ-149 «Об информации» в части локализации новостных аггрегаторов —
  не наш кейс

## 2. 152-ФЗ — персональные данные

### 2.1 Какие ПДн в системе

| Категория | Где | Тип |
| --- | --- | --- |
| ПДн сотрудников | таблицы `users`, `employees`, `pay_slips` | ФИО, телефон, email, паспорт, СНИЛС (для ЗП), банк-реквизиты |
| **Биометрия сотрудников** | `face_control` + KMS | **Спец. категория** — см. § 3 |
| ПДн клиентов (физ. лиц) | `clients`, `orders` | ФИО, телефон, email, адрес доставки |
| ПДн клиентов (юр. лиц / ИП) | `clients`, `contracts` | Реквизиты компании + контактное лицо (ФИО, телефон) |
| Геолокация водителя | `logistics` (в момент доставки) | Координаты GPS с timestamp |

### 2.2 Ключевые требования

#### Ст. 18 ч. 5 — локализация

ПДн граждан РФ хранятся и обрабатываются **на серверах в РФ**. Запрет
на использование AWS / GCP / Cloudflare R2 / Hetzner / OVH для
production-инфраструктуры с реальными ПДн.

**Допустимо для prod (после ADR-0003):**
- Yandex.Cloud (default)
- Selectel
- VK Cloud
- on-prem (своё железо в РФ)

**Прототип на Vercel** допустим только потому, что данные
синтетические (mock); реальные ПДн **запрещены** к вводу — фиксируется
в `prototype/MOCK_NOTICE.txt`.

`TBD после 🔴 #3 — см. Docs/onboarding/owner-questions.md § Q3` →
ADR-0003.

#### Ст. 9 — общее согласие на ПДн

- Каждый сотрудник подписывает согласие при найме.
- Каждый клиент-физ. лицо подписывает согласие при регистрации в
  клиентском кабинете (галочка с текстом политики + хэш версии).
- Юр. лица — не нужно (ПДн нет, только реквизиты).
- Хранение: `consents` таблица, каждое согласие = (user_id, type,
  timestamp, policy_version_hash, signature_method).

#### Ст. 11 — биометрия (отдельный § 3)

#### Ст. 14 — endpoint удаления ПДн

- Любой субъект ПДн вправе требовать удаления своих данных.
- Реализация: `DELETE /api/v1/users/{id}/personal-data` + cron-задача
  на cleanup истёкших ПДн (для уволенных сотрудников по сроку хранения).
- **Исключение:** первичные документы (счета / акты / ТТН) —
  хранятся 5 лет по 402-ФЗ; запрос на удаление откладывается до
  истечения срока.

#### Ст. 22 — уведомление РКН

Оператор ПДн (наша организация-клиент UniPrint) обязан подать
уведомление в РКН через Госуслуги **до начала обработки ПДн**.
Срок рассмотрения ~30 дней.

**Подаёт клиент**, не команда разработки (это юридическая обязанность
оператора). Команда обеспечивает шаблон в `Docs/compliance/rkn-notification.md`.

#### Audit-log

Все доступы и изменения ПДн фиксируются в `audit_log_entries`:
- timestamp + actor (user_id) + action (read / update / delete) +
  object (table + id) + ip + user_agent
- Append-only (нет DELETE / UPDATE) — для целостности.
- Доступ к audit-log — только role=admin.
- Срок хранения audit-log: ≥ 3 года (рекомендация РКН).

## 3. Биометрия — ключевой риск (152-ФЗ ст. 11)

> **Это самый важный compliance-риск проекта.** Биометрические ПДн —
> специальная категория, требующая повышенной защиты. Несоответствие =
> штраф до 18 млн ₽ + repeated offense.

### 3.1 Что считается биометрией

В контексте Face Control:
- Фотографии лица в HD/4K, пригодные для распознавания
- **Вычисленные биометрические шаблоны** (vector embeddings, geometric
  templates) — даже без исходных фото

### 3.2 Требования 152-ФЗ ст. 11

1. **Отдельное письменное согласие** сотрудника **до** регистрации
   шаблона. Общее согласие на ПДн (ст. 9) **не покрывает** биометрию.
2. **Локализация хранения** в РФ (как все ПДн).
3. **Шифрование at-rest** — обязательно (KMS, ротация ключей).
4. **Audit-log на каждый доступ** к шаблону / `verify` / `register`.
5. **Право отзыва согласия** в любой момент → удаление шаблона
   (`deleteTemplate`) в течение 30 дней.
6. **Запрет передачи третьим лицам** без отдельного согласия (включая
   обмен между vendor'ами).

### 3.3 Стандарты vendor'ов (TBD ADR-0002)

`TBD после 🔴 #2 — см. Docs/onboarding/owner-questions.md § Q2`.
Особенности хранения шаблонов у разных vendor'ов:

| Vendor | Хранение шаблонов | Локальная обработка | Замечания |
| --- | --- | --- | --- |
| **NtechLab FindFace** | On-prem или managed Yandex.Cloud (РФ) | Да | Default-рекомендация для РФ — нет проблем с локализацией |
| **Hikvision DeepInMind** | На устройстве (камере) или сервере | Частично | Производитель — Китай; санкционные риски + риск передачи данных за рубеж |
| **Suprema BioStar 2** | На сервере или устройстве | Да | Корея; на момент 2026 — нет санкционных проблем |
| **Самописный CV (OpenCV + InsightFace)** | Локально, под нашим контролем | Полностью | Полный контроль над локализацией, но ML-инженер обязателен (фаза 2+) |

### 3.4 Где храним шаблоны

| Решение | Когда | Описание |
| --- | --- | --- |
| **Yandex Lockbox** (default при cloud-хостинге) | ADR-0003 → cloud | Managed KMS в РФ, нативная интеграция с другими сервисами Yandex.Cloud |
| **on-prem KMS** (HashiCorp Vault или эквивалент) | ADR-0003 → on-prem | Если хостинг on-prem |
| **Защищённая отдельная таблица в PG** с шифрованием через KMS-managed ключ | minimal MVP | Если Lockbox / Vault недоступны |

**В любом случае:** шаблоны **не хранятся** в основной БД рядом с
обычными ПДн. Изоляция — на уровне хранилища И на уровне доступа
(только модуль `face_control_adapter` имеет креды).

### 3.5 Шаблон формы согласия

`TBD после ADR-0002 — Docs/compliance/policies/biometry-consent.md`.

Минимальный обязательный набор полей:

- ФИО сотрудника + дата рождения
- Цель обработки (контроль присутствия на рабочем месте)
- Vendor + наименование биометрической системы
- Срок обработки (срок трудового договора + 30 дней на удаление)
- Право на отзыв согласия в любой момент
- Подпись (физическая или ЭЦП)
- Ссылка на политику обработки ПДн (с версией / хэшем)

### 3.6 Архитектурное следствие

См. `Docs/03-architecture.md` § 4.3 (`face_control_adapter`) и
§ 5 (БД-таблица `face_control_events` хранит **только timestamps +
employee_id + type**, без шаблонов).

## 4. ТК РФ — сдельная ЗП и баланс сотрудника

### 4.1 Контекст

UniPrint реализует **сдельную систему оплаты** (модуль § 6.22, формулы
§ 7.12, § 7.15) с механикой долгового баланса (BR-05): если за период
сотрудник заработал меньше минималки — компания доплачивает в долг,
в следующий период разница компенсируется.

### 4.2 Ст. 136 — расчётный лист

**Требование:** работодатель обязан выдавать расчётный лист **при
каждой выплате** заработной платы. Содержит:
- Состав заработной платы (за что начислено)
- Размер удержаний (НДФЛ, алименты, штрафы)
- Размер к выплате
- Период

**Реализация:** модуль `payroll` генерирует PDF расчётного листа
(шаблон `Docs/compliance/templates/pay-slip.html`) на каждую выплату,
отправляет сотруднику по Email (см. `Docs/05-integrations.md` § 12).

### 4.3 Ст. 137 — ограничения по удержаниям

**Требование:** удержания из ЗП не могут превышать 20% (общее правило)
или 50% (по нескольким исполнительным документам). **Зарплата не
может уйти в минус.**

**Следствие для BR-05:** «долговой баланс» сотрудника — это **займ от
компании**, а не уход ЗП в минус. Юридически фиксируется как
беспроцентный займ или авансирование, оформляется отдельным
соглашением о порядке выплат (`Docs/compliance/policies/payroll-agreement.md` —
TBD).

**Технически:** в `payroll`:
- ЗП за период = max(заработано, минималка)
- Если заработано < минималки → разница попадает в `employee_balance`
  как `loan_balance` (долг сотрудника), не в `pay_slip` как удержание.
- В следующий период при заработке > минималки — `loan_balance`
  погашается частично, но **выплата не уходит ниже минималки**
  (можно также ограничить — не ниже определённого % от заработка).

### 4.4 Ст. 74 — изменение системы оплаты труда

**Требование:** изменение существенных условий труда (включая
систему оплаты) — уведомление сотрудника не менее чем за 2 месяца.

**Применимо при миграции** с фиксированной ЗП на сдельную (если
типография ранее работала на фикс). Не code-side, но фиксируем
в `Docs/compliance/policies/payroll-migration.md` (TBD).

### 4.5 НДФЛ

`apps/payroll` рассчитывает НДФЛ (13% / 15% выше порога) и отчитывает
в формате 6-НДФЛ (квартально) и 2-НДФЛ (годовая справка).
**Возможно делегировать** во внешнюю бухгалтерию (1С / Контур / СБИС)
через экспорт. TBD после Q15.

## 5. 402-ФЗ — бухгалтерский учёт

### 5.1 Ключевое требование

Хранение первичных документов (счета, акты, ТТН, накладные на
материалы) — **5 лет** с момента закрытия отчётного периода.

### 5.2 Реализация

- Модуль `docflow` (см. § 6.23) генерирует счета / договоры / акты /
  ТТН в PDF.
- Хранение PDF — в S3 (см. `Docs/05-integrations.md` § 7) с
  lifecycle policy: **запрет на удаление в течение 5 лет**
  (S3 object lock в режиме compliance).
- Метаданные документов — в таблице `documents` (с FK на `orders`,
  `clients`).
- При запросе на удаление ПДн (152-ФЗ ст. 14) — первичные документы
  **не удаляются** до истечения 5-летнего срока (сообщаем субъекту
  обоснование).

### 5.3 Электронная подпись

- Для **B2B** клиентов с ЭДО (Контур.Диадок / СБИС, фаза 2+) —
  усиленная квалифицированная ЭП (УКЭП) клиента.
- Для **MVP** — простая электронная подпись (ПЭП) или PDF без
  подписи + распечатка с печатью организации.

## 6. 54-ФЗ — онлайн-чеки

### 6.1 Применимость

Онлайн-чек обязателен **при приёме оплаты от B2C-клиентов**
(физических лиц) через эквайринг или наличные.

**Не применим** для B2B-расчётов по р/с (счёт-фактура + акт
заменяют чек).

### 6.2 Что блокирует

`TBD после 🔴 #5 + Q14 — см. Docs/onboarding/owner-questions.md §§ Q5, Q14`.

### 6.3 Реализация (если применимо)

См. `Docs/05-integrations.md` § 5 — `FiscalProvider` interface.

- Онлайн-касса (облачная или физическая) подключена к ОФД-провайдеру
  (Платформа ОФД / Такском / Яндекс ОФД / Первый ОФД).
- При успешном платеже от B2C — `FiscalProvider.emitReceipt`
  отправляет чек в ОФД, ОФД отправляет его клиенту по email/SMS
  (требует **email или phone в платеже** — см. `Docs/05-integrations.md` § 4.6).
- Чек должен быть выпущен **в течение 30 минут** после оплаты.
- При возврате — отдельный чек-возврат.

### 6.4 ENV-vars

См. `Docs/05-integrations.md` § 5.4.

### 6.5 Если только B2B

`FiscalProvider` остаётся `MockFiscal` (no-op), модуль не активен.
Чеки не нужны, достаточно счёта-фактуры (УПД) + акта выполненных
работ + платёжки от банка.

## 7. 44-ФЗ / 223-ФЗ — госконтракты (фаза 2+)

**Применимость:** если UniPrint работает с госзаказчиками (например,
печать агитматериалов для муниципалитета).

**Текущий статус:** out of scope для MVP. Рассматриваем в фазе 2+
если появится госконтракт.

**Что потребуется:**
- Регистрация в ЕИС zakupki.gov.ru
- Электронная подпись для участия в торгах
- Интеграция с ЭТП (если активные торги)
- Раздельный учёт для бюджетных средств

- [ ] BR-05 (долговой баланс) реализован как займ, не уход в минус
      по ст. 137 (тест)
- [ ] Соглашение о сдельной системе оплаты подписано каждым
      сотрудником (если миграция с фикса — за 2 месяца до старта
      по ст. 74)

### 8.5 Audit и backup

- [ ] Audit-log активен на доступы к ПДн (всё чтение + всё изменение)
- [ ] Audit-log хранится append-only ≥ 3 года
- [ ] Backup PG активен (PITR, RPO ≤ 1ч, RTO ≤ 4ч — целевые)
- [ ] Тест восстановления из backup пройден

### 8.6 Каналы связи

- [ ] **SMS буквенный sender зарегистрирован** (5–15 рабочих дней
      регистрации в каждом операторе) — Q7 ✅ SMS — основной канал
      OTP / нотификаций для клиентов
- [ ] DNS SPF / DKIM / DMARC настроены для email-домена
- [ ] WebPush VAPID-ключи сгенерированы и развёрнуты
- [ ] ✅ Q7: Telegram **не используется в продукте** — стратегия каналов
      без него (см. `Docs/05-integrations.md` § 12)

### 8.7 Платежи

- [ ] **🔴 Q5** — эквайринг выбран (ADR-0005)
- [ ] Договор с эквайером подписан
- [ ] **ОФД-провайдер подключён** (Q14 ✅ — B2C есть, ОФД обязателен)
- [ ] Sandbox-режим эквайринга протестирован (success / fail / refund)
- [ ] Email/phone клиента в платеже всегда передаются (для отправки
      электронного чека — 54-ФЗ требование)

      проверен (R3 — доменный консультант)
