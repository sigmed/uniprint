## Контекст в одном абзаце

Вы работаете над UniPrint ERP & Client System — платформой для типографии. Шесть фронтенд-приложений в Turborepo: `client-portal`, `manager-web`, `production-mobile` (PWA), `warehouse-mobile` (PWA), `admin-panel`, `owner-dashboard`. Стек: **Next.js 16 + React 19 + TypeScript + Tailwind 4 + shadcn/ui + MSW** для моков. Сейчас Phase-0 — прототип на Vercel preview с синтетическими данными. Реальные ПДн запрещены.

## Жёсткие правила (нарушение = PR отбрасывается)

### Цвета и шрифты

❌ **НЕ ДЕЛАЙТЕ:**
```tsx
<div style={{ color: '#D9531E', background: '#FBE6D9' }}>
<button className="bg-orange-500 text-white">
<span className="font-mono text-sm" style={{fontFamily: 'Roboto Mono'}}>
```

✅ **ДЕЛАЙТЕ:**
```tsx
<div className="text-brand bg-brand-soft">
<Button variant="primary">
<span className="font-mono">  // JetBrains Mono из tokens
```

### Компоненты

❌ **НЕ ДЕЛАЙТЕ** свой `<div className="rounded-full bg-green-100 px-2 py-1 text-xs">Готов</div>`.

✅ **ДЕЛАЙТЕ:** `<StatusPill status="done">Готов</StatusPill>`. Если status'а такого нет — **сначала** добавьте в enum, **потом** используйте.

### Mobile vs Desktop

❌ **НЕ ДЕЛАЙТЕ** production-mobile экран с сайдбаром, «потому что у других кабинетов так».

✅ **ДЕЛАЙТЕ:** проверьте `DESIGN_SYSTEM.md` § 6 — какой layout у этого кабинета. Production / warehouse — PWA mobile-first, у них **нет** сайдбара, у них есть `<PwaPhone>` и `<PwaTabBar>`.

### Бизнес-правила

❌ **НЕ ДЕЛАЙТЕ** в `production-mobile` кнопку «Зафиксировать брак», даже если «логично, что печатник может». BR-03 это запрещает.

✅ **ДЕЛАЙТЕ:** покажите `<BrCallout code="BR-03">` с текстом про то, что брак фиксирует только складщик. Кнопка — в `warehouse-mobile`.

### Прототип

❌ **НЕ ДЕЛАЙТЕ:**
- Удаление `<ProtoTag>` из шапки
- Использование реальных email/phone/ИНН
- Загрузку реальных макетов в S3
- Telegram-иконку или упоминание Telegram в любом месте

✅ **ДЕЛАЙТЕ:** синтетика только. `Иван Петров`, `+7 999 000 0001`, `UNI-2026-XXXXX`. Каналы: WebPush + Email + SMS.

## Алгоритм выполнения UI-задачи

### Шаг 1. Понять задачу

Прочитайте PRD/тикет. Извлеките:
- Какой кабинет? (`client-portal` / `manager-web` / `production-mobile` / `warehouse-mobile` / `admin-panel` / `owner-dashboard`)
- Какой user journey? (J1..J14 в `02-user-journeys.md`)
- Какие BR-XX задействованы?
- Это новый экран или модификация существующего?

Если хоть один пункт неясен — **остановитесь и задайте вопрос**, не угадывайте.

### Шаг 2. Найти существующие компоненты

Прежде чем писать новый компонент — проверьте `packages/ui/`. Список с описанием — в `DESIGN_SYSTEM.md` § 5.

Если кажется, что нужного компонента нет — **дважды проверьте**. Возможно, нужный компонент называется иначе, чем вы ожидаете. Например, для денежной суммы есть `<Amount>`, не нужно делать `<span className="font-serif">{value} ₽</span>`.

### Шаг 3. Использовать токены

Перед тем, как написать `className="..."`:
- Цвета: `bg-bg`, `bg-surface`, `text-ink`, `text-ink-3`, `border-line`, `text-brand`, `bg-brand-soft`, `text-green-ink` и т.д. (полный список в `tailwind.config.ts`)
- Шрифты: `font-display` (Fraunces), `font-sans` (Manrope), `font-mono` (JetBrains)
- Радиусы: `rounded`, `rounded-md`, `rounded-lg`, `rounded-xl`, `rounded-2xl`, `rounded-full`
- Тени: `shadow-card`, `shadow-card-lg`

### Шаг 4. Реализовать с проверкой

Каждые ~50 строк кода или каждый завершённый компонент:
1. Прогоните через [чек-лист](#чек-лист-перед-pr) ниже
2. Проверьте на брейкпоинтах: 1440, 1280, 900, 720, 380px
3. Если PWA — проверьте touch-targets ≥ 44pt
4. Если связано с BR-XX — проверьте, что правило отображено

### Шаг 5. PR

Заголовок PR: `[<кабинет>] <что сделано>`.

Например:
- `[client-portal] Форма создания заказа с типами BR-07`
- `[warehouse-mobile] Экран фиксации брака (BR-03) с фото-загрузкой`

В описании PR:
1. Какой PRD / тикет / journey
2. Какой кабинет
3. Какие BR-XX задействованы
4. Скриншоты на 1440px и 380px (для PWA)
5. Чек-лист пройден ([ниже](#чек-лист-перед-pr))

## Когда задавать вопросы вместо догадок

Спрашивайте **до того, как начнёте писать код**, если:

1. **Не понятно, какой это user journey.** «Сделать форму заказа» — недостаточно. Нужно знать: J1 (цех), J2 (офис) или J3 (товар)? Это разные формы.

2. **Конфликт с бизнес-правилом.** Если PRD говорит «добавь кнопку X», а BR-XX это запрещает — стоп. Спросите. Возможно, PRD устарел, возможно, правило надо обновить, возможно, вы неправильно поняли.

3. **Нет нужного компонента в design system.** Прежде чем создать новый, спросите: точно ли его нет? Может, можно расширить существующий вариантом?

4. **Mockup противоречит DESIGN_SYSTEM.md.** Документ старше дизайн-системы — баг в документе. Если макет младше — спросите автора, какие принципы изменились и нужно ли обновить design system.

5. **Открытый блокер 🔴 1–5 трогает вашу задачу.** Если делаете оплату — но 🔴 #5 эквайринг не закрыт — это блокер. Не угадывайте провайдера, не делайте «mock-кнопку Тинькофф», спросите.

## Когда **НЕ** задавать вопросы

Не спрашивайте про:
- Цвета — они в токенах
- Шрифты — они в токенах
- Компоненты — они в Storybook
- Layout — он в § 6
- Бизнес-правила — они в `BUSINESS_RULES.md`
- 3 типа заказа — они в `BUSINESS_RULES.md` § BR-07 и `02-user-journeys.md` J1/J2/J3

Если ответ есть в документации — найдите и используйте, не спрашивайте.

## Чек-лист перед PR

(зеркало раздела 9 в `DESIGN_SYSTEM.md` — копируйте его в описание PR)

```markdown
### Токены
- [ ] Нет хардкода цветов (`#XXXXXX`) — только токены
- [ ] Нет хардкода шрифтов — только Fraunces / Manrope / JetBrains Mono
- [ ] Все размеры из Tailwind-шкалы
- [ ] Все радиусы и тени из токенов

### Компоненты
- [ ] Использованы существующие компоненты
- [ ] Новый компонент создан только если нет подходящего
- [ ] Storybook-стори обновлён
- [ ] Иконки из `lucide-react`

### Бизнес-правила
- [ ] BR-XX, связанные с экраном, отображены в UI
- [ ] Поведение совпадает с `BUSINESS_RULES.md`

### Layout
- [ ] Desktop: проверено на 1280 / 1440 / 1920px
- [ ] Mobile-PWA: проверено на 380px, touch-targets ≥ 44pt
- [ ] Брейкпоинты сработали корректно

### Доступность
- [ ] Клавиатура: Tab, Enter, Esc
- [ ] Контраст ≥ AA
- [ ] aria-label на иконочных кнопках

### Прототип
- [ ] PROTOTYPE-бейдж на месте
- [ ] Данные синтетические
- [ ] Telegram не упоминается
- [ ] Каналы: WebPush / Email / SMS

### Контент
- [ ] ID формата `UNI-2026-XXXXX` (моноширинный)
- [ ] Денежные суммы — Fraunces, `1 500 ₽`
- [ ] Даты — `14.05.2026`, время — `14:32`
```

## Если что-то пошло не так

Признаки того, что вы заходите в тупик:

- ⚠️ Третий час правите CSS — стоп, посмотрите `DESIGN_SYSTEM.md`, скорее всего нарушили принцип
- ⚠️ Создали `style={{...}}` инлайном — стоп, переделайте через токены
- ⚠️ Скопировали цвет из макета пипеткой — стоп, найдите ближайший токен или добавьте новый в `colors.ts`
- ⚠️ Делаете компонент третий раз похожий на существующий — стоп, расширьте существующий
- ⚠️ Тесты падают на брейкпоинтах — стоп, layout-правила в § 6

В каждом случае: **перечитайте DESIGN_SYSTEM.md**.
