### Q2. Face Control — какой вендор?

**Варианты:**
- (а) **Hikvision** (камеры + DeepInMind SDK; HQ China — санкционные риски,
  но широко представлен в РФ)
- (б) **Suprema** (BioStar 2 + BioMini SDK; Корея, более premium)
- (в) **NtechLab FindFace** (РФ-вендор; единственный российский на маркете
  с готовым SDK; рекомендую при РФ-юрисдикции)
- (г) Самописный CV (OpenCV + InsightFace, на Yandex Vision/локально) —
  дешевле в разработке, но требует ML-инженера и Phase 2+
- (д) Ещё не выбирали

**Зачем:** влияет на ADR-0002 (Face Control vendor + хранение шаблонов),
**152-ФЗ ст. 11** (отдельное письменное согласие на биометрические ПДн),
смету (готовый SDK ≈ 60–100 ч интеграции, самописный CV ≈ 200–400 ч),
архитектуру `apps/face_control_adapter`.

**Рекомендация:** **(в) NtechLab FindFace** при РФ-юрисдикции — нет
санкционных рисков, есть on-prem развёртывание (важно для 152-ФЗ),
готовый SDK с Python/Java/С++. (а) Hikvision — если уже стоят их
камеры на объекте; в этом случае использовать как hardware-источник,
но FindFace — для распознавания.

**Что блокирует:** ADR-0002, форма биометрического согласия,
`apps/face_control_adapter` интерфейс, `Docs/09-compliance.md`
§ биометрия, `Docs/05-integrations.md` § Face Control.

**Варианты:**
- (а) **Yandex.Cloud** (managed PG + Lockbox + Object Storage + CDN —
  default для РФ)
- (б) Selectel (cheaper, managed PG, но менее зрелые managed-сервисы)

- (д) Вне РФ (только если **нет** ПДн в системе — невозможно для UniPrint

биллинга, native managed PG c PITR, Lockbox для секретов, S3-совместимое
Object Storage для макетов клиентов. (г) on-prem — только если есть
готовое железо и sysadmin (риск сетевых сбоев на самой инфре).

**Что блокирует:** ADR-0003, secrets vault выбор, CI/CD pipeline,
infra/terraform структура, ENV-vars в `.env.example`.

### Q4. Существующая система и миграция данных

**Варианты:**

**Что блокирует:** sprint-1 milestone «миграция данных», смета,

### Q6. Mobile-стек

**Варианты:**
- (а) **Native** (Kotlin Android + Swift iOS) — лучший UX, полный доступ
  к camera/GPS/sensors, но **×2 по стоимости**
- (б) **React Native** (одна команда, Expo/CNG, mature) — баланс цена/UX
- (в) **Flutter** (Dart, Material/Cupertino, single codebase, fast)
- (г) **PWA mobile-first** (Next.js + Service Worker + manifest) — самый
  дешёвый, но ограниченный доступ к hardware (Face Control камера —
  через web API, push на iOS — через WebPush iOS 16.4+)
- (д) Только Android (Kotlin) — если все сотрудники на Android-планшетах
- (е) **Гибрид:** PWA в фазе 1 + native в фазе 2

**Зачем:** влияет на ADR-0001, **смету ×2** (если native обе платформы),
команду (отдельный mobile dev для native vs general-purpose для
RN/Flutter/PWA), сроки. Допущение для прототипа — **PWA mobile-first**.

**Рекомендация:** **(е) Гибрид** — прототип и фаза 1 на PWA для быстрого
выхода в pilot, фаза 2 — нативные приложения если потребуется лучший
UX или offline-mode для производства. Если все сотрудники на Android
устройствах — (д) Kotlin native сразу.

**Что блокирует:** ADR-0001, mobile-команда, смета (`Docs/06-estimate.md`
±50% по часам), структура `mobile/` или `prototype/apps/*-mobile/`.

  с feature-flag `TELEGRAM_ENABLED`

Email для документов (счета, акты).

**Что блокирует:** `apps/notifications` дизайн, ENV-vars
(`TELEGRAM_BOT_TOKEN`, `SMS_*`, `EMAIL_*`), стратегия каналов в

- (а) **2 недели** (стандарт agile — рекомендую)
- (б) 1 неделя (быстрее feedback, но больше overhead на ceremonies)
- (в) 3–4 недели (для крупных архитектурных задач)

**Рекомендация:** **(а) 2 недели** — sprint-planning понедельник,
demo + retro четверг через 2 недели. Sprint-review с владельцем —
30 мин раз в спринт.

**Что меняет:** структура `Docs/sprints/<NN>/`, ритм синхронизаций.

### Q18. Demo URL

**Варианты:**
- (а) **Публичный stage** (Vercel preview — рекомендую для прототипа)
- (б) Закрытый VPN / IP-allowlist
- (в) HTTP basic auth на Vercel
- (г) Только локально через `pnpm dev`

**Рекомендация:** **(а) Vercel preview** для прототипа — генерируется
по push в feature-ветку, удобно показать клиенту. Для prod (после
ADR-0003) — закрытый stage с IP-allowlist + basic auth до публичного
запуска.

**Что меняет:** Vercel project setup, ENV-vars, `Docs/runbook/deploy.md`.

### Q19. Sentry / observability — фаза 1 или 1.5

**Варианты:**
- (а) **Фаза 1** (платим раньше — рекомендую, ~5к ₽/мес)
- (б) Фаза 1.5 (как в типичной спеке)

**Зачем:** без observability slogan «MTTR < 2ч» невыполним. Sentry
self-hosted на Yandex.Cloud стоит ~5к ₽/мес vs стоимость одной
потерянной транзакции в продакшене (потеря заказа = потеря клиента).

**Рекомендация:** **(а)** — стоимость пренебрежимая, польза огромная
на пилоте: каждая ошибка видна в реальном времени.

**Что меняет:** ADR на observability, Sentry в `apps/_settings`,
`SENTRY_DSN` в ENV-vars, фаза установки в `Docs/07-roadmap.md`.

### Q20. PM-канал для коммуникации

**Варианты:**

- (в) **Email** (формально)
- (г) **WhatsApp Business**
- (д) Несколько каналов (Telegram быстрые + Email формальные)

**Рекомендация:** **(д)** — Telegram / WhatsApp для быстрых вопросов
+ Email или GitHub Issues для формальных решений + Vercel preview
ссылки для демо.

**Что меняет:** место хранения переписки и решений, формат stand-up'а.
