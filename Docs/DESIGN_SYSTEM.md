## 1. Принципы

7 правил. Если вы их нарушаете — у вас веская причина и она в ADR.

1. **Источник правды — токены, не значения.** Никогда не пишите `color: #D9531E`. Пишите `var(--brand)` / `theme.colors.brand` / `bg-brand`. Если нужного токена нет — добавьте его в `tokens.ts`, не хардкодите.

2. **Один компонент — одна задача.** `<StatusPill>` показывает статус заказа. Не «универсальный пилл для всего». Для тегов услуг — отдельный `<ServiceChip>`. Слияние компонентов по принципу «они похожи внешне» — отказ.

3. **Mobile-first для production / warehouse.** Эти два кабинета сначала верстаются под 380px viewport, потом расширяются. Touch-target ≥ 44pt (в Tailwind: `min-h-11`). Если на десктопе выглядит «слишком крупно» — это правильно, не уменьшайте.

4. **Desktop-first для client / manager / admin / owner.** Эти четыре сначала верстаются под 1280–1440px. Mobile — адаптация, не основной use-case. Сайдбар скрывается под burger при < 900px.

5. **Бизнес-правила (BR-XX) видны в UI.** Если бэк enforce'ит правило, фронт его показывает: callout, подпись поля, текст ошибки. Список — `Docs/BUSINESS_RULES.md`.

6. **Проторайп-бейдж не убирается.** Пока в БД синтетические данные — `PROTOTYPE · MOCK DATA` отображается в шапке. Удаление бейджа — отдельный PR с чек-листом «реальные ПДн / 152-ФЗ ст. 18 ч. 5 / хостинг РФ закрыт».

7. **Telegram нигде.** По решению владельца от 2026-05-05 (🔴 #7). Никаких иконок Telegram, ссылок, упоминаний в каналах нотификаций. Каналы: WebPush / Email / SMS.

## 2. Дизайн-токены

### 2.1 Цветовая палитра

**Все токены живут в `packages/tokens/colors.ts`.** Импортируются в Tailwind через `tailwind.config.ts` и в CSS-переменные через `globals.css`.

```ts
// packages/tokens/colors.ts
export const colors = {
  // База — кремовая «бумажная» палитра
  bg:        '#FAF6EF',  // основной фон страницы
  surface:   '#FFFFFF',  // карточки, модалки
  surface2:  '#F4EEE2',  // вторичный фон (табы, чипы)
  surface3:  '#FBF8F2',  // ховер строк таблицы, thead

  // Текст
  ink:       '#1A1410',  // основной заголовок
  ink2:      '#3D332A',  // обычный текст
  ink3:      '#7A6F62',  // вторичный, метки
  ink4:      '#B5A899',  // плейсхолдеры, disabled

  // Линии
  line:      '#E8DECC',  // основные границы
  line2:     '#DCCFB6',  // hover границы

  // Бренд (UniPrint orange)
  brand:     '#D9531E',  // основной акцент, primary CTA hover
  brand2:    '#B8401A',  // нажатие, тёмный текст на soft
  brandSoft: '#FBE6D9',  // фоны badge, soft-кнопок

  // Статусы
  green:     '#5C8A3A',  // готов, выдан, успех
  greenSoft: '#E5EDD9',
  greenInk:  '#3D5E22',

  amber:     '#C9842A',  // в работе, печать
  amberSoft: '#F7E8CD',
  amberInk:  '#7A5618',

  red:       '#B8401A',  // брак, ошибка, удаление
  redSoft:   '#F6D9CE',
  redInk:    '#7A2410',

  blue:      '#3F6E8A',  // в очереди, лиды
  blueSoft:  '#DCE8EE',
  blueInk:   '#2E5470',

  purple:    '#7B5896',  // дизайн, согласование макета
  purpleSoft:'#E8DFEC',
  purpleInk: '#4A3158',

  // Акцент (используется аккуратно)
  gold:      '#C8923A',  // премиум-индикаторы, аватары
} as const
```

### 2.2 Цветовая семантика — что когда

| Семантика | Цвет | Где используется |
|---|---|---|
| **Бренд / primary CTA** | `brand` / `brand2` | главные кнопки на ховере, акцентные элементы |
| **Тёмный CTA** | `ink` (фон) + `bg` (текст) | основные кнопки в desktop, переключатели |
| **Готов / Успех** | `green` | заказ готов, выдан, оплачен; checkmark |
| **В работе** | `amber` (с `pulse` анимацией) | производство, печать, текущая операция |
| **В очереди / Лид** | `blue` | новый лид, заказ в очереди |
| **На дизайне / Согласование** | `purple` | макет на дизайне, ожидание клиента |
| **Брак / Ошибка** | `red` | брак (BR-03), неудачное действие, удаление |
| **Новый / Создан** | `brand` (soft) | только что созданный заказ |
| **Архив / Закрыт** | нейтральный `ink3` на `surface2` | закрытые заказы, неактивные строки |

### 2.3 Запрещённые цвета

- `#000` — используйте `--ink`
- `#FFF` без необходимости — на странице фон `--bg`, не белый
- Любой синий за пределами `blue` (нет твиттер-голубого, нет facebook-синего)
- Зелёный «ок-зелёный» вне `green` (нет браво-салатового)
- Градиенты вне списка ниже

### 2.4 Допустимые градиенты

Только эти, и только в обозначенных местах:

```css
/* price-box (предварительная стоимость у клиента) */
linear-gradient(135deg, #FBF1E2, #F7E5CC)

/* P&L profit card (учредитель) */
linear-gradient(135deg, #F0F5E5, #E5EDD9)

/* P&L revenue card */
linear-gradient(135deg, #E5EDF3, #DCE8EE)

/* P&L cost card */
linear-gradient(135deg, #FBE6D9, #F8D5BF)

/* Аватары — генерируются по seed (см. AvatarColors) */
```

## 3. Типографика

**3 шрифта. Не больше. Не меньше.**

| Шрифт | Назначение | Вес | Размеры |
|---|---|---|---|
| **Fraunces** (serif) | Заголовки страниц, цифры в KPI и P&L, названия карточек | 400, 500, 600 | 18, 20, 28, 32, 34, 38 |
| **Manrope** (sans) | Весь UI-текст, кнопки, поля, таблицы, навигация | 400, 500, 600, 700 | 11, 12, 13, 14, 15 |
| **JetBrains Mono** | ID заказов (`UNI-2026-00001`), BR-коды (`BR-01`), коды клавиш (⌘K), время (`14:32`), технические метки | 400, 500, 600 | 10.5, 11, 11.5, 12 |

### 3.1 Правила применения

- **Числа в KPI и P&L → всегда Fraunces 500.** «1 500 ₽», «30», «160 500».
- **Числа в таблицах (суммы) → Fraunces 500.** Курсив запрещён.
- **Числа в формах (input, лимиты) → Manrope.** Чтобы не выбивалось из UI.
- **ID заказов везде → JetBrains Mono.** Никогда обычным шрифтом — иначе сливается с названием.
- **BR-XX коды → JetBrains Mono 600 цветом `brand2`.** Это маркер бизнес-правила.
- **Заголовок страницы (`page-title`) → Fraunces 500 32-38px**, акцентное слово в `<em>` — italic + цвет `brand`.

### 3.2 Запрещённые комбинации

- Fraunces в кнопках (выглядит как «парадная вывеска»)
- JetBrains Mono в обычном тексте
- Manrope для KPI-цифр (выглядит как «обычная админка»)
- Жирность 800–900 (нет в наборе)

## 4. Шкала отступов и радиусов

### 4.1 Spacing (Tailwind базовая шкала, без отсебятины)

```
0.5  = 2px    — иконки внутри пиллов
1    = 4px    — gap внутри chip
2    = 8px    — gap внутри карточки
3    = 12px   — padding small
4    = 16px   — стандартный gap между элементами
5    = 20px   — padding карточки
6    = 24px   — большой gap между блоками
8    = 32px   — отступ от шапки до контента
10   = 40px   — отступ между секциями
```

Запрещено: `padding: 17px`, `margin-top: 23px` и любые «пиксельные» значения вне шкалы.

### 4.2 Радиусы

```
sm   = 6px   — кнопки внутри pill-табов
md   = 8px   — кнопки, инпуты, мелкие пиллы
lg   = 9-10px — крупные инпуты, drop-zone
xl   = 12-14px — карточки малые, status-pills box, mini-stat
2xl  = 14-16px — основные карточки
3xl  = 24px  — обёртка телефона в PWA
full = 9999px — пиллы статусов, чипы, аватары, бейджи
```

### 4.3 Touch targets (ОБЯЗАТЕЛЬНО для PWA)

- Любая кнопка / тач-цель в `production-mobile` и `warehouse-mobile` — **минимум 44×44pt** (≈ `min-h-11 min-w-11` в Tailwind).
- `btn-big` (главные действия складщика и печатника) — **min-height: 56px**.
- Tab-bar items — **min-height: 48px**.

## 5. Компоненты

Все компоненты живут в `packages/ui/src/components/`. Карта компонентов:

### 5.1 Базовые

| Компонент | Файл | Назначение | Когда НЕ использовать |
|---|---|---|---|
| `<Button variant="primary" \| "ghost" \| "danger">` | `Button.tsx` | стандартные действия | для главных мобильных CTA — там `<BigButton>` |
| `<BigButton tone="brand" \| "danger" \| "success" \| "outline">` | `BigButton.tsx` | главные действия в PWA | desktop — там `<Button>` |
| `<Input>` `<Select>` `<Textarea>` | `Form.tsx` | формы | поиск в шапке — там `<SearchInput>` |
| `<SearchInput>` | `SearchInput.tsx` | глобальный поиск с ⌘K | внутри форм |
| `<IconButton>` | `IconButton.tsx` | действия с иконкой только (notif, settings) | если нужен текст — обычный Button |

### 5.2 Структура

| Компонент | Назначение |
|---|---|
| `<Sidebar>` + `<SidebarNav>` + `<SidebarUserCard>` | левый тёмный сайдбар (только desktop-кабинеты) |
| `<TopBar>` + `<Crumbs>` | шапка с хлебными крошками и поиском |
| `<PageHead>` + `<PageTitle>` + `<RoleTag>` | заголовок страницы |
| `<Card>` + `<CardHead>` + `<CardTitle>` | основная карточка с шапкой |
| `<Stat>` (label + value + trend) | KPI-карточка |
| `<StatGrid cols={3 \| 4 \| 5}>` | сетка KPI |

### 5.3 Доменные

| Компонент | Назначение | Бизнес-привязка |
|---|---|---|
| `<StatusPill status="...">` | статус заказа | enum: `done`, `work`, `queue`, `new`, `review`, `design`, `defect` |
| `<OrderId>` | ID заказа `UNI-2026-XXXXX` | моноширинный |
| `<Amount>` | денежная сумма с `₽` | Fraunces |
| `<BrCallout code="BR-01">` | информер о бизнес-правиле | подсвечивает BR-XX и описание |
| `<ProtoTag>` | бейдж проторайпа | держится в шапке всегда |
| `<Avatar tone="brand"\|"green"\|...>` | инициалы пользователя | tone задаётся по роли |
| `<Kanban>` + `<KCol>` + `<KCard>` | канбан-доска (manager-web) | колонки соответствуют статус-машине |
| `<PnlCard variant="profit"\|"revenue"\|"cost">` | P&L блок (owner-dashboard) | только в дашборде учредителя |
| `<BarRow label value percentage>` | строка с прогресс-баром | для доходимости по типам, нагрузки |
| `<PwaPhone>` + `<PwaHeader>` + `<PwaTabBar>` | мобильная PWA-обёртка | только для production / warehouse |
| `<FaceControlFrame>` | окно сканирования биометрии | только в логине производственника |

### 5.4 Запрещено

- Создавать новый «универсальный пилл», если есть `<StatusPill>`. Если нужен новый тип — добавьте в enum.
- Стилизовать `<Button>` инлайном (`style={{background: ...}}`). Только через `variant`.
- Вкладывать `<Card>` в `<Card>` глубже одного уровня.
- Использовать иконки не из `lucide-react` (единый icon set).
- `box-shadow` своими значениями. Только токены `--shadow`, `--shadow-lg`.

## 6. Layout-правила по кабинетам

### 6.1 Desktop-кабинеты (client-portal, manager-web, admin-panel, owner-dashboard)

```
┌─────────┬──────────────────────────────────────┐
│         │  TopBar (sticky, 62px, с поиском)    │
│         ├──────────────────────────────────────┤
│ Sidebar │                                      │
│ 240px   │  Content (max-width: 1400px)         │
│ dark    │  padding: 30px                       │
│ sticky  │                                      │
│         │                                      │
│         │  [Page head]                         │
│         │  [Stats grid]                        │
│         │  [Cards / Tables / Forms]            │
└─────────┴──────────────────────────────────────┘
```

**Правила:**
- Сайдбар тёмный (`--ink`), фиксирован, скроллится внутри себя при переполнении.
- На < 900px сайдбар скрывается, появляется burger в TopBar.
- Контент имеет `max-width: 1400px`, центрирован в main.
- TopBar `sticky top: 49px` (49px = высота role-switcher в проторайпе; в проде это 0).

### 6.2 Mobile-PWA-кабинеты (production-mobile, warehouse-mobile)

```
┌────────────────────┐
│ Status bar (32px)  │  ← имитация iOS/Android (только в проторайпе)
├────────────────────┤
│ PWA Header (dark)  │  ← бренд + аватар + Face Control индикатор
│  + greet line      │
│  + shift bar       │
├────────────────────┤
│                    │
│ Body               │  ← скроллится
│ padding: 16px      │
│ pb: 80px (под tab) │
│                    │
├────────────────────┤
│ Tab Bar (sticky)   │  ← 4 пункта максимум
└────────────────────┘
```

**Правила:**
- Viewport ≤ 420px. На десктопе оборачивается в «телефон» с радиусом 24px (демо-режим).
- Все интерактивные элементы — большие (`btn-big` 56px, tab-bar items 48px).
- Не более 4 пунктов в tab-bar. Больше — открывайте экран «Ещё».
- Шапка тёмная (`--ink`) с зелёным/жёлтым/красным индикатором смены (Face Control).
- Один экран = одно главное действие. Если действий два — два больших равноправных. Три+ — список.

### 6.3 Запрещённые layouts

- Sidebar в production-mobile / warehouse-mobile (это PWA, не desktop)
- TopBar с поиском в PWA (поиск — отдельный экран по тапу)
- Двухколоночный layout < 900px ширины
- Бесконечный horizontal scroll в таблицах (горизонтальный скролл = плохо; либо упрощаем колонки, либо переходим в карточный вид)

## 7. Бизнес-правила в UI

Каждое правило BR-XX из `Docs/BUSINESS_RULES.md` имеет визуальное отражение. Это не «можно показать», это «должно быть видно пользователю в момент действия».

| Правило | Где показывается | Как показывается |
|---|---|---|
| **BR-01** Списание только на заказ | warehouse-mobile, экран «Списать» | в callout перед списанием + поле «Заказ» обязательное, нельзя сохранить с `null` |
| **BR-02** Антидубль клиентов по телефону | manager-web, форма «Новый клиент» | при вводе телефона — поиск, при найденном дубле баннер «Этот клиент уже есть, открыть?» |
| **BR-03** Брак фиксирует только складщик | production-mobile | callout «Брак фиксирует только складщик. Передайте на склад» — кнопки «зафиксировать брак» НЕТ |
| **BR-04** Услуги только из справочника | client-portal, manager-web — формы заказа | поле «Услуга» — только select из справочника, ручной ввод заблокирован |
| **BR-05** Баланс сотрудника, не уходить в минус | production-mobile, экран «Заработок» | если заработано < минимума, показать «Минимум выплачен. Долг 1 200 ₽ погасится в следующем периоде» |
| **BR-06** Face Control read-only для сотрудника | production-mobile, warehouse-mobile, экран «Смена» | события Face Control видны, **редактировать нельзя**. Корректировка времени смены — через отдельный workflow с журналом |
| **BR-07** 3 типа заказа | везде, где есть заказ | `услуга-цех / услуга-офис / продажа-товар` — лейблом, всегда mono, в фильтрах таблиц обязательный фасет |
| **BR-09** FIFO-списание | warehouse-mobile, форма списания | при выборе материала автоматически выбирается старая партия с подписью «партия P-2026-018, 12.03.2026» |

**Если в задаче трогаете UI, относящийся к BR-XX — обновите этот раздел.**

## 8. Прототип vs Production

### 8.1 Что разрешено в прототипе (Phase-0, Vercel preview, MSW моки)

- ✅ Синтетические данные (`Иван Петров`, `Алексей Кузнецов`, `UNI-2026-00001`)
- ✅ Бейдж `PROTOTYPE · MOCK DATA` в шапке
- ✅ Vercel-хостинг (за пределами РФ — допустимо т.к. данных ПДн нет)
- ✅ Mock Service Worker (MSW) вместо реального бэка
- ✅ Локальное хранилище в localStorage / IndexedDB

### 8.2 Что запрещено в прототипе

- ❌ Реальные ПДн (152-ФЗ ст. 18 ч. 5 — хостинг ПДн обязан быть в РФ)
- ❌ Реальные ИНН, ОГРН, банковские реквизиты
- ❌ Загрузка реальных макетов клиентов
- ❌ Любые реальные email/phone — только синтетика (`@uniprint.ru`, `+7 999 ...`)
- ❌ Скрытие или удаление `PROTOTYPE` бейджа

### 8.3 Чек-лист перехода в production

Перед удалением `PROTOTYPE` бейджа:

- [ ] Хостинг — в РФ (Yandex.Cloud / Selectel / VK Cloud / on-prem) — закрыт ADR-0003
- [ ] Юрисдикция определена (РФ / ИП / ООО / СНО) — закрыт 🔴 #1
- [ ] Все mock-data заменены на реальные API
- [ ] Audit-log на доступы к ПДн работает
- [ ] Cookies / Privacy notice настроены (152-ФЗ)
- [ ] Биометрические согласия собраны (152-ФЗ ст. 11) — для Face Control
- [ ] WORM / lifecycle policy в S3 для документов (402-ФЗ, хранение 5 лет)

## 9. Чек-лист перед коммитом

Прогоните этот список на каждом UI-PR. Если хотя бы один пункт не выполнен — PR не мержится.

### Токены
- [ ] Нет хардкода цветов (`#XXXXXX`) — только токены
- [ ] Нет хардкода шрифтов — только Fraunces / Manrope / JetBrains Mono
- [ ] Нет «магических чисел» в padding / margin вне Tailwind-шкалы
- [ ] Все радиусы из шкалы

### Компоненты
- [ ] Использованы существующие компоненты, новый создан только если нет подходящего
- [ ] Если создан новый — есть Storybook-стори
- [ ] Если меняется существующий — обновлены все места использования
- [ ] Иконки из `lucide-react`, не SVG-инлайн (за исключением декоративных)

### Бизнес-правила
- [ ] Если экран связан с BR-XX — правило отображено в UI (callout / подпись / поведение)
- [ ] Поведение совпадает с `Docs/BUSINESS_RULES.md`

### Layout
- [ ] Desktop: max-width контента ≤ 1400px
- [ ] Desktop: проверено на 1280, 1440, 1920px
- [ ] Mobile-PWA: viewport 380px, touch-targets ≥ 44pt
- [ ] Адаптивные брейкпоинты: 1100px, 900px, 720px, 520px

### Доступность
- [ ] Все интерактивные элементы — клавиатура (Tab, Enter, Esc)
- [ ] Контраст текста ≥ AA (4.5:1 для обычного, 3:1 для крупного)
- [ ] У форм — связанные `<label>`
- [ ] У иконочных кнопок — `aria-label` или `title`

### Прототип
- [ ] Бейдж `PROTOTYPE · MOCK DATA` на месте
- [ ] Все данные синтетические
- [ ] Telegram нигде не упоминается
- [ ] Каналы нотификаций корректные (WebPush / Email / SMS)

### Содержимое
- [ ] Тексты на русском, без англицизмов где есть русский эквивалент
- [ ] ID заказов формата `UNI-2026-XXXXX` (моноширинный)
- [ ] Денежные суммы — Fraunces, формат `1 500 ₽` (неразрывный пробел перед валютой)
- [ ] Даты — `14.05.2026` или `14 мая` (не `5/14/26`)
- [ ] Времена — `14:32` (24-часовой формат)
