## 1. Состав команды

**Senior-команда специалистов с опытом в производственных ERP-системах.** Состав формируется под скоп каждой фазы и фиксируется в приложении к контракту с резюме (CV) каждого участника.

**Минимальные требования к участникам:**
- Senior-уровень, минимум 5 лет опыта в коммерческой разработке.
- Опыт работы в ERP / CRM / WMS / производственных системах.
- Свободное владение основным стеком проекта (Node.js / TypeScript для backend, React / Next.js для frontend).
- Опыт работы с РФ-compliance (152-ФЗ, ТК, 402-ФЗ, 54-ФЗ) для ролей, связанных с этими доменами.

**Замены участников** допускаются только с письменным согласованием с заказчиком и handover-периодом 1 нед на критических ролях.

**Привлекаемые консультанты по необходимости:**
- Compliance / RA-аналитик — для 152-ФЗ baseline (S7) и pre-pilot аудита (S9-S10).
- Финансовый аналитик — для валидации формул себестоимости / P&L / налогов в Phase 2.
- R3 — полиграфический консультант — для импорта эталонного справочника услуг и материалов в Stage A S2.
- UX/Product Designer — для новых экранов вне прототипа (payroll detail, equipment detail и т.д.) в Phase 2.

## 2. Юридические условия

### 2.1 NDA (соглашение о неразглашении)

Подписывается до начала работ. Покрывает:
- Доступ к вашим данным (клиенты, заказы, финансы).
- Раскрытие архитектурных решений и кода вашим конкурентам.
- Коммерческую информацию (цены, маржа, объёмы).

NDA взаимный — мы не передаём информацию о вас третьим лицам, вы не раскрываете нашу методологию работы и pricing.

### 2.2 IP (интеллектуальная собственность)

После полной оплаты соответствующего этапа:
- **Stage A полная оплата** → IP на код Stage A, дизайн, документацию переходит к вам.
- **Stage B полная оплата** → IP на Stage B переходит к вам.
- И т.д.

**Что НЕ переходит к вам:**
- Open-source библиотеки (NestJS, Prisma, Next.js и т.д.) — остаются под своими лицензиями (MIT / Apache 2.0).
- Наш внутренний tooling и методологии (CI/CD конфиги шаблоны, internal scripts).
- Документация процессов разработки на стороне студии.

### 2.3 Гарантия (warranty)

**Stage A:** 4 недели после go-live (M6 → M7).
**Stage B:** 2 недели после go-live.
**Phase 2 и 3 (T&M):** 2 недели после каждого спринт-acceptance.

В период гарантии бесплатно фиксим:
- **Critical** (блокер production / data loss / security): SLA 24 ч.
- **High** (major функционал не работает): SLA 3 раб. дня.
- **Medium** (есть workaround): в backlog, SLA 1-2 спринта.

После гарантии:
- Critical / High → платно, по тарифам поддержки (отдельный контракт).
- Medium / Low → CR workflow.

### 2.4 Поддержка после warranty

**Тариф «Базовый»** (опционально, заключается отдельным договором):
- 8/5 (рабочие часы Москвы, пн-пт), реакция на critical в 4 ч, на high — в 1 раб. день.
- Включает: bug fixes, security patches, minor updates библиотек.
- Не включает: новые фичи (это T&M), мажорные обновления стека (отдельный CR).
- Стоимость: фиксированная подписка от 200 тыс ₽/мес или часовой пакет от 60 тыс ₽/40 ч (4 раб. дня).

**Тариф «24/7»** (для production-critical):
- 24/7 on-call, реакция на critical в 1 ч.
- Стоимость: от 500 тыс ₽/мес.

### 2.5 Ответственность

Стандартная ответственность по ГК РФ + ограничения в контракте:
- **Прямой ущерб**: ограничен суммой контракта по соответствующему этапу.
- **Упущенная выгода / косвенный ущерб**: исключаются (ст. 15 ГК).
- **Force majeure** (война, природные катастрофы, санкции, отключение интернета по решению гос. органов): сроки сдвигаются на период действия обстоятельств.
- **Скрытые дефекты в коде после warranty**: исправляются по тарифу поддержки или CR.

### 2.6 Расторжение контракта

**По инициативе клиента:**
- В любой момент с предупреждением за 10 раб. дней.
- Оплата фактически выполненных работ + 50% от ближайшего милстоуна (если работы по нему уже начаты).
- IP на выполненные работы переходит к клиенту после полной оплаты.

**По инициативе исполнителя:**
- Только при просрочке оплаты >15 раб. дней или нарушении NDA.
- С предупреждением за 10 раб. дней.

**По взаимному согласию:**
- В любой момент, по подписанному соглашению с разрешением IP-вопроса.

## 3. Change Request (CR) — изменение скопа

### 3.1 Что считается CR

| Тип запроса | CR? | Оплата |
| --- | --- | --- |
| Уточнение требования из спецификации | Нет | Включено |
| Микро-косметика UI (цвет, текст, padding) | Нет | Включено до 5% бюджета спринта |
| Новое поле в существующей сущности | **CR-light** | Paid (оценка до 1 ч бесплатно) |
| Новый workflow / новый модуль | **CR-major** | Paid (оценка до 4 ч бесплатно) |
| Изменение бизнес-правила (BR-XX) | **CR-major** | Paid + ADR + addendum к контракту |
| Изменение тех-стека / архитектуры | **CR-major** | Paid + ADR + обсуждение Steering Committee |
| Bug fix в текущем спринте до acceptance | Нет | Включено |
| Bug fix critical / high после acceptance в warranty | Нет | Включено (gratis) |
| Bug fix medium / low после warranty | **CR-light** | Paid |

### 3.2 Workflow CR

| Шаг | Кто | SLA |
| --- | --- | --- |
| 1. Клиент пишет CR в Jira (template: Description, Why, Priority) | Вы | — |
| 2. PM подтверждает получение, классифицирует (light / major) | PM | 1 раб. день |
| 3. Tech Lead делает estimate (часы + impact на текущий sprint) | Tech Lead | 2 раб. дня (light) / 5 раб. дней (major) |
| 4. PM формирует CR-document: scope, hours, cost, новый deadline, импакт | PM | 1 раб. день после estimate |
| 5. Клиент аппрувит / отклоняет / просит revise | **Только владелец или назначенный proxy в PoA** | 5 раб. дней |
| 6. Подписание CR-addendum к контракту (для major) | Юристы | 5 раб. дней |
| 7. Включение в backlog / sprint | PM | На ближайшем sprint planning |

### 3.3 Минимальная стоимость CR

- **CR-light**: минимум 4 ч работы (даже если работы на 30 мин — admin overhead).
- **CR-major**: минимум 16 ч.
- **Strategic CR** (>160 ч или меняет архитектуру): минимум 40 ч + встреча Steering Committee.

### 3.4 Кто аппрувит со стороны клиента

- **CR-light** (до 40 ч): владелец или назначенный proxy.
- **CR-major** (40-160 ч): только владелец, e-mail approval с подписью.
- **Strategic CR** (>160 ч): встреча владелец + наш PM + Tech Lead, signed addendum к контракту.

## 4. Условия приёмки результатов

### 4.1 Definition of Acceptance (acceptance form)

Каждый milestone имеет signed form:

```
Milestone Acceptance Form: M_X
Project: UniPrint ERP
Date: __________

DoD checklist (см. 02-roadmap.md § Sprint X) — все пункты пройдены:
[ ] Item 1
[ ] Item 2
...

Demo состоялся: __________ (date)
Acceptance criteria verified by: __________ (Owner signature)

Comments / открытые баги:
- bug #123 (medium, в P1.5 backlog)
...

Signed:
Владелец: ____________  Date: _____
PM: _________________  Date: _____
```

### 4.2 Default acceptance

Если вы не прислали acceptance в течение **3 рабочих дней** после demo (с напоминаниями на 1-й и 3-й день) — milestone считается **принятым по умолчанию**, платёж выставляется.

Это защита от затягивания процесса. Вы всегда можете запросить дополнительные правки через CR-workflow.

### 4.3 Acceptance Criteria формат

Каждая User Story в backlog имеет AC в формате **Given-When-Then**:

```
Story: "Менеджер создаёт заказ типа цех с привязкой к клиенту"

AC1: Given менеджер залогинен и имеет роль 'manager_<city>',
     When он открывает форму создания заказа и заполняет client_id, type='цех', services[],
     Then заказ создаётся в статусе 'new', audit-log пишет создание, returned 201 с order_id

AC2: Given существует другой заказ с таким же client_id,
     When менеджер видит форму,
     Then UI показывает badge "у клиента N активных заказов" (BR-02 контекст)

AC3: Given в catalog нет услуги X,
     When менеджер пытается добавить услугу X,
     Then UI отклоняет (BR-04), не отправляет POST
```

**Все AC покрыты автотестами** (unit / integration / E2E) перед demo.

## 6. FAQ — частые вопросы

### Q: Почему так дорого? Я слышал, типографские системы стоят дешевле.

**A:** Большинство «дешёвых» систем — это либо урезанный CRM (без склада / производства), либо коробочные решения 1С (которые потом долго и дорого допиливать), либо самописные «на коленке». UniPrint — полноценная ERP под производство и склад, с компонентами под 152-ФЗ, ТК, 54-ФЗ, биометрию. Сравнимые системы (например, Type, OptiPrint, ASystem) стоят 20-40 млн ₽ за внедрение и не покрывают всё, что у нас в скопе.

Phased-подход позволяет начать с 7-9 млн (Stage A) и проверить, нужно ли дальше.

### Q: Можно ли сделать ещё дешевле?

**A:** Да, есть варианты:
- **Урезать Stage A до 3 кабинетов** (без admin-panel) — экономия ~15% (5.5-7 млн вместо 7-9). Но это компромисс на гибкость управления.
- **Фиксированная команда меньшего размера** на Stage A — экономия ~20%, но сроки растягиваются до 6 мес вместо 4.
- **Stage A без compliance** (без 152-ФЗ baseline) — экономия ~10%, но запуск pilot невозможен по закону. **Не рекомендуем.**

Floor для Stage A — 6.5 млн ₽ (markup ~20%, защита команды). Ниже — мы не возьмёмся.

### Q: Что если я хочу запустить только клиентский портал, без всего остального?

**A:** Это не сработает. Клиентский портал требует backend с заказами, статусами, документами, оплатой. Без Stage A он бесполезен.

Если у вас уже есть какая-то система (например, 1С УНФ), можно сделать клиентский портал поверх неё — но это другой проект, отдельный КП.

### Q: А если мне нужна функциональность Phase 2 раньше, прямо в Stage A?

**A:** Это change request. Например, если хотите Face Control в Stage A — оценка добавления +400 ч (~1.5 млн ₽), сроки +3 нед. Подписываем addendum к контракту.

Однако **рекомендуем не делать**: Phase 2 функционал требует стабильной production-системы (Stage A pilot ≥4 нед) для корректной работы. Иначе Face Control будет глючить из-за проблем core-системы, которые ещё не все выловлены.

### Q: Что если я хочу остановиться после Stage A?

**A:** Это валидный сценарий. После M7 (Warranty End, нед 23.5) система работает, контракт закрыт. Вы можете:
- Продолжить с Stage B / Phase 2 / Phase 3 в любой момент (новый addendum к контракту).
- Заключить отдельный контракт на поддержку (тариф «Базовый» от 200 тыс/мес).
- Не продолжать — система ваша по IP, можете развивать самостоятельно или передать другой команде. Документация (PRD, ADR, runbook'и) всё описывает.

### Q: А если мы не сработаемся — есть ли способ уйти на полпути?

**A:** Да, см. § 2.6. По инициативе клиента контракт расторгается с 10-дневным уведомлением, оплата фактически выполненных работ + 50% от ближайшего милстоуна. IP на выполненное переходит к вам.

### Q: Откуда возьмутся данные для тестирования?

**A:** Мы используем синтетические fixtures в development и UAT. R3-консультант поможет с эталонными справочниками услуг и материалов (30+ позиций). Реальные данные клиентов и заказов появляются только после go-live.

Если есть данные из старой системы (1С / Excel) — **миграция отдельным CR** (~160 ч / 600 тыс ₽). Это закладываем в S7 если Q4 = «да».

### Q: А если мой бухгалтер не использует 1С?

**A:** Мы поддерживаем стандартный экспорт в Excel + JSON для любой системы. Также можем сделать integration с CRM/ERP вашего бухгалтера (СБИС / Контур / Бухсофт) — это CR-major, оценка по обстоятельствам.

### Q: Кто отвечает за безопасность production?

**A:** В рамках Stage A — мы (внешний pentest, OWASP ASVS L1, мониторинг). После warranty — вы (или контракт на поддержку с нами).

### Q: Что если я хочу свою команду разработчиков подключить к проекту?

**A:** Возможно, но усложняет процесс. Опция «extended team»: ваши разработчики работают параллельно нашим, под нашим scrum-процессом. +20% к PM часам на координацию. Их часы биллятся вами отдельно.

Альтернатива — handover после Stage A: мы передаём вашей команде полный onboarding, документацию, ставим вас на rails. Стоимость handover: 2 нед × full team ≈ 800 тыс ₽.

## 7. Глоссарий

| Термин | Что означает |
| --- | --- |
| **ERP** | Enterprise Resource Planning — единая система управления ресурсами предприятия (заказы, склад, производство, финансы). |
| **CRM** | Customer Relationship Management — управление взаимоотношениями с клиентами (лиды, сделки, история). |
| **WMS** | Warehouse Management System — система управления складом. |
| **PWA** | Progressive Web App — веб-приложение, которое устанавливается на телефон как обычное (без AppStore / Google Play), работает off-line. |
| **MVP** | Minimum Viable Product — минимальный жизнеспособный продукт, в нашем случае Stage A. |
| **Pilot** | Первый запуск с реальными пользователями (не полный rollout). |
| **UAT** | User Acceptance Testing — тестирование с участием конечных пользователей перед production. |
| **DoD** | Definition of Done — критерии готовности задачи / спринта / милстоуна. |
| **Acceptance** | Приёмка — подписанный документ, что результаты соответствуют требованиям. |
| **CR** | Change Request — заявка на изменение скопа после начала работ. |
| **Sprint** | 2-недельный цикл разработки с demo в конце. |
| **Milestone (M1, M2, ...)** | Контрольная точка с проверяемым результатом и платежом. |
| **T&M** | Time & Materials — оплата по часам по факту выполненных работ. |
| **Fix-price** | Фиксированная цена за зафиксированный скоп. |
| **Retainer** | Фиксированный месячный пакет часов (как абонемент). |
| **Backlog** | Список задач на разработку, расставленных по приоритету. |
| **CI/CD** | Continuous Integration / Continuous Delivery — автоматическое тестирование и деплой. |
| **152-ФЗ** | Федеральный закон «О персональных данных». |
| **ТК РФ** | Трудовой кодекс РФ. |
| **402-ФЗ** | Закон «О бухгалтерском учёте». |
| **54-ФЗ** | Закон о применении контрольно-кассовой техники. |
| **ОФД** | Оператор фискальных данных — посредник между кассой и налоговой по 54-ФЗ. |
| **СНО** | Система налогообложения (УСН / ОСН / ПСН). |
| **РКН** | Роскомнадзор (контролирует ПДн). |
| **BR-XX** | Business Rule — бизнес-правило в нашей системе (всего 36, см. `BUSINESS_RULES.md`). |
| **ADR** | Architecture Decision Record — документ с обоснованием архитектурного решения. |
| **OpenAPI** | Стандарт описания REST API (контракт между frontend и backend). |
| **WebPush** | Push-уведомления в браузере (без AppStore). |
| **Audit-log** | Журнал всех важных действий пользователей (кто что когда изменил). |
| **RBAC** | Role-Based Access Control — ограничение доступа по ролям. |
| **DSAR** | Data Subject Access Request — право клиента запросить все свои данные (152-ФЗ). |
| **FIFO** | First In, First Out — старейшая партия списывается первой. |
| **PERT** | Program Evaluation and Review Technique — метод оценки часов с буфером. |
| **Yandex.Cloud** | Российское облако (managed PostgreSQL, S3, Kubernetes). |
| **YooKassa** | Платёжный сервис (эквайер) от Яндекса. |
| **Hikvision** | Производитель систем видеонаблюдения (предположительный vendor для Face Control). |

## 8. Документы, которые войдут в комплект поставки

После M6 (Pilot Ready) вы получаете:

1. **Code repository** на GitHub (private) с полной историей коммитов.
2. **Production environment** на Yandex.Cloud, с доступом для вашей команды.
3. **Документация:**
   - `README.md` — как развернуть систему с нуля.
   - `ARCHITECTURE.md` — описание архитектуры (extended `Docs/03-architecture.md`).
   - `BUSINESS_RULES.md` — все 36 BR с описанием enforce'ов.
   - `API.md` — OpenAPI spec.
   - `DATABASE.md` — схема БД, миграции, seed данных.
   - `DEPLOY.md` — runbook для deploy'а.
   - `RUNBOOKS/` — 5 операционных runbook'ов (incident response, DB failure, integration outage, secret rotation, scale up).
   - `ADR/` — все ADR'ы с обоснованием решений.
   - `PRD/` — PRD на каждый модуль (25 шт).
4. **CHANGELOG.md** — история всех релизов.
5. **Test reports** — последний полный test pass (typecheck, lint, build, unit, integration, e2e).
6. **Pentest report** — итоговый отчёт внешнего аудитора.
7. **UAT report** — итоговый отчёт UAT с подписью владельца.
8. **Acceptance forms** — все 7 milestone-acceptance в физическом виде (или DocuSign).
