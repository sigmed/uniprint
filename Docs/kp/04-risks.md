# Риски и их нивелирование

> Что может пойти не так — и как мы это предотвращаем. Этот раздел важен потому, что **отсутствие списка рисков ≠ отсутствие рисков**. Лучше знать заранее.

| # | Вопрос | Влияние на проект | Дедлайн ответа | Что если ответ позже |
| --- | --- | --- | --- | --- |
| **Q1** | **Юрисдикция:** ИП или ООО? Какая система налогообложения (УСН 6% / УСН 15% / ОСН с НДС / ПСН)? | Compliance scope, цена ±10%, формулы налогового модуля в Phase 2 | **до 29 мая 2026** | Старт +2 нед, контракт продлевается |

| **Q3** | **Хостинг:** Yandex.Cloud / Selectel / VK Cloud / on-prem? | DevOps часов ±15-25%, способ хранения данных | **до 29 мая 2026** | Старт +2 нед |
| **Q4** | **Миграция данных** из 1С / Excel / другой системы: есть или нет? | Если «да» — +160 ч (~600 тыс ₽), +2 нед в Stage A S7 | **до 15 июня 2026** (S2 demo) | Если миграция нужна, но не подтверждена — block в S7 |

## 2. Технические риски

8 ключевых рисков, связанных с технической реализацией.

### TR-01. Vendor lock-in на Face Control (Phase 2)

**Что может случиться:** Hikvision (или другой выбранный vendor) меняет API, попадает под санкции, прекращает поддержку Linux. Система Face Control перестаёт работать.

**Вероятность:** средняя. **Влияние:** высокое (без Face Control нет автоматического учёта смен, ручной возврат).

**Как защищаемся:**
- В коде написана **provider-абстракция** (face_control_adapter): смена vendor — это новая реализация интерфейса, не переписывание модуля.
- В Phase 3 закладываем 2-ю реализацию (NtechLab) как backup.
- Договорной buffer 80 ч на смену vendor (без переоценки контракта).

### TR-02. Race condition при списании материалов (Stage A)

**Что может случиться:** двое складщиков одновременно списывают последнюю партию материала → отрицательный остаток.

**Вероятность:** средняя (особенно в час пик). **Влияние:** высокое (нарушение учёта, конфликт между сотрудниками).

**Как защищаемся:**
- Транзакционные блокировки на уровне БД (`SELECT ... FOR UPDATE`).
- CHECK на уровне БД: `qty_remaining ≥ 0` — отрицательные остатки физически невозможны.
- Off-line очередь PWA идемпотентна по client-side UUID — повторная отправка не вызывает дубль.
- Concurrency-test обязателен в QA-suite перед каждым релизом (тест с 1000 одновременных списаний).

### TR-03. Корректность расчёта зарплаты при изменениях задним числом (Phase 2)

**Что может случиться:** учредитель меняет норматив за прошлый месяц → ломаются уже закрытые ведомости → конфликт с сотрудниками.

**Вероятность:** средняя. **Влияние:** высокое (трудовые споры, штрафы по ТК).

**Как защищаемся:**
- В каждом заказе сохраняется **snapshot нормативов** на момент создания (BR-14). Расчёт идёт по snapshot, не по текущему значению.
- `payroll_period.status = closed` блокирует пересчёт ведомости.
- Корректировки только через workflow с подписью учредителя и audit-log.
- Property-based tests на инвариант «закрытый период не меняется».

### TR-04. PWA не работает на слабом 3G в цеху

**Что может случиться:** в производственном цеху WiFi плохой / 3G слабый. Действия складщика / производственника не отправляются на сервер. Данные теряются.

**Вероятность:** средняя. **Влияние:** среднее.

**Как защищаемся:**
- **Service Worker + IndexedDB queue**: действие сохраняется локально, отправляется на сервер при reconnect.
- Visual indicator в углу экрана: «не отправлено N действий» с возможностью «попробовать ещё раз».
- Sync на reconnect с retry+backoff (3 попытки).
- Critical e2e-тест на throttle 3G в Playwright перед каждым релизом.

### TR-05. Деплой обновлений на работающее производство

**Что может случиться:** обновление системы во время рабочей смены → производство останавливается на 5-10 минут → потеря времени.

**Вероятность:** средняя (без правильной техники развёртывания). **Влияние:** высокое.

**Как защищаемся:**
- **Blue-green deployment**: деплоим новую версию рядом, потом переключаем трафик за 1 секунду. Откат — тоже за 1 секунду.
- Миграции БД в split-schema: новая колонка → backfill async → переключение чтений → удаление старой колонки. Каждый шаг безопасен.
- Deploy-windows вне рабочих часов (по умолчанию вечером в пятницу или утром в субботу).
- Feature-flags для рискованных изменений: новая логика выкатывается «выключенной», включается после проверки.
- Runbook на rollback за 15 мин — у DevOps под рукой.

### TR-06. ОФД-провайдер падает (Stage B, после интеграции онлайн-оплаты)

**Что может случиться:** Платформа ОФД (или Такском) недоступна → фискальный чек не выпускается. По 54-ФЗ это нарушение → штраф до 100% выручки.

**Вероятность:** средняя (1-2 раза в год у любого провайдера). **Влияние:** высокое.

**Как защищаемся:**
- **Outbox pattern**: чеки сохраняются в очередь, отправляются на ОФД отдельным процессом. Падение ОФД не ломает оплату.
- **Off-line queue** ≥30 дней: чеки накапливаются, отправляются при восстановлении.
- **Fallback провайдер**: возможность переключить на second provider (Такском ↔ Платформа ОФД) через изменение конфига за 5 минут.
- Daily reconciliation report → admin-alert, если очередь > N или возраст самого старого чека > 1 час.
- Runbook на эскалацию ОФД.

### TR-07. Кража биометрических шаблонов (Phase 2, после Face Control)

**Что может случиться:** взлом системы → утечка биометрических шаблонов сотрудников. По 152-ФЗ ст. 11 — штраф до 700 тыс ₽ + иски от пострадавших.

**Вероятность:** низкая (требует серьёзной атаки). **Влияние:** высокое.

**Как защищаемся:**
- Шаблоны хранятся в **Yandex Lockbox + KMS** с автоматической ротацией ключей. Изолированы от основной БД.
- Доступ к шаблонам — только через service-account через VPC. Никаких доступов от пользователей.
- **Audit-log на каждое чтение** биометрии — видно, кто и когда обращался к шаблону.
- Revoke-flow с cleanup ≤24 ч (если сотрудник отозвал согласие — шаблон удаляется в течение суток).
- **Pentest в Stage A S9 включает биометрию scope** (хотя сама фича в Phase 2 — security-аудит делаем заранее).

### TR-08. Дрифт прототипа и production-системы

**Что может случиться:** прототип на моках работает «правильно», production — иначе. Frontend начинает ломаться при подключении к real API.

**Вероятность:** средняя. **Влияние:** среднее (тратим время на синхронизацию).

**Как защищаемся:**
- **OpenAPI 3.1 spec** — один источник правды для всех контрактов между frontend и backend.
- **Codegen TypeScript-types** в `packages/types/` — frontend получает свежие типы автоматически.
- **Contract-tests** в CI (Pact или Schemathesis) — изменения в API ломают сборку до merge.
- Первая интеграция Stage A — синхронизация MSW-handlers (mock'ов прототипа) с real API в Sprint 1.

## 3. Бизнес и проектные риски

### BR-01. Поздние ответы на 🔴 Q1-Q5

См. § 1 выше. Митигация: deadline в контракте, простой команды компенсируется по T&M ставкам.

### BR-02. Изменение скопа после старта (Scope Creep)

**Что может случиться:** после старта работы вы захотите «ещё одну фичу», «ещё одно поле», «ещё один отчёт». Скоп раздувается, fix-price размывается.

**Вероятность:** высокая (в greenfield-проектах — почти неизбежно).

**Как защищаемся:**
- **Change Request workflow** (детально в `05-appendix.md`):
  - Уточнение требований — бесплатно.
  - Микро-косметика (цвет кнопки, текст) — до 5% бюджета спринта бесплатно.
  - Новое поле — paid CR (оценка в 1 ч бесплатно, реализация по ставкам).
  - Новый модуль или изменение бизнес-правила — paid CR + ADR + addendum к контракту.
- Scope register обновляется на каждом standup, виден всем.
- Weekly review с PM: что в backlog, что меняется, что cost'ит сколько.

### BR-03. Неоплата транша / задержка

**Что может случиться:** транш не оплачен в срок. Команда работает «в долг». Финансовый риск студии.

**Митигация:**
- SLA: 10 рабочих дней от signed acceptance.
- Просрочка >5 дней: e-mail напоминание PM + ваш бухгалтер.
- Просрочка >10 дней: эскалация + пеня 0.1% / день (стандарт ст. 395 ГК РФ).
- Просрочка >15 дней: stop work, команда переключается на других клиентов. Простой при возобновлении компенсируется по T&M.

### BR-04. Уход ключевого инженера

**Что может случиться:** заболел, уволился, ушёл в отпуск Tech Lead или ключевой Backend.

**Вероятность:** средняя за 4 мес (Stage A) и 10 мес (всё). **Влияние:** среднее.

**Митигация:**
- **Bus factor ≥2** на каждый модуль: знают как минимум 2 человека.
- Pair-programming + code review обязательны.
- Onboarding-doc на каждый модуль в Confluence.
- Weekly knowledge transfer sessions внутри команды.
- Запасной инженер из bench-пула — может подключиться в течение 5 раб. дней.
- На критических ролях (Tech Lead, DevOps) — backup изначально подобран и онбордится в фоне.
- Запрет на отпуск ключевых ролей на M4 (Stage A demo, 21 авг) и M6 (Pilot deploy, 16 окт).

### BR-05. Команда клиента не выделяет время на UAT / demo

**Что может случиться:** UAT окно (нед 19, 5 раб. дней) — а вы или ваши сотрудники не успели проверить. Pilot deploy сдвигается.

**Митигация:**
- **Контрактный SLA на ваши ответы** (24 ч на blocker, 3 раб. дня на feature decision, 5 раб. дней на demo acceptance).
- **Default acceptance** через 5 раб. дней молчания после demo (с напоминанием на 1-й и 3-й день).
- Demo окна резервируются за 1 нед заранее.
- UAT-окно S10 — обязательно 5 раб. дней владельца + сотрудников на staging.

### BR-06. Праздники и отпуска

**Что может случиться:** новогодние праздники (28 дек 2026 — 8 янв 2027), 23 фев 2027, 8 марта, 1-9 мая 2027 — производительность падает.

**Митигация:**
- Календарь учитывает все РФ-праздники.
- 30% buffer заложен в каждой фазе.
- Отпуска инженеров согласуются с PM за 30 дней, запрещены на критических спринтах.

### BR-07. Outage внешних провайдеров (YooKassa, ОФД, SMS, Hikvision)

**Что может случиться:** отказ работает один из провайдеров → часть функционала недоступна.

**Митигация:**
- **Provider abstraction** в коде на каждый внешний сервис.
- Mock-режим работает изначально (для разработки и при сбое).
- Outage runbook на каждый провайдер.
- Backup-провайдер по возможности (SMSAero ↔ MTS Exolve, Платформа ОФД ↔ Такском).
- SLA провайдера в **вашем** контракте — мы не отвечаем за outage YooKassa, но обходим его технически (queue, retry, fallback).

## 4. Регуляторные риски (РФ-законодательство)

### RR-01. 152-ФЗ — биометрия (Phase 2)

См. TR-07. Дополнительно:
- Письменное согласие сотрудника на обработку биометрических ПДн (ст. 11) — отдельная форма с электронной подписью / распиской.
- Уведомление РКН подаётся **вашим юристом** до старта pilot (через Госуслуги).
- Если кандидат отказывается от Face Control — alternative workflow (manual старт смены).

### RR-02. ТК РФ — сдельная зарплата (Phase 2)

**Риск:** ошибка в расчётах ЗП → трудовой спор / штрафы.

**Митигация:**
- BR-05: баланс сотрудника не уходит в минус (защита по ст. 137).
- Расчётный лист (ст. 136) формируется автоматически в PDF.
- Корректировки только через workflow с одобрением учредителя.
- Property-based tests на критические инварианты: «сумма выплачено ≤ сумма заработано + минималка», «закрытый период не меняется».

### RR-03. 402-ФЗ — хранение первичных документов

**Риск:** утеря счетов / актов / ТТН → штрафы при налоговой проверке.

**Митигация:**
- S3 retention 5 лет (BR-33), immutable после finalize.
- Daily backup, restore drill каждые 3 мес.

### RR-04. 54-ФЗ — фискальные чеки (Stage B)

См. TR-06.

### RR-05. ПДн в РФ-юрисдикции

**Риск:** хранение ПДн за границей нарушает 152-ФЗ ст. 18 ч. 5.

**Митигация:**
- Хостинг **только в РФ** (Yandex.Cloud / Selectel / VK Cloud / on-prem).
- Запрещено: AWS, GCP, Cloudflare R2, Hetzner, OVH (даже бесплатные планы).
- Прототип на Vercel допустим, потому что в нём моки без реальных ПДн.

## 6. Phased-подход как мета-митигация

Сама стратегия «Stage A → Stage B → Phase 2 → Phase 3» — это сильная мера управления рисками:

1. **Финансовый риск меньше**: вы платите 7-9 млн вначале, а не 15-19 за всё сразу. Если Stage A не нравится — не платите за Stage B.
2. **Можно остановиться после любого этапа**: каждый этап даёт production-готовую систему. Stage A — типография работает «по старинке для клиентов», но внутри — современный ERP. Stage B — клиенты получают портал. Phase 2 — финансы. Phase 3 — масштаб.
3. **Тестирование скопа**: Stage A показывает реальную способность команды и клиента работать вместе. Stage B / Phase 2 / Phase 3 контракты подписываются уже с проверенной командой.
4. **Меньше блокеров**: Stage A не зависит от Q5 (эквайринг), поэтому может стартовать сразу после Q1, Q3.
5. **Меньше cash-flow shock**: 8 траншей по Stage A, доли 5-20% — никаких «больших платежей в один момент».
