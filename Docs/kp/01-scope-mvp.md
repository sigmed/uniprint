# Что мы строим: скоп системы

> Подробное описание функционала по этапам. Внутри — что именно появится, скрины прототипа, на чём фокусируемся в каждый момент времени.

---

## 1. Архитектура решения (общая картина)

UniPrint — **единая платформа** с шестью рабочими местами («кабинетами»). Все кабинеты работают с одной базой данных, поэтому изменения в одном месте сразу видны в другом (заказ создан менеджером — производство видит его в очереди).

```
┌─────────────────┐  ┌─────────────────┐  ┌──────────────────┐
│  Клиент         │  │  Менеджер       │  │  Производство    │
│  (web)          │  │  (web)          │  │  (PWA / телефон) │
└────────┬────────┘  └────────┬────────┘  └────────┬─────────┘
         │                    │                    │
         └────────────────────┼────────────────────┘
                              │
                  ┌───────────▼───────────┐
                  │   Backend + БД        │
                  │   (Yandex.Cloud, РФ)  │
                  └───────────┬───────────┘
                              │
         ┌────────────────────┼────────────────────┐
         │                    │                    │
┌────────▼────────┐  ┌────────▼────────┐  ┌──────────────────┐
│  Склад          │  │  Администратор  │  │  Учредитель      │
│  (PWA / телефон)│  │  (web)          │  │  (web)           │
└─────────────────┘  └─────────────────┘  └──────────────────┘
```

**Технические решения** (для тех, кому интересно):
- Backend: современный TypeScript-стек на Node.js (NestJS + Prisma + PostgreSQL 16).
- Frontend: Next.js 16 + React 19 — шесть отдельных кабинетов на одном дизайн-системе.
- Mobile: PWA (Progressive Web Apps) — устанавливается на телефон сотрудника как обычное приложение, работает off-line.
- Хостинг: Yandex.Cloud (РФ-юрисдикция, соответствует 152-ФЗ).
- Безопасность: шифрование данных, audit-log на действия с персональными данными, внешний pentest перед запуском.

Подробности — в `Docs/03-architecture.md` (для разработчиков).

## 2. Stage A — Запуск производства (первые 4 месяца)

**Цель Stage A:** запустить **внутреннюю работу** типографии. Менеджеры принимают заказы, склад их обслуживает, производство выполняет, администратор управляет справочниками. Заказчики продолжают работать как раньше (звонок менеджеру, оплата по реквизитам / наличными).

### 2.1 Кабинет Менеджера (web)

Главный экран команды продаж. Что будет работать:

**Лиды и клиенты:**
- Воронка лидов с этапами (новый → квалифицирован → коммерческое → конверсия → отказ).
- Карточка клиента с историей заказов, контактами, контрагентами.
- **Антидублирование клиентов по телефону** (BR-02): попытка создать клиента с уже существующим номером — система предлагает merge или добавляет к существующей карточке. Это решает классическую боль типографий: «один и тот же клиент в базе 5 раз».

**Заказы:**
- Три типа заказов (BR-07): **цех** (наружная реклама — баннеры, вывески), **офис-полиграфия** (визитки, листовки, печать), **продажа товара** (готовая продукция со склада).
- Для каждого типа — своя статус-машина (стадия → стадия с проверкой).
- Расчёт суммы заказа автоматически по справочнику услуг и нормативам.
- История изменений каждого поля в audit-log.

### 2.2 Кабинет Производства (PWA на телефоне)

Мобильное приложение для печатников и станочников. Устанавливается на телефон как обычное приложение, работает в любой сети — даже в производственном цеху со слабым 3G.

**Что будет работать:**
- **Очередь задач** — печатник видит, что ему предстоит сделать, в порядке приоритета.
- Pickup задачи (взять в работу) — кнопка на 56px высотой, удобно тыкать пальцем в перчатке.
- Старт / пауза / финиш операции с таймером.
- Загрузка фото готовой работы (бэкап перед отправкой на склад).
- **Off-line режим:** если связь пропала на 5 минут — действия сохраняются локально и синхронизируются автоматически на reconnect. **Действие сотрудника не теряется.**
- Список ваших активных задач со статусами.

### 2.3 Кабинет Склада (PWA на телефоне)

Мобильное приложение для складщиков. Тоже PWA, работает off-line.

**Что будет работать:**
- **Списание материалов только на конкретный заказ** (BR-01) — нельзя списать «в цех вообще», обязательно к заказу. Это решает главную боль склада: «куда делись 200 м² баннерной ткани?».
- **FIFO по партиям** (BR-09): система автоматически списывает старейшую партию (срок годности, актуальная цена). Никаких ручных решений.
- **Антидублирование материалов** при приёмке: вводите название — система ищет похожее по нормализованному имени, артикулу, штрихкоду. Если уже есть — предлагает добавить к существующему материалу, а не создавать дубль.
- Инвентаризация через мобилу — складщик ходит по складу с телефоном, сканирует / выбирает материал, вводит фактический остаток. После завершения — отчёт о расхождениях.
- **Брак фиксирует только складщик** (BR-03) — производство не может «списать брак самостоятельно», это критичное правило для контроля качества. Складщик принимает дефектную продукцию, фотографирует, классифицирует (производственный / материальный / транспортный), система автоматически создаёт переделку (rework) и пересчитывает себестоимость заказа.

### 2.4 Кабинет Администратора (web)

Управление справочниками, пользователями, настройками.

**Что будет работать:**
- **Справочник услуг** с привязкой к типам заказов и нормативам расхода материалов (BR-04, BR-15).
- **Справочник материалов** с антидублированием.
- **Справочник нормативов** — сколько баннерной ткани на 1 м² с учётом подвёртки, отходов, температурного режима.
- **Пользователи и роли**: до 6 ролей (учредитель / директор / менеджер / производство / склад / клиент), у каждой свой доступ к данным (RBAC).
- **Журнал действий** (audit-log): кто что когда делал. По требованию 152-ФЗ + для разбора инцидентов.

**Скрин:** `Docs/design/screenshots/v5-5-admin-1440.png` ([uniprint-admin.vercel.app](https://uniprint-admin.vercel.app))

### 2.5 Что НЕ войдёт в Stage A

Эти функции — **в Stage B и Phase 2**, чтобы не раздуть Stage A:

| Функция | Куда переносится | Почему |
| --- | --- | --- |
| Клиентский портал (внешний кабинет для заказчиков) | Stage B | Клиенты пока продолжают работать как раньше — звонок, реквизиты, бумаги |
| Кабинет учредителя (KPI dashboard) | Stage B | Администратор частично закрывает потребность; полный dashboard — после Stage A pilot |
| Согласование макетов внутри системы | Stage B | В Stage A — макеты согласуются по почте / WhatsApp как раньше |
| Автоматическая генерация PDF (счёт, акт) | Stage B | Счета через 1С / банк, акты вручную |
| Гарантийные обращения (aftercare) | Stage B | Ведутся менеджерами вручную в CRM |
| Онлайн-оплата с фискальным чеком (54-ФЗ) | Stage B | Оплата по реквизитам / наличными |
| Контроль времени по лицу (Face Control) | Phase 2 | В Stage A — старт/стоп смены вручную через приложение |
| Сдельная зарплата с долговым балансом | Phase 2 | В Stage A — расчёт зарплаты в Excel / 1С |
| Себестоимость, P&L, налоги | Phase 2 | Учёт ведётся бухгалтером отдельно |
| Логистика, оборудование с амортизацией | Phase 2 | — |
| Несколько складов, BI, ЭДО | Phase 3 | Один цех, один склад в MVP |

## 3. Stage B — Клиентский опыт (+5-6 нед)

Запускается **после 2 недель стабильного pilot Stage A**. Превращает «внутреннюю систему» в **современный сервис для ваших заказчиков**.

### 3.1 Клиентский портал (web)

Ваши заказчики получают свой личный кабинет. Что они смогут:
- Видеть статус своего заказа в режиме реального времени (BR-32 — клиент видит только свои данные).
- Получать уведомления при смене статуса (e-mail, push в браузере).
- Скачивать документы (счёт, акт, договор-оферту) в PDF.
- Оплачивать заказ онлайн через YooKassa (карта / СБП / счёт юр. лица). Чек уходит на e-mail и в личный кабинет (54-ФЗ).
- Согласовывать макет (одобрить / запросить правки) — это видно дизайнеру и менеджеру в режиме реального времени.

**Скрины:**

**Live URL:** [uniprint-client.vercel.app](https://uniprint-client.vercel.app)

### 3.2 Кабинет Учредителя (web)

Полный dashboard для собственника:

- Нагрузка цеха — в реальном времени, по сменам.
- Drill-down: клик на «Выручка декабря» → разбивка по типам заказов → конкретные заказы → конкретные клиенты.
- Воронка лидов с конверсией по этапам.

**Live URL:** [uniprint-owner.vercel.app](https://uniprint-owner.vercel.app)

### 3.3 Автоматический документооборот

Раньше счёт делал бухгалтер в 1С. Теперь система сама генерирует:
- **Счёт** — после подтверждения заказа.
- **Акт выполненных работ** — после статуса «выдан».
- **Договор-оферту** — для одноразовых клиентов без подписания договора.
- **ТТН (товарно-транспортная накладная)** — для заказов с доставкой (Phase 2).

Все документы — в S3 с retention 5 лет (требование 402-ФЗ), immutable после finalize, шаблоны под ваш фирменный стиль.

### 3.4 Онлайн-оплата с фискальным чеком (54-ФЗ)

- Интеграция с YooKassa (или другим эквайером — Тинькофф / Сбер / СБП).
- Интеграция с ОФД (Платформа ОФД / Такском) для генерации фискального чека на каждую B2C-оплату.
- Idempotency-Key защита от двойного списания.
- Реконсиляция платежей с заказами ежедневно.
- Поддержка возвратов с чеком коррекции.

## 4. Phase 2 — Финансы и Face Control (+2 мес)

Запускается через 1-3 мес после стабилизации Stage B (или Stage A, если без B).

### 4.1 Контроль времени по лицу (Face Control)

Интеграция с системой видеонаблюдения (предполагаем Hikvision; финальный вендор — после ответа на Q2):
- Сотрудник проходит мимо камеры на входе в цех — система автоматически фиксирует начало смены.
- На выходе — конец смены.
- Опоздания, ранние уходы, idle-time — автоматические уведомления руководителю.
- **Биометрические согласия** по 152-ФЗ ст. 11 — отдельный workflow с электронной подписью сотрудника, шаблоны хранятся в защищённом storage (Yandex Lockbox + KMS).

### 4.2 Сдельная зарплата

Полный модуль расчёта ЗП:
- Двойная формула: оклад + сдельная (BR-31).
- Норматив времени на каждую операцию (закреплён в snapshot, BR-14).
- **Долговой баланс сотрудника** (BR-05): если за месяц человек не выработал минималку — компания доплачивает «в долг», следующий период вычитает из заработка. Без ухода в минус (защита по ТК ст. 137).
- Расчётный лист в PDF (требование ТК ст. 136).
- Корректировки задним числом — только через workflow с одобрением учредителя и audit-log.

### 4.3 Финансовый блок

- **Себестоимость заказа:** материалы FIFO + работа сдельная + амортизация оборудования + логистика + накладные расходы. По каждому заказу видно «сколько мы заработали».
- **P&L (отчёт о прибылях и убытках)** по периоду — день / неделя / месяц / квартал.
- **Налоги по СНО:** автоматический расчёт по выбранной системе налогообложения (УСН 6% / УСН 15% / ОСН / ПСН). Для ОСН — формирование первички для НДС.

### 4.4 Логистика

- Маршруты доставки на Yandex Maps.
- GPS-трекинг машин.
- Справочник транспортных средств с амортизацией.
- ТТН в PDF.

### 4.5 Оборудование

- Учёт станков, плоттеров, принтеров.
- Амортизация (BR-18) — линейная по сроку полезного использования.
- Уведомления о ТО (BR-29) — за 14 дней до планового технического обслуживания.
- Журнал использования оборудования по заказам (для расчёта себестоимости).

## 5. Phase 3 — Зрелость (+1 мес)

Запускается через ~6 мес работы Phase 2.

- **Поддержка нескольких складов / филиалов / городов** — для роста бизнеса.
- **Расширенная BI:** прогноз загрузки на месяц вперёд, cohort-анализ клиентов, ABC-XYZ.
- **ЭДО** (Контур.Диадок / СБИС) для B2B-документооборота с крупными корпоративными клиентами.
- **NPS-опросы** после aftercare + retargeting на лояльных.
- **Performance-оптимизация** после 6+ мес реальной нагрузки.
- **A/B testing** инфраструктура для клиентского портала.

## 7. Бизнес-правила в системе (краткий список)

В системе зашиты **36 бизнес-правил** (BR-XX), которые гарантируют корректность работы. Самые важные:

| BR | Что обеспечивает | Где работает |
| --- | --- | --- |
| **BR-01** | Материалы списываются только на конкретный заказ | Склад |
| **BR-02** | Нет дублей клиентов по телефону | Менеджер |
| **BR-03** | Брак фиксирует только складщик | Склад |
| **BR-04** | Услуги — только из справочника, не free-text | Менеджер, Производство |
| **BR-05** | Зарплата сотрудника не уходит в минус (долговой баланс) | Phase 2 |
| **BR-07** | Тип заказа неизменяем после создания | Менеджер |
| **BR-09** | FIFO-списание со склада | Склад |
| **BR-13** | Все изменения важных полей — в audit-log | Все |
| **BR-21** | Биометрические согласия — обязательно перед Face Control | Phase 2 |
| **BR-32** | Клиент видит только свои данные | Клиентский портал |

Полный список — в `BUSINESS_RULES.md` (приложение к контракту).
