## 1. Контекст и цели

### 1.1 Зачем КП

Phase-0 (прототип на моках) закрыт 2026-05-07. Прототип на Vercel принят владельцем, дизайн утверждён, скоп 25 модулей и 36 BR детально описан в `Docs/04-modules.md` и `BUSINESS_RULES.md`. Логичный следующий шаг — переход к production-разработке. Для этого клиенту нужно КП с:

- понятным описанием **что строим**;
- **жёсткими границами MVP** (что входит / что НЕ входит);
- **ценой и сроками** (с явным обозначением, где диапазоны и почему);
- **календарём по спринтам** и **графиком платежей**;
- **рисками** и **мерами их нивелирования**;
- **условиями** (приёмка, change requests, гарантия).

### 1.2 Аудитория

**Hands-on владелец бизнеса** (не IT-эксперт). Читает раз в две недели по 30 минут. Хочет:
- понять, что предлагается, без жаргона;
- увидеть **картинку** (живые скрины прототипа на Vercel);
- понять, **сколько и когда платить**;
- понять, **за что отвечает он**, а за что мы.

### 1.3 Принципы

1. **Доходчивость > полноты.** Если выбор между «полно описать» и «понятно объяснить» — выбираем второе.
2. **Честность > продажи.** Реальный диапазон цен (включая большой), реальные риски, реальные блокеры. Никаких «у нас всё под контролем».
3. **Минимум жаргона.** Технические термины — расшифровка в скобках или в глоссарии в `05-appendix.md`.
4. **Привязка к прототипу.** Везде, где возможно, — линки на live URL Vercel + скрины из `Docs/design/references/`.
5. **Каждая цена имеет «что входит» и «что вызовет доплату».** Никаких сюрпризов.
6. **График платежей привязан к milestones, не к календарю.** Клиент платит за результат.

## 3. Ключевые цифры (Phased: Stage A + Stage B)

> **🔒 ВНУТРЕННИЙ РАСЧЁТ (НЕ для КП).** Все часы и команда ниже учитывают AI-инструменты разработки (Claude Code agents). В КП клиенту это **НЕ упоминается** ни в одном файле, ни в одной таблице. Клиент видит только: фиксированную цену по этапам, календарные сроки, абстрактное описание команды («senior-команда специалистов»).
>
> **Реальный коэффициент ускорения** — взвешенный по типу задач:
> - Архитектура, дизайн, debug, integration с реальными API: 1.0-1.5×
> - Code review (senior проверяет AI-код): 1.5-2×
> - CRUD endpoints, REST, FE-компоненты: 3-5×
> - Tests (TDD), документация: 4-6×
> - **Средневзвешенное по проекту: ~2.7×**
>
> **Команда:** 4-5 senior-инженеров вместо 9-10 (Tech Lead + 1-2 backend + 1 frontend/PWA + DevOps/QA hybrid + part-time PM). В КП «состав» описывается абстрактно («senior-команда» без числа).

### 3.1 Что такое Stage A и Stage B

**Stage A — «Minimum Viable Typography»** — минимально достаточная система для **внутренней работы** типографии: производство, склад, заказы, сотрудники. Клиенты типографии работают как раньше (звонок менеджеру, наличные / банковский перевод по реквизитам, бумажные накладные / счета от 1С). Внутри — современный ERP.

**Stage B — «Customer Experience extension»** — добавляет к Stage A: клиентский портал, owner-dashboard, автоматические документы (счёт + акт + договор + ТТН), приёмку онлайн-оплаты с фискальными чеками. Это «современная типография» с self-service для клиентов.

Stage B запускается **после успеха Stage A в production** (минимум 2 нед стабильного pilot). Клиент может остановиться после Stage A — это валидное состояние системы.

## 4. Ценообразование (Phased)

### 4.1 Расчёт

> **🔒 Внутренняя экономика (не для КП). Параметры от владельца 2026-05-08:**
> - **ЗП gross 170 тыс ₽/мес/senior** (≈148 тыс на руки)
> - **+43% налоги на ЗП** (НДФЛ 13% + страховые 30%) → **FOT per senior = 243 тыс ₽/мес**
> - **+50% overhead** (офис, инструменты, маркетинг, бенч-buffer, бухгалтерия) — считается от FOT
> - **+ Claude API ~30 тыс ₽/мес/инженер** (отдельно от overhead, потому что прямой production cost)
> - **Total business cost per FTE ≈ 363 тыс ₽/мес** = ~91 тыс ₽/чел-нед при 0.85 утилизации = эквивалент **2 700 ₽/AI-час**
> - **Маржа: markup 50% от internal cost** (price = cost × 1.5) — gross margin ~33% от revenue
>
> **Для клиента** в КП показываем **только итоговую цену по этапам** — без часов, без ставок, без размера команды. Fix-price = клиент не считает часы.

**Внутренний расчёт (не для клиента):**

| Stage | AI-часов | Чел-мес | Internal cost (363 тыс/мес + 10% risk) | КП-цена | Эффективный markup |
| --- | ---: | ---: | ---: | ---: | ---: |
| **Stage A** (fix-price) | ~1 645 | ~14.6 (4 FTE × 3 мес + PM 0.5×3) | **~5.4 млн ₽** | **7-9 млн ₽** (mid 8) | 30-67% (mid 50%) |
| **Stage B** (fix-price) | ~515 | ~4.6 (4 FTE × 1 мес + PM 0.3×1) | **~1.7 млн ₽** | **2-3 млн ₽** (mid 2.5) | 18-76% (mid 47%) |

> **⚠️ Защита от давления вниз на Stage A:** floor = **7 млн ₽** (markup ~30%). Ниже — можно только урезать скоп (выкинуть admin-panel, оставить 3 кабинета).

> **⚠️ T&M-фазы (P2/P3): markup ниже** (mid 35%) — это **компромисс** между «региональными» ставками клиенту и фиксированной internal cost. Это нормально для T&M-бизнеса (типичный markup на T&M в РФ = 25-40%). Если хочется выше — поднимать ставки в КП до Москва/СПб уровня (4 500-6 000 ₽/ч), но владелец явно выбрал региональные.

### 4.2 Пошаговая логика для клиента

1. **Шаг 1:** заплатить за Stage A (**7-9 млн ₽**), получить production-готовую систему для внутренней работы за ~4 мес. **Можно остановиться**, типография работает.
2. **Шаг 2 (опционально):** ещё через 2-4 мес после стабилизации Stage A — заплатить за Stage B (**2-3 млн ₽**), получить клиентский портал, owner-dashboard и автоматические документы.
3. **Шаг 3 (опционально):** через 1-3 мес — стартануть Phase 2 (Face Control + payroll + finance + logistics + equipment) на T&M (**~4-5 млн ₽**).
4. **Шаг 4 (опционально):** через ~6 мес — Phase 3 (multi-branch + BI + ЭДО) на T&M или retainer (**~1.5-2 млн ₽**).

**Total ownership cost (всё вместе):** **~15-19 млн ₽** за ~8-10 мес.

### 5.3 Phased-подход как способ нивелирования рисков

Phased сам по себе — мощная мера управления рисками:
- **Финансовый риск меньше**: клиент платит 7-9 млн вначале, не 9-12 (Full). Риск перерасхода на стороне команды, а не клиента.
- **Можно остановиться**: после Stage A типография уже работает. Если Stage B оказался не нужен — клиент не платит.
- **Тестирование скопа**: Stage A показывает реальную способность команды и клиента работать вместе. Stage B контракт подписывается уже с тестированной командой.
- **Меньше блокеров**: Stage A не зависит от Q5 (эквайринг), поэтому может стартовать сразу после Q1, Q3.
