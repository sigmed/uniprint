## Что это

**UniPrint ERP & Client System** — единая цифровая платформа для
типографии и производства наружной рекламы. Заменяет связку
«Excel + мессенджеры + устные договорённости» на единый контур:
заказ от клиента → дизайн → производство → склад → выдача → расчёт
себестоимости и ЗП → клиентский кабинет с историей и документами.
Greenfield (без legacy-кодовой базы), допущение по юрисдикции — РФ
(финал после ответа 🔴 #1).

## Бизнес-проблема (AS-IS)

Из ТЗ § 2.2, агрегировано до 7 болей владельца:

1. **Финансовые потери** — неучтённый брак, перерасход материалов,
   ошибки в расчётах. Реальная себестоимость заказа неизвестна.
2. **Отсутствие прозрачности** — невозможно отследить действия
   сотрудников, определить ответственного за брак или простой.
3. **Списание материалов «в цех», а не на заказ** — критическая
   проблема (ТЗ § 2.1): нельзя посчитать себестоимость, выявить
   перерасход, найти виновного.
4. **Дублирование данных** — клиенты («Иванов» × 3 раза в базе),
   услуги («Баннер» / «баннер» / «банер» / «другое»). Аналитика
   недостоверна.
5. **Слабое управление производством** — задачи передаются устно
   или в чате, время операций не фиксируется, простои невидимы.
6. **Фиксированная ЗП** — нет связи между результатом и доходом
   сотрудника. Низкая мотивация, низкий КПД.
7. **Нет клиентского кабинета** — вся коммуникация через менеджера,
   менеджер — bottleneck. Заказы регистрируются с задержкой 2-3 дня.

## Решение (6 контуров)

| Контур | Что закрывает | Ключевые модули ТЗ |
| --- | --- | --- |
| **Продажи (CRM)** | Лиды, заказы, антидублирование клиентов, расчёт цены, выездные продажи | 6.1, 6.2, 6.3, 6.14 |
| **Производство** | 3 типа заказа (цех / офис-полиграфия / товар), задачи, операции, учёт времени, нормирование | 6.5, 6.6, 6.7, 6.8, 6.10, 6.13 |
| **Склад** | Антидублирование материалов, приёмка, **списание только на заказ (BR-01)**, инвентаризация, **брак (BR-03)** | 6.11, 6.12 |
| **Финансы** | Себестоимость, прибыль по заказу, ЗП сдельная, баланс сотрудника, амортизация, налоги, логистика | 7.1-7.18, 6.22 |
| **HR / контроль** | Face Control, KPI, рейтинг, простои, постобслуживание | 6.9, 6.20, 6.17 |
| **Клиентский кабинет** | Самообслуживание клиента: заказы, статусы, история, оплата, документы | 6.15, 6.23, 9.11 |

Дополнительно (модули из дополнения ТЗ): **6.23** документооборот
(счёт / договор / акт / ТТН), **6.24** логистика (водители + Yandex Maps),
**6.25** мониторинг оборудования (наработка часов, ТО).

**Каналы.** **Mobile = PWA mobile-first** (Next.js 16 + Service Worker
+ manifest) — единая web-сборка, ставится на смартфон / планшет
сотрудника как иконка, работает в реальном времени с production-mobile
и warehouse-mobile интерфейсами. **Web Panel** — менеджеры, дизайнеры,
руководство, **клиентский кабинет**. **Каналы нотификаций** — SMS +
Email + WebPush (Telegram **не используется** как продуктовый канал,
см. ответ 🔴 #7 от 2026-05-05). Native-приложения — рассматриваются
позже (фаза 3+) при появлении нужды в hardware-доступе.

## Ключевые метрики продукта (KPI)

Целевые значения — рабочая гипотеза, калибруются по baseline после
миграции (🔴 #4). Полный разбор — `01-vision.md` § KPI.

| KPI | Метрика | Цель v1 | Источник |
| --- | --- | --- | --- |
| Прозрачность списаний | % материалов, списанных на конкретный заказ | **100%** (BR-01) | `apps/warehouse` |
| Точность себестоимости | дельта (план − факт) по заказу | < ±10% | `apps/finance` |
| Брак | % заказов с зафиксированным браком | < 3% | `apps/defects` |
| Перерасход материалов | факт − норма по заказу | < +5% | `apps/warehouse` |
| Время «заказ → выдача» | медиана по типу заказа (цех / офис / товар) | цех ≤ 5 дней, офис ≤ 1 день, товар ≤ 1 час | статус-машина |
| Adoption клиентского кабинета | % заказов, созданных клиентом самостоятельно | ≥ 30% к концу года | client-portal |
| Простои Face Control | (присутствие − работа) / присутствие | < 15% | модуль 6.20 |
| Сдельная ЗП к ФОТ | доля переменной части | ≥ 60% | `apps/payroll` |
| Скорость регистрации заказа | от обращения клиента до записи в системе | < 5 мин (vs 2-3 дня сейчас) | `apps/orders` |
| Удовлетворённость клиента | оценка постобслуживания (модуль 6.17) | ≥ 4.5 / 5 | `apps/feedback` |

## Ограничения и допущения

**Закрытые ответы клиента (2026-05-05) — фиксированные допущения:**

- **Mobile-стек** — **PWA mobile-first** (Next.js + Service Worker +
  manifest). Native не в Phase-1.
- **Telegram** — НЕ используется в продукте. Все нотификации — через
  SMS + Email + WebPush. (Telegram остаётся **только** PM-каналом
  команда↔владелец — внутреннее, не часть продукта.)
- **Карты** — Yandex Maps (логистика 6.24, ETA доставки).
- **Хранилище макетов** — Yandex Object Storage (S3-совместимое).
- **Один цех + один склад MVP**, multi-warehouse FK в данных, UI на
  одно подразделение.
- **Документооборот** — простая генерация PDF (счёт / договор-оферта
  / акт / ТТН) в S3 + audit-log. ЭДО (Контур.Диадок / СБИС) — фаза 2+.
- **54-ФЗ.** Микс B2B + B2C → **ОФД нужен** для B2C-чеков
  (под-блокер 🔴 #5 эквайринг).
- **Справочник услуг и материалов** — собирается **с нуля**
  параллельным треком в sprint-0 (R3 типограф-консультант + владелец).
- **Объём.** 10 заказов/день старт, 20 одновременных пользователей
  (NFR 9.1), 30 сотрудников всего, 50 активных клиентов на старте,
  ~200 SKU материалов (поправим при появлении точных данных).
- **Сдельная ЗП.** Двойная формула — «тариф ₽ × количество» **ИЛИ**
  «% от стоимости работы для клиента» (per operation/role).
- **Tracker** — GitHub Issues. **Спринт** — 2 недели. **Demo URL** —
  Vercel preview.
- **Sentry / observability** — отложено (по умолчанию фаза 1.5+).

**Открытые блокеры (5 🔴):**

- 🔴 **Q1 Юрисдикция** (ИП/ООО, СНО, НДС) — блокирует ADR-0003,
  ADR-0005, договоры. Допущение: **РФ** по терминологии ТЗ.
- 🔴 **Q2 Face Control vendor** (Hikvision / Suprema / NtechLab /
  самописный) — блокирует ADR-0002 + форму биометрического согласия.
- 🔴 **Q3 Хостинг** (Yandex.Cloud / Selectel / VK Cloud / on-prem) —
  блокирует ADR-0003.
- 🔴 **Q4 Миграция legacy** (1С / Excel / нет) — блокирует sprint-1
  milestone, объём.
- 🔴 **Q5 Эквайринг** (YooKassa / Тинькофф / Сбер / СБП) + ОФД-

**Compliance baseline:** 152-ФЗ ст. 11 (биометрия Face Control —
отдельное письменное согласие сотрудника), ТК РФ ст. 136-137
(расчётный лист + ограничение удержаний по сдельной ЗП), 402-ФЗ
(хранение первички 5 лет), 54-ФЗ (микс B2B + B2C → ОФД нужен),

## Что НЕ входит (out of scope)

Согласовано с владельцем (см. `team-structure.md` § Out of scope):

- **Дизайн макетов клиентов** — это работа дизайнеров типографии,
  не платформенная функция. Платформа лишь хранит и согласует.
- **Юр-сопровождение** (договор оферты, политика ПДн, согласия
  биометрии — текстовый контент готовит юрист-аутсорс ~80–150 тыс ₽).
- **Закупка оборудования** (печатные станки, лазер, плоттер, монтажный
  инструмент) — решение владельца.
- **Закупка материалов** (бумага, плёнка, краски, баннерная ткань) —
  оперативная закупка по уведомлениям системы, но не сама закупка.
- **Бухгалтерия и налоговая отчётность** — аутсорс (1С / СБИС / Контур).
  UniPrint выдаёт первичку (счёт / акт / ТТН), бухгалтер заносит в 1С.
- **Цветопроба, ICC-калибровка, Pantone** — типография делает на своём
  оборудовании; платформа лишь хранит требования к макету.
- **Аренда производственного помещения, HR (наём печатников)** —
  не платформенно.
- **ЭДО (Контур.Диадок / СБИС), ЭЦП клиента** — фаза 2+ при появлении
  B2B-спроса.
- **Native mobile-приложения** — фаза 3+ при появлении нужды в
  hardware-доступе.

## Команда

Виртуальная команда «как-если-бы» из 18 ролей (+ 3 резервных) — полный
маппинг на агентов и скиллы в [`team-structure.md`](team-structure.md).
Ключевые роли для UniPrint: **Senior Frontend** (PWA mobile-first —
основной канал производства и склада), **Compliance / RA-QM**
(биометрия 152-ФЗ ст. 11), **Финансовый аналитик** (валидация формулы
сдельной ЗП), **R3 — полиграфический консультант** (припуски, m.п.,
оракал, ICC, наполнение справочника с нуля).
