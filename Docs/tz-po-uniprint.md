# 1. Общие положения

Система предназначена для автоматизации управления производством наружной рекламы, контроля сотрудников, складского учёта и взаимодействия с клиентами.

## 1.1 Название системы

- **Полное наименование:** Информационная система управления бизнес-процессами и клиентским взаимодействием компании UniPrint
- **Краткое наименование:** UniPrint ERP & Client System

## 1.2 Основание для разработки

Основанием для разработки системы является необходимость комплексной цифровизации бизнес-процессов компании UniPrint, включая внутренние процессы и взаимодействие с клиентами.

Разработка системы инициирована руководством компании в связи с выявленными проблемами:

- отсутствие прозрачного учёта материалов и операций
- высокий уровень брака без фиксации причин и ответственных
- неэффективная система оплаты труда сотрудников
- отсутствие автоматизированного расчёта себестоимости и цен
- разрозненные каналы коммуникации (устно, мессенджеры, Excel)
- высокая зависимость от человеческого фактора
- отсутствие удобного интерфейса для клиентов (заказы, статус, расчёт)

Система разрабатывается как единая цифровая платформа, объединяющая: управление заказами, производство, склад, финансы, персонал, клиентский кабинет.

## 1.3 Цели и задачи системы

**Цель системы.** Создание единой цифровой платформы для автоматизации и управления всеми бизнес-процессами UniPrint, а также обеспечения удобного и прозрачного взаимодействия с клиентами.

**Основные задачи системы:**

- Автоматизация приёма заказов (включая клиентский кабинет)
- Мгновенный расчёт стоимости услуг для клиентов и менеджеров
- Учёт и контроль материалов на складе
- Контроль производственных процессов
- Учёт брака и финансовых потерь
- Расчёт заработной платы сотрудников (сдельная модель)
- Контроль эффективности сотрудников (KPI)
- Обеспечение прозрачности всех операций
- Предоставление клиентам доступа к: оформлению заказов, отслеживанию статуса, истории заказов
- Формирование аналитики и отчётности для руководства

## 1.4 Область применения

Система предназначена для использования в компании UniPrint и охватывает как внутренние, так и внешние бизнес-процессы.

**Внутренние процессы:**

- приём и обработка заказов
- производство полиграфической продукции
- складской учёт материалов
- управление персоналом
- финансовый учёт и анализ

**Внешние процессы (клиентский кабинет):**

- оформление заказов клиентами
- автоматический расчёт стоимости
- отслеживание статуса заказов
- доступ к истории заказов

**Пользователи системы:**

- Внутренние: менеджеры по продажам, дизайнеры, сотрудники производства (полиграфисты), складщики, бухгалтерия, руководство
- Внешние: клиенты компании

## 1.5 Определения и термины

| Термин | Определение |
| --- | --- |
| Заказ | Заявка клиента на изготовление продукции |
| Клиентский кабинет | Интерфейс для клиентов для создания и отслеживания заказов |
| Материал | Расходуемый ресурс (баннер, оракал, краска и др.) |
| Списание | Передача материалов со склада в производство |
| Брак | Испорченная продукция или материал |
| Норма расхода | Установленный расход материала на единицу продукции |
| Перерасход | Превышение фактического расхода над нормативом |
| Партия (Batch) | Группа материалов, поступивших одновременно |
| Себестоимость | Суммарные затраты на выполнение заказа |
| Прибыль | Разница между доходом и затратами |
| Операция | Конкретное действие (печать, резка, монтаж и т.д.) |
| KPI | Показатели эффективности сотрудников |
| Сдельная оплата | Система оплаты труда, зависящая от объёма выполненной работы |

# 2. Бизнес-контекст

## 2.1 Описание текущих процессов (AS-IS)

На текущий момент бизнес-процессы компании UniPrint характеризуются низким уровнем автоматизации, отсутствием стандартизации и высокой зависимостью от человеческого фактора.

### Приём и регистрация заказов

- Заказы поступают через: мессенджеры, телефон, устные договорённости
- Менеджеры вносят заказы в систему с задержкой до 2–3 дней

**Проблемы:** потеря данных, искажение информации, невозможность контроля сроков, снижение качества обслуживания.

### Ввод услуг

- Услуги вводятся вручную
- Примеры: «Баннер», «баннер», «банер», «другое»

**Проблемы:** дублирование услуг, невозможность корректной аналитики, хаос в базе.

### Учёт клиентов

- Клиенты создаются вручную, отсутствует проверка на дубли

**Проблемы:** один клиент может быть зарегистрирован несколько раз, база клиентов становится неструктурированной.

### Классификация заказов

- Тип заказа (товар / услуга) не фиксируется

**Проблемы:** ошибки в финансовом учёте, невозможность корректной аналитики.

### Производство

- Задачи передаются устно или через чат
- Отсутствует система контроля выполнения
- Не фиксируется время операций

**Проблемы:** низкая производительность, отсутствие прозрачности.

### Складской учёт

- Склад ведётся частично в системе, структура данных неудобна для анализа
- Отсутствует информация о последней закупке, количестве последней закупки
- Данные о поступлениях есть, но: требуется открывать каждую запись отдельно; отсутствует агрегированная аналитика

**Итог:** замедление принятия решений, ошибки в закупках.

### Учёт расхода материалов

- Материалы списываются в цех, а не на конкретный заказ

**Проблемы:** невозможно определить, куда ушёл материал; отсутствует связь с заказом; невозможно рассчитать точную себестоимость; невозможно определить ответственного; не выявляется перерасход.

> Это является **критической проблемой системы**.

### Учёт брака

- Брак не фиксируется, отсутствует система ответственности

**Проблемы:** прямые финансовые потери, отсутствие контроля.

### Управление персоналом

- Фиксированная заработная плата, отсутствует связь между результатом и доходом

**Проблемы:** низкая мотивация, низкий КПД.

### Работа с клиентами

- Отсутствует клиентский кабинет, вся коммуникация через менеджеров

**Проблемы:** перегруз менеджеров, медленный сервис, снижение конверсии.

## 2.2 Проблемы бизнеса

На основе анализа выявлены ключевые системные проблемы:

- **Финансовые потери:** неучтённый брак, перерасход материалов, ошибки в расчётах
- **Отсутствие прозрачности:** невозможно отследить действия сотрудников, невозможно определить ответственность
- **Низкая эффективность процессов:** задержки в обработке заказов, ручные операции, ошибки
- **Дублирование данных:** услуги, клиенты
- **Неэффективное управление складом:** неудобная аналитика, отсутствие ключевых показателей
- **Слабая система продаж:** отсутствие контроля менеджеров, задержка регистрации заказов
- **Отсутствие клиентского сервиса:** нет онлайн-заказа, нет отслеживания статуса, нет удобства для клиента

## 2.3 Цели автоматизации (TO-BE)

Целевая модель предполагает переход компании UniPrint от ручного и разрозненного управления к единой цифровой системе, обеспечивающей полный контроль процессов, данных и финансов.

- **Автоматизация заказов:** регистрация заказа в момент обращения, фиксация времени и менеджера
- **Стандартизация услуг:** единый справочник, запрет ручного ввода
- **Управление клиентами:** антидублирование, проверка по номеру телефона
- **Классификация заказов:** обязательное поле «товар / услуга»
- **Управление складом:** удобная структура данных
- **Аналитика склада:** агрегированные данные, быстрый доступ
- **Контроль движения материалов (КЛЮЧЕВОЕ):** материалы списываются на заказ, а не в цех. Система должна фиксировать: заказ, материал, количество, сотрудника, этап. Результат: точная себестоимость, контроль перерасхода, прозрачность
- **Нормативы расхода:** внедрение норм, сравнение план/факт
- **Учёт брака:** фиксация, определение ответственного, расчёт потерь
- **Управление персоналом:** сдельная оплата, KPI, персональный баланс
- **Клиентский кабинет:** оформление заказов, расчёт цены, статус
- **Финансовая система:** расчёт себестоимости, расчёт прибыли, распределение

## 2.4 Ожидаемый эффект (деньги, эффективность)

- **Финансовый эффект:** снижение потерь, увеличение прибыли, контроль себестоимости
- **Операционный эффект:** ускорение процессов, снижение ошибок
- **Эффект для персонала:** рост эффективности, повышение ответственности
- **Эффект для продаж:** рост конверсии, ускорение работы
- **Эффект для клиентов:** удобство, прозрачность
- **Управленческий эффект:** полный контроль бизнеса, прозрачность, аналитика
- **Стратегический эффект:** переход от «ручного бизнеса» к цифровой системе, возможность масштабирования, формирование конкурентного преимущества

# 3. Общая архитектура системы

## 3.1 Общая схема системы

Информационная система UniPrint представляет собой многоуровневую цифровую платформу, включающую:

- клиентский уровень (интерфейсы пользователей)
- серверный уровень (бизнес-логика)
- уровень хранения данных (база данных)

**Общая логика работы:**

```
Пользователь (менеджер / складщик / клиент)
  → взаимодействует через приложение (мобильное, веб или Telegram-бот)
  → данные отправляются на сервер (Backend)
  → сервер обрабатывает данные
  → данные сохраняются в базе
  → результат возвращается пользователю
```

Все процессы проходят через единую систему, что обеспечивает: прозрачность, контроль, целостность данных.

## 3.2 Компоненты системы

### 3.2.1 Мобильное приложение (Mobile App)

Приложение предназначено для сотрудников.

**Пользователи** (могут также через Telegram-бот): учредитель, водитель, начальник цеха, печатник, лазерщик, плотник, складщик, монтажник, полиграфист, менеджер по продажам, клиенты.

**Основные функции:**

- **Склад:** приёмка материалов, списание на заказ, контроль остатков, инвентаризация
- **Производство:** просмотр задач, выбор заказа, выбор операции
- **Учёт времени:** кнопка «Начать работу», кнопка «Завершить работу», автоматический таймер
- Каждая операция фиксируется: заказ, сотрудник, операция, время
- **Учёт брака:** фиксация, фото, причина, сотрудник

**Особенности:** быстрый интерфейс, минимальные действия, работа в реальном времени.

### 3.2.2 Веб-интерфейс (Web Panel)

**Пользователи:** отдел развития, руководство, администраторы, клиенты (клиентский кабинет).

**Основные функции:**

- создание и управление заказами
- расчёт стоимости
- аналитика и отчёты
- управление сотрудниками
- клиентский кабинет

**Отдельный модуль — Клиентский портал:** оформление заказов, отслеживание статуса, история заказов.

### 3.2.3 Backend (Сервер)

Центральный элемент системы, реализующий бизнес-логику.

**Основные функции:**

- обработка всех операций
- управление заказами
- расчёт: себестоимости, расхода материалов, зарплаты сотрудников, времени выполнения работ

**Ключевая логика:**

- материалы списываются только на заказ
- все действия привязаны к сотруднику
- все операции фиксируются

**Дополнительно:**

- антидублирование клиентов и услуг
- контроль бизнес-правил: обязательные поля, валидации
- интеграция с системой Face Control: обработка данных о присутствии сотрудников

> Backend является «ядром системы».

### 3.2.4 База данных (Database)

Хранит все данные системы.

**Основные сущности:** заказы, клиенты, материалы, движения материалов, сотрудники, операции, брак, финансы.

**Требования:** целостность данных, отсутствие дублей, высокая скорость доступа.

### 3.2.5 Telegram-бот (дополнительный модуль)

Используется как вспомогательный инструмент.

**Функции:**

- уведомления: новый заказ, назначение задачи
- быстрые действия: начать работу, завершить работу
- просмотр заработанной суммы

**Ограничения:** не используется для сложных операций, не заменяет основную систему.

### 3.2.6 Система контроля доступа (Face Control)

Интегрируется для фиксации присутствия сотрудников.

**Функции:** фиксация входа, фиксация выхода, передача данных в систему.

**Использование:** сравнение времени присутствия и времени работы.

**Результат:** расчёт занятости, выявление простоев, контроль дисциплины.

## 3.3 Модули системы

Система включает следующие функциональные модули:

1. Модуль управления заказами (CRM)
2. Модуль выездных продаж (Leads)
3. Модуль управления клиентами
4. Модуль услуг и товаров
5. Модуль дизайна
6. Модуль производства
7. Модуль учёта рабочего времени
8. Модуль контроля загрузки сотрудников
9. Модуль нормирования операций
10. Модуль склада
11. Модуль контроля качества (брак)
12. Модуль продаж
13. Модуль аналитики
14. Модуль управления пользователями

# 4. Роли пользователей

## 4.1 Складщик

**Назначение роли:** ответственный за учёт материалов, контроль остатков и контроль качества готовой продукции (учёт брака).

**Функции:**

- **Склад:** приёмка материалов, внесение поступлений, контроль остатков, списание материалов на конкретный заказ, возврат материалов, инвентаризация
- **Контроль качества (ключевая роль):** приёмка готового заказа от производства, проверка качества продукции

**Учёт брака.** При обнаружении брака складщик обязан:

- зафиксировать брак в системе
- сделать фото
- указать количество
- указать предполагаемый этап: дизайн, производство, материал, конкретно чья ошибка

**Повторное производство:** инициировать переделку заказа, списать материалы повторно, передать заказ обратно в производство.

**Ответственность:** точность складского учёта, контроль остатков, выявление брака, корректная фиксация потерь.

## 4.2 Производство (полиграфист / лазерщик / монтажник)

**Назначение роли:** выполнение производственных операций по заказам.

**Функции:** просмотр задач, выбор заказа, выполнение операций (печать, резка, монтаж и др.).

**Учёт работы:** «Начать работу», «Завершить работу», фиксация времени.

**Ограничение:** производство НЕ фиксирует брак в системе — ответственность за фиксацию брака лежит на складщике.

**Ответственность:** выполнение задач, соблюдение сроков, качество работы.

## 4.3 Менеджер

**Назначение роли:** работа с клиентами и оформление заказов.

**Функции:**

- создание заказа в момент обращения клиента
- выбор услуг из справочника
- выбор/создание клиента (без дублей)
- указание типа заказа: услуга (цех), услуга (офис / полиграфия), продажа (товар)

**Работа с ценой:** автоматический расчёт стоимости, согласование с клиентом.

**Контроль:** отслеживание статуса заказа, взаимодействие с производством.

**Ответственность:** корректность данных, скорость обработки, отсутствие ошибок.

## 4.4 Дизайнер

**Назначение роли:** подготовка макетов и передача заказов на мастер для проверки ошибок (грамматических).

**Функции:** получение задач, подготовка макета, загрузка файла, передача в производство.

**Важные действия:** изменение статуса «На дизайне» → «Готов к производству».

**Ответственность:** качество макета, отсутствие ошибок, соответствие требованиям.

## 4.5 Администратор

**Назначение роли:** управление системой и настройками.

**Функции:** управление пользователями, настройка ролей, управление справочниками (услуги, материалы, операции).

**Контроль:** настройка нормативов, контроль системы.

**Ответственность:** корректная работа системы, актуальность данных.

## 4.6 Руководитель (Owner / Director)

**Назначение роли:** контроль бизнеса и принятие решений.

**Функции:** просмотр аналитики, контроль прибыли, анализ брака, перерасхода, сотрудников.

**Ответственность:** управление эффективностью бизнеса.

## 4.7 Клиент

**Назначение роли:** внешний пользователь системы (клиентский кабинет).

**Функции:** создание заказов, просмотр стоимости, отслеживание статуса, история заказов.

**Ограничения:** нет доступа к внутренним данным, к себестоимости, к сотрудникам.

# 5. Бизнес-процессы (на основе BPMN)

## 5.1 Определение типа заказа

**Участники:** менеджер (выездной или офисный).

**Описание процесса.** При создании заказа менеджер обязан выбрать тип выполнения заказа, от которого зависит дальнейший бизнес-процесс.

**Варианты типа заказа:**

1. Услуга (цех) — наружная реклама и сложные изделия
2. Услуга (офис / полиграфия) — оперативная печать
3. Продажа (товар) — реализация готовой продукции

**Результат:** система автоматически определяет сценарий обработки заказа.

### 5.1.1 Услуга (цех)

#### 5.1.1.1 Поиск клиента и первичный контакт

**Участники:** выездной менеджер.

**Процесс:** поиск клиента (обход, рекомендации и др.), установление контакта, выявление потребностей клиента, предварительное обсуждение услуги.

**Результат:** создан потенциальный заказ (лид).

#### 5.1.1.2 Замер и оформление заказа

**Участники:** выездной менеджер.

**Процесс:** выезд на объект, выполнение замеров, фиксация (размеров, фото, требований), создание заказа в системе.

**Результат:** заказ зарегистрирован в системе.

#### 5.1.1.3 Передача заказа дизайнеру

**Участники:** менеджер / система.

**Процесс:** отправка заказа дизайнеру, передача всех данных и требований.

**Результат:** статус «На дизайне».

#### 5.1.1.4 Подготовка макета (дизайн)

**Участники:** дизайнер.

**Процесс:** разработка макета, проверка размеров и расположения, загрузка макета в систему.

**Результат:** макет подготовлен.

#### 5.1.1.5 Проверка макета (руководитель производства)

**Участники:** руководитель производства.

**Процесс:** проверка макета (грамматика, размеры, корректность).

**Варианты:**

- ❌ есть ошибки → возврат дизайнеру
- ✅ всё корректно → утверждение

**Результат:** макет проверен и утверждён.

#### 5.1.1.6 Согласование макета с клиентом

**Участники:** менеджер (или система — Telegram-бот) / клиент.

**Процесс:** отправка макета клиенту, получение обратной связи.

**Варианты:**

- ❌ правки → возврат дизайнеру
- ✅ подтверждение → переход дальше

**Результат:** макет согласован.

#### 5.1.1.7 Подготовка макета к производству

**Участники:** дизайнер.

**Процесс:** адаптация макета под оборудование и под формат печати.

**Результат:** макет готов к производству.

#### 5.1.1.8 Передача заказа в производство

**Участники:** система.

**Процесс:** передача заказа в цех, формирование задач.

**Результат:** статус «В производстве».

#### 5.1.1.9 Выполнение заказа в производстве

**Участники:** производственные сотрудники.

**Процесс:** выбор заказа, выбор операции, выполнение операций в зависимости от типа изделия.

**Фиксируется:** операция, сотрудник, заказ.

**Результат:** изделие изготовлено.

#### 5.1.1.10 Учёт рабочего времени

**Участники:** производство.

**Процесс:** «Начать работу» → «Завершить работу», автоматический расчёт времени.

**Результат:** данные для анализа эффективности.

#### 5.1.1.11 Списание материалов на заказ

**Участники:** складщик.

**Процесс:** выбор заказа, выбор материалов, указание количества, подтверждение.

**Результат:** материалы списаны, остатки обновлены.

#### 5.1.1.12 Передача готового заказа на склад

**Участники:** производство.

**Процесс:** передача изделия складщику.

**Результат:** статус «На проверке».

#### 5.1.1.13 Контроль качества и учёт брака

**Участники:** складщик.

**Процесс:** проверка качества.

**Варианты:**

- ✅ всё корректно → заказ готов
- ❌ брак: фиксация, фото, указание этапа

**Результат:** заказ принят или отправлен на переделку.

#### 5.1.1.14 Повторное производство (при браке)

**Участники:** складщик / производство.

**Процесс:** инициирование переделки, возврат заказа в производство.

**Результат:** заказ повторно выполняется.

#### 5.1.1.15 Выдача заказа клиенту

**Участники:** менеджер / складщик.

**Процесс:** уведомление клиента, выдача заказа.

**Результат:** статус «Завершён».

### 5.1.2 Услуга (офис / полиграфия)

- **5.1.2.1** Приём и регистрация заказа: создание заказа, ввод параметров
- **5.1.2.2** Быстрая подготовка макета: дизайн выполняется сразу, согласование с клиентом
- **5.1.2.3** Производство (в офисе): печать выполняется на месте
- **5.1.2.4** Выдача заказа клиенту: заказ передаётся клиенту

### 5.1.3 Продажа (товар)

- **5.1.3.1** Приём и оформление заказа: выбор товара, проверка наличия
- **5.1.3.2** Оформление продажи и оплата: фиксация продажи, проведение оплаты
- **5.1.3.3** Выдача товара клиенту: передача товара

## 5.2 Общие процессы

### 5.2.1 Управление складом

Приёмка, учёт остатков, контроль закупок.

### 5.2.2 Инвентаризация

Проверка остатков, корректировка.

### 5.2.3 Постобслуживание клиента (звонок, отзыв, оценка)

**Процесс:** через 1–2 дня создаётся задача, сотрудник звонит клиенту.

**Фиксируется:** оценка клиента, комментарий, дата, сотрудник.

**Особенность:** сейчас — офисный менеджер, в будущем — колл-центр.

### 5.2.4 Аналитика и контроль

Анализ прибыли, брака, эффективности сотрудников, удовлетворённости клиентов.

# 6. Функциональные требования

## 6.1 Модуль управления заказами (CRM)

**Назначение.** CRM является частью системы и отвечает за управление заказами и клиентами.

**Функции:** создание заказа, редактирование заказа, выбор типа заказа (услуга цех / услуга офис / продажа), управление статусами, просмотр истории изменений.

**Правила:**

- заказ создаётся в момент обращения клиента
- антидублирование клиентов (по телефону)
- обязательное указание типа заказа (товар / услуга)
- заказ создаётся с обязательными полями
- статус изменяется только по логике процессов
- запрещено ручное создание услуг (только справочник)

## 6.2 Модуль выездных продаж (Leads)

**Назначение.** Работа с потенциальными клиентами.

**Функции:** создание лида, фиксация замеров, загрузка фото, перевод лида в заказ.

**Правила:** лид может быть преобразован только в один заказ; обязательна фиксация размеров.

## 6.3 Модуль управления клиентами

**Назначение.** Ведение клиентской базы.

**Функции:** создание клиента, редактирование, поиск, история заказов.

**Правила:** антидублирование по номеру телефона; один клиент = один профиль.

## 6.4 Модуль услуг и товаров (справочники)

**Назначение.** Хранение перечня услуг и товаров.

**Функции:** добавление услуг, категоризация, настройка параметров.

**Правила:** услуги выбираются только из справочника; запрещено ручное дублирование.

## 6.5 Модуль дизайна

**Назначение.** Подготовка макетов.

**Функции:** постановка задач, загрузка макета, управление версиями, возврат на доработку.

**Правила:** каждый заказ имеет минимум одну версию макета; сохраняется история изменений.

## 6.6 Модуль проверки и согласования макетов

**Назначение.** Контроль качества макета.

**Функции:** проверка руководителем, возврат на доработку, согласование с клиентом.

**Правила:** без проверки макет не допускается в производство; статус фиксируется.

## 6.7 Модуль производства

**Назначение.** Выполнение заказов.

**Функции:** постановка задач, выбор операций, выполнение операций.

**Правила:** операции привязаны к заказу; все действия фиксируются.

## 6.8 Модуль учёта рабочего времени

> Думаю хорошо будет, если система будет учитывать, какой сотрудник сколько времени бездельничал, когда есть заказы, который можно выполнить физически.

**Назначение.** Контроль эффективности сотрудников.

**Функции:** начало работы, завершение работы, учёт времени.

**Правила:** время фиксируется автоматически; обязательна привязка к заказу.

## 6.9 Модуль контроля загрузки и эффективности сотрудников

**Назначение.** Контроль занятости сотрудников и выявление простоев.

**Функции:** отображение активных заказов, отображение доступных сотрудников, фиксация (кто начал работу / кто не приступил).

**Система должна рассчитывать:** фактическое рабочее время, время простоя, количество выполненных операций, загрузку сотрудников.

**Аналитика:** рейтинг сотрудников, выявление неактивных, сравнение сотрудников.

**Правила:** сотрудник обязан выбрать заказ; отсутствие активности фиксируется как простой.

## 6.10 Модуль нормирования операций и услуг

**Назначение.** Настройка стандартов выполнения работ администратором.

**Функции:** добавление новых услуг, редактирование услуг, настройка типа (цех / офис / товар), операций, норм времени.

**Примеры:** печать баннера → X мин/м²; буквы → X мин/шт.

**Использование:** сравнение план/факт, анализ эффективности, расчёт KPI.

**Правила:** только администратор может изменять; все изменения логируются.

## 6.11 Модуль склада

**Назначение.** Учёт материалов.

**Функции:** учёт остатков, приёмка материалов, списание материалов.

**Правила:**

- склад всегда актуален
- все движения фиксируются
- списание только на заказ (с определённым запасом)

## 6.12 Модуль контроля качества (брак)

**Назначение.** Фиксация брака.

**Функции:** фиксация брака, фото, указание причины.

**Правила:** фиксирует только складщик; обязательно указание этапа.

## 6.13 Модуль повторного производства

**Назначение.** Управление переделками.

**Функции:** инициирование переделки, возврат в производство.

## 6.14 Модуль продаж (товары)

**Назначение.** Продажа готовых товаров.

**Функции:** оформление продажи, проверка наличия, фиксация оплаты.

## 6.15 Модуль клиентского кабинета

**Назначение.** Работа клиента с системой.

**Функции:** создание заказа, просмотр статуса, история.

## 6.16 Модуль уведомлений и задач

**Назначение.** Коммуникация в системе.

**Функции:** уведомления, задачи, напоминания.

## 6.17 Модуль постобслуживания клиентов

**Назначение.** Сбор обратной связи.

**Функции:** задача на звонок, фиксация оценки, комментарии.

**Правила:** звонок обязателен через 1–2 дня; данные сохраняются.

## 6.18 Модуль аналитики и отчётов

**Назначение.** Контроль бизнеса.

**Функции:** отчёты, анализ KPI, анализ прибыли.

## 6.19 Модуль управления пользователями и ролями

**Назначение.** Контроль доступа.

**Функции:** создание пользователей, назначение ролей, управление правами.

## 6.20 Модуль контроля присутствия сотрудников (Face Control)

**Назначение.** Фиксация присутствия сотрудников.

**Функции:** получение данных из Face Control; фиксация входа и выхода.

**Система должна:** сопоставлять присутствие и рабочее время.

**Аналитика:** кто был, но не работал; опоздания; ранний уход.

**Правила:** сотрудник не может изменить данные; данные поступают автоматически.

## 6.21 Интеграционный модуль (Telegram-бот)

**Назначение.** Дополнительный интерфейс.

**Функции:** уведомления, быстрые действия.

## 6.22 Модуль учёта баланса сотрудников

**Назначение.** Учёт финансового баланса сотрудника между периодами.

**Функции:** отображение заработка, отображение доплаты, отображение долга, автоматический перерасчёт.

**Сотрудник видит:** сколько заработал, сколько получил, текущий баланс.

**Правила:** расчёт происходит автоматически; сотрудник не может изменить данные.

## 6.23 Модуль документооборота

- Автоматическая генерация счетов, договоров и актов выполненных работ на основе данных заказа.
- Возможность скачивания документов клиентом в Личном кабинете.
- Хранение архива закрывающих документов по каждому контрагенту.

## 6.24 Модуль управления логистикой

- Назначение водителя на доставку или монтажника на выезд.
- Расчёт стоимости логистики в зависимости от удалённости объекта (интеграция с картами).
- Статус заказа «В пути / На монтаже» для отслеживания в реальном времени.

## 6.25 Модуль мониторинга оборудования

- Учёт наработки часов или пройденных метров погонных для каждого станка.
- Автоматические уведомления администратору о необходимости технического обслуживания (ТО).

# 7. Логика расчётов

## 7.1 Расчёт стоимости заказа

**Назначение.** Определение цены для клиента.

**Расчёт включает:** стоимость материалов, стоимость работ (время), дополнительные расходы, маржу.

**Формула:** `Стоимость = Материалы + Работа + Доп.расходы + Маржа`

## 7.2 Расчёт расхода материалов

**Назначение.** Определение количества материалов для заказа.

**Основа:** нормы расхода, параметры заказа (размер, количество).

**Пример:** баннер → X м², краска → X мл/м².

## 7.3 Расчёт норм времени (план)

**Назначение.** Определение планового времени выполнения.

**Основа:** нормы, заданные администратором; тип операции.

**Формула:** `Плановое время = Норма × Объём`

## 7.4 Учёт фактического времени

**Назначение.** Фиксация реального времени работы.

**Источник:** кнопки «Начать / Завершить».

**Фиксируется:** сотрудник, заказ, операция, время.

## 7.5 Сравнение план vs факт

**Назначение.** Анализ отклонений.

**Формула:** `Отклонение = Факт − План`

**Интерпретация:**

- факт > план → низкая эффективность
- факт < план → высокая эффективность

## 7.6 Расчёт загрузки сотрудников

**Назначение.** Определение занятости.

**Формула:** `Загрузка = Рабочее время / Общее время присутствия`

## 7.7 Расчёт простоев сотрудников

**Назначение.** Выявление бездействия.

**Источники:** Face Control (присутствие), таймер (работа).

**Формула:** `Простой = Время присутствия − Время работы`

## 7.8 Расчёт эффективности сотрудников

**Назначение.** Оценка производительности.

**Формула:** `Эффективность (%) = (План / Факт) × 100`

## 7.9 Расчёт себестоимости заказа

**Назначение.** Определение реальных затрат.

**Включает:** материалы, время, брак, перерасход.

**Формула:** `Себестоимость = Материалы + Работа + Потери`

## 7.10 Учёт перерасхода материалов

**Назначение.** Контроль отклонений.

**Формула:** `Перерасход = Факт − Норма`

## 7.11 Учёт брака и потерь

**Назначение.** Фиксация убытков.

**Учитывается:** испорченные материалы, переделки.

## 7.12 Расчёт заработной платы сотрудников

**Назначение.** Определение оплаты труда.

**Основа:** выполненные операции, время, эффективность.

**Возможная модель:** `ЗП = База + Оплата за операции − штрафы (брак)`

## 7.13 Расчёт прибыли по заказу

**Назначение.** Определение доходности.

**Формула:** `Прибыль = Выручка − Себестоимость`

## 7.14 Общая финансовая аналитика

**Назначение.** Контроль бизнеса.

**Включает:** прибыль, расходы, эффективность, загрузка сотрудников.

## 7.15 Расчёт баланса сотрудника (аванс / перерасчёт)

**Назначение.** Обеспечение стабильного дохода сотрудников с учётом переменной загрузки.

**Логика:**

- Если сотрудник за расчётный период заработал меньше установленного минимума:
  - компания доплачивает разницу
  - разница фиксируется как «долг сотрудника перед компанией»
- Если сотрудник в следующем периоде зарабатывает больше:
  - часть дохода направляется на погашение долга

**Формула:** `Баланс = Выплачено − Заработано`

**Варианты:**

- ❌ Если заработано меньше: начисляется доплата; баланс уходит в минус
- ✅ Если заработано больше: происходит компенсация долга

**Пример:**

- **Месяц 1:** заработал 1.8 млн; выплата 3 млн; долг −1.2 млн
- **Месяц 2:** заработал 5 млн; выплата 3.8 млн; закрытие долга 1.2 млн

**Правила:** минимальная выплата фиксирована; баланс сотрудника хранится в системе; баланс автоматически учитывается при расчётах.

## 7.16 Учёт налоговой нагрузки

- Автоматическое добавление НДС (или иных налогов) к итоговой стоимости заказа.
- Разделение чистой прибыли и налоговых отчислений в финансовой аналитике.

## 7.17 Расчёт амортизации оборудования

Включение стоимости износа оборудования (лазер, плоттер, принтер) в себестоимость каждой выполненной операции.

## 7.18 Учёт логистических затрат

**Формула:** `Затраты на логистику = ГСМ + Амортизация авто + Оплата времени водителя`

Автоматическое добавление этих затрат в общую себестоимость заказа для корректного расчёта маржинальности.

# 8. Безопасность

## 8.1 Авторизация

**Назначение.** Обеспечение доступа пользователей к системе только после подтверждения личности.

**Требования:**

- вход в систему по номеру телефона и паролю
- возможность восстановления пароля
- поддержка выхода из системы (logout)

**Дополнительно:**

- хранение паролей в зашифрованном виде
- ограничение количества попыток входа
- автоматический выход при длительном бездействии

## 8.2 Роли и права доступа

**Назначение.** Разграничение доступа пользователей к функциям системы.

**Система должна поддерживать роли:** Администратор, Руководитель (наблюдатель / Read-only) 🔥, Менеджер, Производство, Складщик, Клиент.

### Администратор

**Права:** полный доступ ко всем функциям; управление системой; настройка модулей; управление пользователями и ролями.

### Руководитель (наблюдатель)

**Права:** просмотр всех заказов, финансов, аналитики, сотрудников, склада.

**Ограничения:** не может изменять данные, удалять, менять настройки, управлять пользователями.

### Менеджер

**Права:** создание и управление заказами, работа с клиентами, расчёт стоимости.

### Производство

**Права:** просмотр задач, выполнение операций, учёт рабочего времени.

### Складщик

**Права:** управление материалами, списание материалов, фиксация брака.

### Клиент

**Права:** создание заказов, просмотр своих заказов, отслеживание статуса.

### Общие правила

- доступ к функциям определяется ролью
- пользователь видит только разрешённые данные
- все действия ограничены правами

## 8.3 Логирование действий

**Назначение.** Фиксация всех действий пользователей в системе.

**Система должна записывать:**

- создание и изменение заказов
- списание материалов
- фиксацию брака
- изменения данных
- вход и выход пользователей

**Фиксируется:** пользователь, действие, дата и время, объект изменения.

**Правила:** логи не могут быть изменены пользователем; доступ к логам имеет только администратор.

## 8.4 Защита данных

**Назначение.** Обеспечение сохранности и конфиденциальности данных.

**Требования:**

- защита от несанкционированного доступа
- защита базы данных
- резервное копирование

**Дополнительно:**

- регулярные автоматические бэкапы
- возможность восстановления данных
- защита персональных данных клиентов

# 9. Нефункциональные требования

## 9.1 Производительность

- время отклика системы: не более 2 секунд для стандартных операций; не более 5 секунд для сложных
- система должна поддерживать одновременную работу не менее 20 пользователей (на старте)
- обработка операций (заказ, списание, таймер) должна происходить в реальном времени

## 9.2 Надёжность

- стабильная работа без сбоев
- сохранение данных при любых операциях
- защита от потери данных
- автоматическое сохранение, защита от ошибок пользователя, восстановление после сбоев

## 9.3 Масштабируемость

- поддержка роста количества пользователей, заказов, объёма данных
- добавление новых модулей и расширение функционала без изменения ядра

## 9.4 Доступность системы

- доступность 24/7
- поддержка работы через мобильное устройство и веб-интерфейс

## 9.5 Работа в реальном времени

- обновление данных без перезагрузки
- моментальная фиксация времени работы, статусов, действий

## 9.6 Безопасность данных

- защита персональных данных
- защита от несанкционированного доступа
- разграничение прав пользователей

## 9.7 Резервное копирование (Backup)

- регулярное автоматическое создание резервных копий
- хранение резервных копий
- возможность восстановления системы

## 9.8 Удобство использования (Usability)

- простой и понятный интерфейс
- минимальное количество действий для выполнения операций
- обучение пользователя не более 1 дня

## 9.9 Совместимость

- поддержка современных браузеров
- интеграция Telegram-бот
- адаптация под разные устройства

## 9.10 Хранение медиа-файлов (макетов)

- использование внешнего облачного хранилища для тяжёлых графических файлов наружной рекламы
- срок гарантированного хранения рабочих макетов — не менее 12 месяцев

## 9.11 Методы оплаты

- поддержка оплаты: наличные, банковская карта (эквайринг), расчётный счёт (B2B)
- автоматическая синхронизация статуса оплаты с производственным циклом заказа
