## Миссия

**Каждый материал, операция и рубль — на конкретном заказе.**

UniPrint превращает производство наружной рекламы и полиграфию из
«ручного бизнеса с Excel и мессенджерами» в прозрачную цифровую
систему, где владелец видит реальную себестоимость каждого заказа,
сотрудник зарабатывает по результату, а клиент сам оформляет, оплачивает
и получает заказ — без бутылочного горлышка в виде менеджера.

Мы не делаем «ещё одно 1С». Мы решаем три критические дыры одновременно:
**учёт материалов на заказ** (BR-01), **сдельную мотивацию сотрудников**
(BR-05) и **самообслуживание клиента** (клиентский кабинет 6.15) —
только в комплексе они дают эффект, а не точечно.

## Цели TO-BE

Из ТЗ § 2.3, развёрнуто.

### 1. Каждый материал — на конкретный заказ

Сегодня: материал «уехал в цех», и куда — неизвестно. Списания общие,
себестоимость угадывается.

Завтра: списание возможно **только на заказ** (BR-01), enforce'ится на
бэке. На каждом списании — заказ, материал, количество, сотрудник,
этап, отметка времени. Себестоимость заказа складывается из реальных
списаний, а не из «средних по цеху». Видна разница план / факт по
норме (модуль 6.10), виден перерасход (расчёт 7.10).

### 2. Каждое действие — фиксируется

Сегодня: задачи по чату, время операций «на глаз», брак не фиксируется.

Завтра: «Начать работу» / «Завершить работу» в Mobile-PWA с
автоматическим таймером, привязка к заказу и операции. Брак фиксирует
**только складщик** (BR-03) с фото, количеством, этапом и
ответственным. Все действия — в audit-log, не редактируются.

### 3. Каждый сотрудник зарабатывает по результату

Сегодня: фиксированная ЗП, нет мотивации делать быстрее / лучше.

Завтра: сдельная ЗП по выполненным операциям. Формула двойная — для
каждой операции / роли настраивается одно из двух: «**тариф ₽ ×
количество**» **или** «**% от стоимости работы для клиента**» (модуль
6.22, ответ 🔴 #8 от 2026-05-05). Минимальная гарантированная выплата
с долговым балансом (BR-05) — если в месяц заработал меньше минималки,
компания доплачивает в долг, в следующем месяце вычитает (без ухода в
минус по ТК ст. 137). Сотрудник видит свой баланс в Mobile-PWA в
реальном времени. Минимум за период / штрафы за брак / бонусы —
детализируем в момент настройки `apps/payroll`.

### 4. Каждый клиент сам оформляет заказ

Сегодня: клиент звонит / пишет менеджеру, тот вносит заказ в систему
с задержкой 2-3 дня.

Завтра: клиентский кабинет (6.15) — клиент сам выбирает услугу из
справочника, видит расчёт цены, оформляет, оплачивает (B2C через
эквайринг, B2B по р/с — конкретный провайдер после 🔴 #5),
отслеживает статус, скачивает счёт / акт. Менеджер подключается
только к нестандартным заказам и B2B-сделкам.

### 5. Каждый дубль — устраняется

Сегодня: «Иванов» в базе 3 раза, услуга «Баннер / банер / другое»
плодится бесконечно. Аналитика недостоверна.

Завтра: антидублирование клиентов **по телефону** (BR-02), услуги
**только из справочника** (BR-04) — ручной ввод запрещён на бэке.
Один клиент = один профиль, одна услуга = одна запись в каталоге.

### 6. Каждое присутствие — сравнивается с работой

Сегодня: «вроде был на работе, а что делал — непонятно».

Завтра: Face Control (модуль 6.20) фиксирует время присутствия,
кнопки «Начать / Завершить» — время работы. Простой = присутствие − работа
(расчёт 7.7). Видны опоздания, ранний уход, работа без присутствия
(саботаж Face Control). События **не редактируются** сотрудником
(BR-06) — только просмотр + ручная корректировка времени смены через
отдельный workflow с журналом.

### 7. Каждый заказ закрывается документами

Сегодня: счёт делает менеджер вручную в Excel, акт забывается, ТТН
на бумажке у водителя.

Завтра: автогенерация счёт / договор-оферта / акт / ТТН по данным
заказа (модуль 6.23). PDF в Yandex Object Storage + audit-log. Клиент
скачивает в кабинете. Архив документов по контрагенту хранится 5 лет
(402-ФЗ). ЭДО (Диадок / СБИС) — фаза 2+.

### 8. Каждый рубль себестоимости — учтён

Сегодня: считается «материалы + примерное время × ставка». Амортизация
оборудования, ГСМ водителей, налоги — не учитываются.

Завтра: себестоимость = материалы + работа + амортизация оборудования
(7.17) + логистические затраты (7.18, через Yandex Maps по адресам
монтажа) + налоги (7.16) + потери (брак / перерасход). Прибыль по
заказу — реальная, не оценочная.

## Пользовательские сегменты (персоны)

10 ключевых персон. Для каждой — контекст, основные задачи, pain
points (что болит сегодня), ключевые экраны.

### 1. Учредитель / руководитель типографии

**Контекст.** Владелец компании. На объекте бывает 1-3 раза в неделю,
основное время — переговоры, продажи, стратегия. Принимает решения
по закупке оборудования, найму, развитию.

**Задачи.**
- Дневной обзор: сколько заказов в работе, какой план / факт по
  выручке, есть ли проблемы (брак, простои, конфликты).
- Контроль прибыльности — какие заказы / клиенты / услуги приносят
  деньги, какие в минус.
- Контроль сотрудников — кто эффективен, кто бездельничает, кому
  переплатили / недоплатили.
- Принятие решений о ценообразовании, скидках, инвестициях.

**Pain points (AS-IS).** Не видит реальной себестоимости заказа.
Не знает, кто и сколько работает по факту. Принимает решения «вслепую»,
по интуиции и устным докладам.

**Ключевые экраны.** Owner Dashboard (Web) — KPI типографии в одном
окне (выручка, прибыль, брак, простои, баланс ЗП), drill-down по
заказам. Mobile-PWA того же — для дороги.

**Цитата.** «Я хочу открыть телефон утром и за 2 минуты понять,
заработал я вчера или потерял».

### 2. Начальник цеха / руководитель производства

**Контекст.** Управляет 5-15 производственниками. На объекте каждый
день. Распределяет задачи, проверяет макеты перед запуском в цех,
отвечает за сроки.

**Задачи.**
- Видеть загрузку каждого сотрудника и доступные задачи.
- Проверять макеты перед запуском в производство (ТЗ § 5.1.1.5).
- Реагировать на брак — кого наказать, кого переучить.
- Контролировать сроки заказов.

**Pain points.** Задачи распределяет вручную в чате. Не видит общей
картины загрузки. Брак вскрывается «постфактум», когда заказ уже не
успеть.

**Ключевые экраны.** Production Dashboard (Web) — kanban задач по
этапам, фильтр по сотрудникам и сложности. Экран проверки макета.
Алерты по простоям и срывам сроков (WebPush + Email).

### 3. Производственник (печатник / лазерщик / монтажник / плотник)

**Контекст.** В цехе или на монтажном объекте. Андроид-планшет или
смартфон под рукой. Руки часто заняты / в краске / в рукавицах.

**Задачи.**
- Утром пройти Face Control, начать смену.
- Выбрать задачу из своего списка, нажать «Начать работу».
- Выполнить операцию (печать / резка / монтаж / сварка).
- Нажать «Завершить работу», передать заказ дальше.
- Видеть свой заработок за период.

**Pain points.** Не понимает, сколько заработал. Не видит, что
делать следующим. Сейчас — фиксированная ЗП, нет связи с результатом.

**Ключевые экраны.** Mobile-PWA (production-mobile) — список задач
(одна большая кнопка на каждой), таймер активной работы, история
заработка. WebPush-уведомления о новых задачах.

**Особенность.** **Не фиксирует брак** — это зона складщика (BR-03).
**Не создаёт услуги вручную** — только из справочника (BR-04).

### 4. Складщик

**Контекст.** На складе. Принимает материалы от поставщиков, выдаёт
в производство, принимает готовые заказы для контроля качества.

**Задачи.**
- Приёмка материала: сканировать, ввести количество, поступление в
  систему (партия, цена, срок годности).
- Списание на заказ — выбрать заказ, материал, количество, подтвердить.
- Принять готовый заказ от производства, проверить качество.
- При обнаружении брака — зафиксировать в системе (фото, количество,
  этап, ответственный) и инициировать переделку.
- Инвентаризация раз в месяц / квартал.

**Pain points.** Сейчас списывает «в цех», не на заказ — не видит,
куда ушло. Брак не фиксируется, потери уходят «в воздух». Инвентаризация —
вручную с листом и ручкой.

**Ключевые экраны.** Mobile-PWA (warehouse-mobile) — сканер
штрих-кодов через `BarcodeDetector` API / web-фотокамера, форма
списания на заказ (с поиском заказа по номеру), форма фиксации брака
с фото и геометкой. Web-дашборд для остатков и аналитики.

**Особенность.** Уникальная роль — **только складщик** фиксирует
брак (BR-03). Bottleneck-риск, если складщик один (см. `08-risks.md`
O2).

### 5. Дизайнер

**Контекст.** В офисе или удалённо. Работает в Adobe Suite / CorelDRAW.
Получает задачи, готовит макеты, отдаёт на проверку и согласование.

**Задачи.**
- Получить задачу с требованиями (размер, формат, текст, фото).
- Подготовить макет в Illustrator / Photoshop / CorelDRAW.
- Загрузить в систему (большие файлы — до 2 ГБ, в Yandex Object Storage
  через multipart upload).
- Передать на проверку начальнику производства (ТЗ § 5.1.1.5).
- При замечаниях — доработать.
- После согласования с клиентом — адаптировать под производство
  (припуски, ICC-профиль, m.п.).

**Pain points.** Файлы пересылает по почте / в мессенджерах. Версии
теряются. Согласование с клиентом — отдельная переписка, не привязано
к заказу.

**Ключевые экраны.** Web-кабинет дизайнера — очередь задач, форма
загрузки макета (multipart upload), история версий, статус
согласования. Возможность комментариев от клиента.

### 6. Менеджер по продажам (офисный)

**Контекст.** В офисе. Принимает звонки, пишет в мессенджерах,
оформляет заказы за клиентов.

**Задачи.**
- Создать заказ при обращении клиента — выбрать клиента (или
  создать с проверкой на дубль), выбрать услугу из справочника,
  указать тип заказа (цех / офис / товар), параметры, рассчитать цену.
- Согласовать с клиентом, отправить макет на согласование.
- Отслеживать статус заказа, реагировать на проблемы.
- Закрыть заказ — оплата, документы, выдача.

**Pain points.** Сейчас вносит заказы с задержкой. Постоянно дёргают
производство «как там мой заказ?». При оплате — счёт делает в Excel.

**Ключевые экраны.** Web-кабинет менеджера — форма создания заказа
(с автокалькулятором), список своих заказов с фильтром по статусу,
карточка клиента с историей. Уведомления по WebPush + Email.

### 7. Менеджер выездных продаж

**Контекст.** На объекте у клиента. Делает замеры наружной рекламы,
принимает решение «по фотографии». Смартфон + рулетка.

**Задачи.**
- Найти клиента (обход / рекомендации / звонок).
- Выехать, замерить, сфотографировать.
- Создать **лид** в системе (модуль 6.2) — фото, размеры, требования.
- При согласии клиента — конвертировать лид в заказ.
- Передать заказ дизайнеру (или сделать самому если простое).

**Pain points.** Замеры записывает в блокнот / на планшет, теряет
фото. По возвращении в офис заказ оформляет с задержкой.

**Ключевые экраны.** Mobile-PWA — форма лида с камерой и геометкой
(`navigator.geolocation`), конвертация лида в заказ одной кнопкой.
Карта объектов на маршруте через Yandex Maps API.

### 8. Водитель / монтажник (логистика — модуль 6.24)

**Контекст.** В машине или на монтажном объекте. Доставка готовых
заказов на адреса, монтаж рекламных конструкций.

**Задачи.**
- Получить задание на доставку / монтаж.
- Загрузить заказ со склада (отметка в системе).
- Доехать по маршруту (**Yandex Maps**, по ответу 🔴 #10).
- На месте — отметить «Прибыл», после монтажа — «Завершил».
- Сфотографировать установленный объект — фотофиксация.
- Получить документы (ТТН, акт) для подписи клиентом, загрузить
  скан в систему.

**Pain points.** Сейчас всё на бумаге. Учёт ГСМ — отдельный лист,
амортизация авто никак не учитывается в себестоимости.

**Ключевые экраны.** Mobile-PWA — карта маршрута на Yandex Maps,
статус «В пути / На монтаже / Завершён», фотофиксация. GPS-трек для
расчёта логистических затрат (7.18).

### 9. Администратор системы

**Контекст.** Может быть совмещённой ролью (учредитель + админ) или
отдельной. Управляет справочниками, ролями, нормативами.

**Задачи.**
- Заводить новых сотрудников, выдавать роли (RBAC).
- Управлять справочником услуг / материалов (модуль 6.4) — добавление
  только админ (BR-04 enforce).
- Настраивать нормы расхода материалов и времени операций (6.10).
- Контролировать audit-log.
- Резервное копирование, восстановление.

**Pain points.** Сейчас этой роли по сути нет — справочники в Excel,
права доступа «у всех ко всему».

**Ключевые экраны.** Admin Panel (Web) — управление пользователями
и ролями, редакторы справочников, журнал действий, настройки системы.

### 10а. Клиент B2B (корпоративный)

**Контекст.** Сотрудник отдела закупок / маркетинга компании-клиента.
Заказывает регулярно (раз в неделю / месяц), большими тиражами.
Платит по р/с, требует счёт-фактуру, акт, отгрузочные документы.

**Задачи.**
- Войти в кабинет (Email + magic-link **или** SMS-код на телефон).
- Создать заказ из шаблона предыдущего или с нуля.
- Загрузить макет (через систему загрузки большого файла в
  Yandex Object Storage).
- Получить счёт, отправить в бухгалтерию на оплату.
- Отслеживать статус заказа.
- После выполнения — скачать акт, ТТН, подписать ЭЦП (фаза 2+).

**Pain points.** Сейчас вся переписка — через менеджера. Документы —
сканы по почте. Статус заказа — звонком.

**Ключевые экраны.** Client Portal (Web) — каталог услуг, форма заказа,
загрузка макетов, корзина, оплата по р/с (счёт PDF), история заказов,
архив документов.

### 10б. Клиент SMB / физлицо (B2C)

**Контекст.** Физлицо или микро-бизнес. Печатает 50 визиток, баннер
для одной выставки, небольшую партию листовок. Платит картой / СБП.

**Задачи.**
- Зайти на лендинг, посмотреть цены.
- Зарегистрироваться (Email + magic-link / SMS-код).
- Оформить заказ из стандартного шаблона.
- Оплатить онлайн (эквайринг — после 🔴 #5; провайдер B2C-чека — после
  выбора ОФД).
- Отследить статус, забрать в офисе или с доставкой.

**Pain points.** Не понимает «как у вас заказать», «сколько это
стоит», «когда будет готово». Уходит к конкурентам с понятным сайтом.

**Ключевые экраны.** Client Portal (упрощённый) — короткая форма
заказа, оплата картой / СБП, push-уведомление «готов к выдаче»
(WebPush в браузере + дублирование SMS).

## Продуктовые принципы

7 принципов — они rule out альтернативные дизайн-решения и
embed'ятся в acceptance criteria каждого PRD.

### 1. Списание материалов — только на заказ. Без исключений.

**BR-01.** Бэк не позволяет создать запись `MaterialMovement` с
`order_id = null` для типа «списание». Альтернативы — «списать в цех»,
«списать с одобрения начальника», «списать с пометкой» — **не
предлагаем**. Это критическая проблема AS-IS, и мы её решаем
жёсткой инвариантой.

Исключение: списание на брак — тоже на заказ, но через специальный
тип `Defect` с привязкой к исходному заказу.

### 2. Mobile-first для производства и склада через PWA

Сотрудники в цехе / на складе / на монтаже работают руками. На
ноутбуке не сидят. **Mobile-PWA** (Next.js + Service Worker + Web App
Manifest) — основной канал, не «ещё один». Ставится на смартфон /
планшет иконкой, работает оффлайн через Service Worker, push-нотификации
через Web Push API. Touch-target ≥ 44pt, одна задача = одна большая
кнопка, минимум скроллинга, читается под наклоном смартфона в
перчатке.

Web-кабинет (тот же codebase, desktop-layout) — для офисных ролей
(менеджер, дизайнер, руководство, админ) и клиентов.

Native-приложения (Kotlin / Swift) — рассматриваются только если PWA
не покроет hardware-доступ (например, специфический Face Control SDK
требует native-кода). Решение — фаза 3+.

### 3. Антидублирование везде

- **Клиенты по телефону** (BR-02) — нормализация формата (+7 / 8 / 7
  → единый формат через libphonenumber), поиск с fuzzy-matching при
  создании.
- **Услуги по справочнику** (BR-04) — ручной ввод запрещён, только
  выбор + параметры.
- **Материалы по SKU** — поиск перед заведением новой позиции.

Дубли разрушают аналитику и портят качество данных. Это
единственный способ построить достоверный отчёт.

### 4. Прозрачность для всех ролей через RBAC

- **Учредитель** видит всё, кроме персональных данных клиентов
  (есть, но скрыто за RBAC + audit-log).
- **Сотрудник** видит свои задачи, свой заработок, свой баланс,
  но не зарплаты других.
- **Клиент** видит свои заказы и документы, но не себестоимость,
  не маржу, не сотрудников.
- **Складщик** видит остатки и движения, но не финансовые отчёты.

Каждый видит то, что нужно для работы. Не больше, не меньше.

### 5. Каждое действие фиксируется. Ничего не теряется.

Audit-log на все ключевые действия:
- создание / изменение / удаление заказа,
- списание материала,
- фиксация брака,
- начало / завершение операции,
- изменение статуса,
- доступ к ПДн (152-ФЗ требует),
- расчёт ЗП.

Логи **не редактируются** пользователем. Только просмотр для админа.
Хранение — 5 лет (402-ФЗ для финансовых событий).

### 6. Брак фиксирует только складщик

**BR-03.** Производственник не имеет UI для фиксации брака. Чтобы
зафиксировать брак, заказ должен прийти на склад (после
«Завершить работу»), складщик проверяет качество (ТЗ § 5.1.1.13)
и при обнаружении брака — фиксирует.

Это устраняет конфликт интересов: производственник не наказывает
сам себя; складщик объективно проверяет качество.

### 7. Нотификации — параллельно по 3 каналам, без Telegram

По решению владельца от 2026-05-05 (🔴 #7), Telegram-бот **не
используется** в продукте — ни как канал нотификаций, ни как канал
авторизации, ни как Mini App. Все уведомления — через
`apps/notifications` с провайдер-абстракцией:

- **WebPush** (primary) — клиенту в браузере / PWA-сотруднику.
- **SMS** — для критичных событий (заказ принят / готов / брак),
  буквенный sender_id «UNIPRINT» через SMS-провайдера РФ (SMSAero /
  MTS Exolve — выбор после ADR-0004).
- **Email** — для документов (счёт / акт / ТТН) и формальных
  уведомлений; домен с настроенным SPF / DKIM / DMARC.

Авторизация клиента — Email + magic-link **параллельно** с SMS-кодом.
Telegram Login и Telegram Mini App в кабинет **не делаем**.

Telegram остаётся **только** PM-каналом команда↔владелец (внутреннее,
не часть продукта).

## KPI продукта

Метрики разделены на **бизнес-KPI** (что меняется в типографии),

SLI** (как работает система). Целевые значения — рабочая гипотеза
v1, калибруются после baseline-измерений.

| Метрика | Формула | Baseline (AS-IS) | Цель v1 (Phase 2) | Цель v3 |
| --- | --- | --- | --- | --- |

| % заказов с зафиксированным браком | brak_orders / total_orders | TBD (вероятно 0% — не фиксируется) | 3-5% (это **рост** — впервые видим реальный брак) | < 3% (после оптимизации) |

## Анти-цели — что осознанно НЕ делаем в v1

Чтобы держать фокус и не превратиться в «ещё одно 1С».

| Не делаем в v1 | Почему | Когда вернёмся |
| --- | --- | --- |
| **Бухгалтерия и налоговая отчётность** | Out of scope (см. 00-summary.md). Платформа выдаёт первичку, бухгалтер заносит в 1С. | Никогда — это аутсорс. Возможно — интеграция через export в фазе 3. |
| **ЭДО (Диадок / СБИС)** | Сложная интеграция, нужна только для крупного B2B. По 🔴 #13 (закрыт) — MVP делает простую генерацию PDF. | Фаза 2+ при появлении B2B-клиентов с требованием ЭДО. |
| **Telegram-бот / Mini App / Telegram Login** | По 🔴 #7 (закрыт) — Telegram **не используется** в продукте. Каналы — SMS + Email + WebPush. | Не возвращаемся к этому в продукте. |
| **Native mobile-приложения (Kotlin / Swift)** | По 🔴 #6 (закрыт) — Phase-1 = PWA mobile-first. Native не нужен пока PWA покрывает hardware-сценарии. | Фаза 3+ если PWA не справится с Face Control камерой / специфическим SDK. |
| **CV-анализ макетов** (распознавание шрифтов, ICC, припусков) | ML-фича, требует датасета и обучения. Pre-flight валидацию делаем на простых правилах. | Резервная роль R2 (фаза 2+). |
| **AI-чатбот для клиента** | Нужен прозрачный сервис, не «угадайка». Пока — менеджер + статусы. | Фаза 3+ при достаточной базе обращений для обучения. |
| **Multi-warehouse / multi-tenancy в UI** | По 🔴 #9 (закрыт) — один цех / один склад MVP. FK в модели данных закладываем, UI один. | Фаза 2+ при росте. ADR-0006. |
| **Интеграция с подрядчиками-курьерами** (СДЭК / ПЭК / Деловые линии) | Свои водители достаточно для широкоформатной рекламы (по 🔴 #10 — Yandex Maps). | Фаза 3+ при росте офис-полиграфии и выходе в регионы. |
| **Cold Storage и долгосрочное архивирование макетов** | На старте достаточно 12 месяцев Hot Storage в Yandex Object Storage (NFR 9.10). | Фаза 2+ — lifecycle-policy в S3 (Cold после 90 дней). |
| **Маркетинговые рассылки (массовые)** | Это для B2C-консьюмер-бренда; UniPrint — B2B-tooling. | Никогда платформенно; только реактивация через email-cron. |
| **Соцсети / SMM-инструменты для типографии** | Out of scope полностью. | Никогда. |
| **Рейтинг сотрудников «глобальный»** (gamification, leaderboard) | Может демотивировать слабых; пока KPI без публичного ранжирования. | Не раньше v3, после культурного fit-теста. |

## Эволюция v1 → v2 → v3

### v1 (Phase 1 — MVP, продажи + производство + склад)

**Сценарий пилота.** Один цех, один склад, 1 типография работает на
новой системе. Цель — закрыть критические дыры AS-IS.

**Что работает:**
- 3 типа заказа (цех / офис-полиграфия / товар) — статус-машины разные.
- Антидублирование клиентов (BR-02) и услуг (BR-04).
- Списание материалов **только на заказ** (BR-01).
- Брак — только складщик (BR-03), фото + причина + этап.
- **Mobile-PWA** для производства и склада (production-mobile,
  warehouse-mobile в Turborepo).
- Web для менеджеров, дизайнеров, руководства, клиентов (тот же codebase).
- Базовый клиентский кабинет (создание заказа, статус, история).
- 6 ролей (учредитель / нач. цеха / менеджер / производство /
  складщик / клиент) + админ.
- Audit-log, RBAC, базовая безопасность.
- Уведомления через `apps/notifications`: WebPush + Email + SMS.
- Yandex Object Storage для макетов (multipart upload).

**Что НЕ работает в v1:**
- Сдельная ЗП (есть учёт времени, но расчёт ЗП — вручную).
- Face Control (приходит в v2).
- Документооборот (счёт / акт — простой PDF, без ЭДО).
- Логистика (без GPS-трека и автомаршрутов на Yandex Maps).
- Амортизация / налоги в себестоимости.

### v2 (Phase 2 — финансы, ЗП, Face Control)

**Цель.** Замкнуть финансовый контур и интегрировать биометрический
контроль.

**Новое:**
- **Сдельная ЗП** работает по двойной формуле (тариф ₽ × кол-во **или**
  % от стоимости работы — настраивается per operation/role) — модуль
  6.22, BR-05 (баланс сотрудника).
- **Face Control** интегрирован с vendor'ом (🔴 #2 → ADR-0002), модуль
  6.20, BR-06 (read-only события).
- **152-ФЗ ст. 11 согласия** собраны со всех сотрудников.
- **Расчёт себестоимости** включает амортизацию оборудования (7.17)
  и логистические затраты (7.18 — упрощённо).
- **Налоги в стоимости** (7.16) — НДС / УСН в зависимости от СНО.
- **Постобслуживание клиента** (6.17) — задача-звонок через 1-2 дня.
- **Интеграция с эквайрингом** (🔴 #5 → ADR-0005) — B2C онлайн-оплата.
- **ОФД / 54-ФЗ** — провайдер выбран (под-блокер от Q14, который
  закрыт «микс B2B+B2C»). Онлайн-чеки B2C через ОФД.

### v3 (Phase 3 — логистика, документооборот, мониторинг оборудования)

**Цель.** Закрыть полный цикл «заказ → монтаж → акт» и обеспечить
эксплуатацию оборудования.

**Новое:**
- **Документооборот 6.23** — автогенерация счёт / договор-оферта /
  акт / ТТН по данным заказа, скачивание клиентом, архив 5 лет
  (402-ФЗ).
- **Логистика 6.24** — назначение водителя, расчёт стоимости по
  **Yandex Maps** (по ответу 🔴 #10 от 2026-05-05), статус «В пути /
  На монтаже», GPS-трек.
- **Мониторинг оборудования 6.25** — наработка часов / м.п., авто-
  уведомления о ТО.
- **ЭДО** (фаза 3.5+) — Контур.Диадок или СБИС если есть B2B-спрос.
- **Multi-warehouse / multi-tenant** (ADR-0006) если развиваются
  филиалы (FK уже в модели данных с фазы 1).
- **R1 — Data Engineer** активирован, DWH для аналитики (когда PG
  не справляется).
- **R2 — AI / ML Engineer** активирован при появлении CV-фичей
  (контроль макетов, OCR накладных).
- **Native mobile-приложения** — рассмотрение, если PWA упёрлась в
  hardware-ограничения (Face Control камера через native SDK).

### За v3 (опционально, когда понятно куда расти)

- Расширение на другие типографии — UniPrint как мульти-tenant SaaS.
- AI-помощник клиенту в кабинете (FAQ + статус заказа).
- Прогноз времени производства по истории.
- Маркетплейс типографий — клиент создаёт заказ, несколько типографий
  делают предложения.
